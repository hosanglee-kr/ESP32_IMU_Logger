2단계: 슬라이딩 윈도우 / hop_size 적용

이건 전처리 구조 개편 직후에 넣는 것이 좋습니다.

지금은 사실상:

frame_size = 256

hop_size = 256


고정입니다.

이걸 아래처럼 바꾸면 됩니다.

추가 설정

frame_size

hop_size


동작 정책

hop_size == frame_size → non-overlap

hop_size < frame_size → overlap

hop_size == 128 with frame 256 → 50% overlap

hop_size == 64 → 75% overlap


내부 구현 포인트

현재 raw frame buffer를 “256개 채우면 바로 처리” 방식으로 쓰고 있는데, 슬라이딩 윈도우를 넣으려면 frame assemble 방식이 바뀌어야 합니다.

권장 방향은:

센서 task는 샘플을 계속 circular sample buffer에 적재

process task는 hop_size 기준으로 frame 추출

frame window는 frame_size 길이로 복사

처리 후 다음 시작점 += hop_size


즉, 지금의 “완성된 frame queue 전달” 모델에서 다음 중 하나로 바뀝니다.

권장안

샘플 ring buffer + hop trigger 방식

sensor task

샘플 저장

누적 샘플 수 증가


process task

available_samples >= frame_size

첫 프레임 처리 후 consume = hop_size

남은 샘플은 유지



이 구조가 overlap에 가장 잘 맞습니다.

왜 이걸 2단계로 두는가

전처리 pipeline과 sliding window가 함께 들어가면 “frame 단위 처리 객체” 개념이 명확해집니다. 그래서 둘을 붙여 처리하는 게 맞습니다.

권장 정책

RECORDING → stop measurement


long press:

optional event marker

or wifi/web enable toggle

or calibration



이 단계의 핵심

이제부터 start()는 “processing task 시작”이고, 실제 측정은 별도의 measurement_start() 개념으로 분리하는 게 좋습니다.

즉:

begin() : init

start() : realtime engine/task start

measurementStart() : 기록 세션 시작

measurementStop() : 기록 세션 종료


이 구조로 가야 SD 저장과 웹 제어가 자연스럽게 붙습니다.

구현 난이도/리스크 기준으로 본 핵심 주의점

1. 슬라이딩 윈도우가 제일 중요하고 제일 영향 큼

이건 단순 옵션 추가가 아니라 샘플 수집 구조를 frame-complete 방식에서 hop-driven 방식으로 바꾸는 일입니다.

2. stage 배열화는 지금 안 해두면 나중에 더 크게 뜯어야 함

여러 필터, normalize, clipping, detrend 등을 넣으려면 지금 구조로는 계속 if가 늘어납니다.

3. 버튼 start/stop은 기록 계층보다 먼저 state machine부터

버튼만 먼저 붙이면 나중에 stop 의미가 달라져서 다시 뜯게 됩니다.

4. 저장은 binary 우선

CSV는 느리고 용량이 큽니다. raw는 binary 기본, CSV는 옵션이 맞습니다.

5. metadata/event는 JSONL이 가장 확장 친화적

나중에 event type, label, marker, note, config change 등 확장할 때 편합니다.
