## II. :blue_square: 검토 결과 :blue_square:

  - 제시해주신 방향은 T20 시스템을 단순한 '상태 감시' 도구에서 **'자율 학습형 예지보전 솔루션'**으로 진화시키는 매우 정교한 설계입니다.
  - 특히 **N순위 주파수 피크 추출**과 **학습 모드의 분리**는 향후 TinyML 모델(Autoencoder, GMM 등)을 온디바이스에서 훈련하거나, 클라우드에서 정밀 레이블링을 수행하기 위한 핵심 데이터 파운데이션이 될 것입니다.

#### 1. 스펙트럼 N순위 주파수 특징량 (N-Peak Features) 설계
  - 단순히 전체 에너지를 보는 것이 아니라, 스펙트럼에서 가장 지배적인 성분 N개를 추출하여 기계의 핵심 거동(RPM, 베어링 진동 등)을 핀포인트로 추적합니다.

#### **A. 인근 주파수 배제(Non-Maximum Suppression) 로직**
  - 하나의 물리적 진동 성분은 FFT 상에서 인접한 여러 Bin에 에너지가 퍼져 나타납니다. 이를 중복 추출하지 않기 위한 **'Peak Guard Band'** 로직을 적용합니다.
    1) :cherries: **Search:** 전체 Magnitude 배열에서 최댓값(Peak_1)을 찾습니다.
    2) :cherries: **Masking:** Peak_1의 인덱스를 기준으로 전후 W개(예: \pm 3 bins) 범위를 검색 대상에서 제외(Zero-masking)합니다.
    3) :cherries: **Iterate:** 남은 영역에서 다음 최댓값(Peak_2)을 찾는 과정을 N번 반복합니다.
    4) :cherries: **Feature:** 최종적으로 ${Freq_1, Mag_1, \dots, Freq_N, Mag_N}$의 1D 벡터를 생성합니다.

#### **B. Delta 및 Delta-Delta 피처 추가**
  - 주파수 피크의 **시간적 변화율**은 결함 진행 속도를 파악하는 결정적 지표가 됩니다.
    * :apple: **Delta Peak:** 주파수 이동(Frequency Drift)을 감지합니다. (예: 볼트 풀림 시 고유 주파수가 낮아짐)
    * :apple: **Delta Mag:** 에너지 급증을 감지합니다.
    * :apple: **활용 방안:** 설비 가속/감속(RUN 상태 전이) 중에는 Delta Freq가 높게 나타나므로, 이를 통해 '과도기 상태'를 인식하고 트리거 오작동을 방지하는 알고리즘에 활용합니다.

### 2. 별도 학습 모드(Learning Mode) 및 데이터 구조 설계
  - 정상 상태를 완벽히 이해해야 이상(Anomaly)을 정의할 수 있습니다. 운영 모드와 완전히 분리된 전용 파이프라인을 구축합니다.

#### **A. 학습 모드에서의 프로파일링 전략**
  * **데이터 격리:** 학습 모드에서 생성된 데이터는 /t20_data/train/ 폴더에 별도 저장됩니다. 파일 헤더에 MODE_LEARN 플래그를 삽입하여 향후 레이블링 시 자동 분류되게 합니다.
  * **1차 프로파일링 (Baseline):** * 사용자가 설비를 **정상 IDLE** 및 **정상 RUN** 상태로 가동하며 각각 학습 버튼을 누릅니다.
  * 각 상태별로 \{RMS, MFCC_{1-13}, Peak\_Freq_{1-N}, Band\_Energy_{1-3}\}$의 통계값($\mu, \sigma)을 산출하여 전용 프로파일 파일(profile_idle.json, profile_run.json)로 굽습니다.

#### **B. 적응형 임계값 (Adaptive Threshold) 적용 로직**
  * **운영 중 판별:** 시스템 가동 시 현재 추출된 피처가 profile_idle에 가까운지 profile_run에 가까운지 거리 산출(Mahalanobis Distance 등)을 통해 상태를 실시간 인식합니다.
  * **동적 가드링:** 현재가 RUN 상태라면 profile_run의 통계치를 기준으로 임계값을 자동 스위칭합니다.
   
### 3. 피처(Feature) 검토 및 레이블링 확장 방안
  - 기존 피처와 신규 피처를 조합하여 어떻게 설비의 운명을 예측할지 검토합니다.

| 피처 카테고리 | 대상 피처 | 활용 및 기대 효과 |
|---|---|---|
| :basketball: **에너지 (Magnitude)** | RMS, N-Peak Mag, Band Energy | 전체적인 파손 및 부하 급증 감시 |
| :basketball: **음색 (Timbre)** | MFCC (Static, \Delta, \Delta\Delta) | 기계 마찰음의 미세한 변화(Squeal 등) 포착 |

#### **향후 레이블링 및 TinyML 확장 워크플로우**
  1) **Step 1 (On-Device):** 학습 모드에서 정상 프로파일링 수행 ➔ profile.json 생성.
  2) **Step 2 (Data Logging):** 운영 중 발생하는 모든 트리거 이벤트(이상 징후)를 피처 데이터셋으로 저장.
  3) **Step 3 (Post-Labeling):** 웹 UI의 'File Explorer'에서 저장된 파일을 재생/확인 후, 관리자가 "베어링 결함", "정상 가감속" 등의 태그를 부여(Labeling).
  4) **Step 4 (Optimization):** 레이블링된 피처 데이터를 PC에서 학습시켜 최적화된 DNN/SVM 모델을 생성 ➔ 다시 T20 기기로 배포(Model Update).

# :one: 100. 후보 :car: :car: :car: :car:

T20 시스템을 단순한 데이터 로거에서 **현장의 맥락을 스스로 이해하는 '지능형 모니터링 노드'**로 진화시키기 위한 상세 설계를 진행합니다.
이번 설계의 핵심은 **"상태 기반 적응형 지능(State-Aware Adaptive Intelligence)"**입니다. 설비의 정상적인 작동 변화(가속, 감속, 부하 변동)를 고장으로 오판하지 않으면서, 아주 미세한 충격이나 이상 징후는 놓치지 않는 구조를 지향합니다.
### 1. 트리거 히스테리시스 (Hysteresis) 및 디바운스 강화 설계
단순한 임계값 비교는 노이즈가 섞인 경계값에서 파일시스템에 큰 부담을 줍니다. 이를 위해 **'상태 확정(Confirmation)'** 로직을 도입합니다.
 * **구조체 보완 (ST_T20_TriggerCtx_t):**
   * bool is_latched: 현재 트리거 유지 여부.
   * uint8_t on_counter / off_counter: 상태 전이 확정을 위한 프레임 카운터.
 * **로직 상세:**
   * **상태 진입 (Latching ON):** Current\_Energy > Threshold_{High} 조건이 **3프레임 연속** 만족될 때 즉시 is_latched = true로 설정하고 기록 시작.
   * **상태 유지:** Energy가 Threshold_{Low} (Threshold_{High} \times 0.85) 위를 유지하는 동안은 무조건 기록 유지.
   * **상태 해제 (Latching OFF):** Current\_Energy < Threshold_{Low} 조건이 **10프레임(약 1.6초) 연속** 만족되고, 동시에 설정된 Hold_Time이 만료되어야 기록 종료.
 * **효과:** 전동기 기동 시의 일시적 서지나 외부 간섭으로 인한 '짧은 파일 파편화'를 완벽히 방어합니다.
### 2. 설비 상태 인식 (Machine State Awareness) 설계
설비의 현재 모드(OFF / IDLE / RUN)를 먼저 파악해야 임계값의 의미가 생깁니다. 이미 구현된 **MFCC와 Band Energy**를 특징량(Feature)으로 활용합니다.
 * **상태 분류 로직 (Feature-based Classification):**
   1. **OFF (전원 꺼짐):** RMS < Noise\_Floor 이며 주파수 스펙트럼이 완전히 평탄(Spectral Flatness \approx 1)한 상태.
   2. **IDLE (대기):** RMS는 낮으나 특정 기계적 기본 주파수(예: 60Hz 팬 진동)가 관찰됨. MFCC의 첫 번째 계수(C_0)가 일정 범위 내 유지.
   3. **RUN (가동):** 전체 RMS가 급증하고, MFCC 벡터의 변동성(Variance)이 커지며 지배적 주파수(Dominant Freq)가 이동함.
 * **속도 변화(VFD 제어 등) 대응:**
   * 설비 속도가 변하면 주파수 피크가 이동합니다. 이를 '이상'으로 보지 않도록 **'피크 추적 윈도우(Peak Tracking Window)'**를 설정하여, 지배 주파수가 이동하더라도 전체 에너지 보존 법칙에 따라 상태가 'RUN'임을 유지합니다.
### 3. 적응형 임계값 (Adaptive Threshold) 및 프로파일링 학습
학습(NOISE_LEARN) 시 단순히 평균 하나만 구하는 것이 아니라, **상태별 정상 범위를 프로파일링**합니다.
 * **멀티-스테이트 프로파일링 (State Profiling):**
   * NOISE_LEARN 버튼을 누르면 기기는 일정 시간 동안 데이터를 수집하여 IDLE_Profile과 RUN_Profile을 각각 생성합니다.
   * 저장 데이터: 각 상태별 RMS, Band\_Energy[3], MFCC\_Mean[13], MFCC\_Std[13].
 * **수학적 임계값 결정:**
   * 현재 인식된 상태에 맞는 프로파일을 불러옵니다.
   *    * 만약 설비가 RUN 상태라면 IDLE 때보다 높은 임계값을 자동으로 적용하여 **"작업 중 발생하는 정상 진동"**에 의한 오동작을 방지합니다.
 * **노이즈 플로어 추적 (Continuous Tracking):**
   * 기록 중이 아닐 때, 아주 느린 Alpha 필터(\alpha = 0.001)를 통해 배경 노이즈의 변화를 실시간 업데이트하여 계절적/환경적 변화를 흡수합니다.
### 4. 피크 홀드(Peak-Hold) 기반 충격 탐지 설계 (Detailed)
RMS는 시간 평균이므로 날카로운 '퍽' 소리나 금속 충격(Impact)을 희석시킵니다. 이를 위해 **고주파 통과 필터(HPF) 경로**를 별도로 설계합니다.
 * **상세 파이프라인:**
   1. **HPF 전처리:** 200Hz 이상의 고역 통과 필터(IIR HPF)를 적용하여 모터의 웅웅거리는 기본 진동(저주파)을 제거합니다.
   2. **절대값 변환 (Rectification):** y[n] = |x_{hpf}[n]|
   3. **피크 검출:** 현재 프레임(256 샘플) 내에서 가장 큰 값 Max\_Abs를 추출합니다.
   4. **임계값 비교:** Max\_Abs > Threshold_{Impact} 이면 RMS 수치와 상관없이 즉시 트리거를 가동합니다.
 * **HPF 방식의 이점:**
   * 베어링의 초기 크랙이나 볼트 풀림에 의한 '딱' 하는 금속성 타격음은 주로 고주파 영역에 에너지가 집중됩니다. HPF 경로는 저주파 노이즈에 의한 마스킹 효과를 제거하여 **충격 탐지의 감도를 5배 이상 향상**시킵니다.
   * 이 로직은 **프리트리거(Pre-trigger)**와 연동되어, 충격이 감지된 시점의 **4초 전 데이터부터** 즉시 기록을 시작하게 합니다.
