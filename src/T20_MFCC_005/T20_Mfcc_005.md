1) 분류기 입력 벡터 vs ring buffer 시퀀스 형태 비교

지금까지 만든 39차원 벡터는 보통 이렇게 씁니다.

1프레임 기준 입력:
MFCC(13) + Delta(13) + DeltaDelta(13) = 39


이 방식의 장점은 단순합니다.

계산량이 작음

메모리 사용이 적음

전통적인 ML 분류기(SVM, RandomForest, 작은 MLP)에 바로 넣기 좋음

지연(latency)이 작음


단점도 분명합니다.

“시간 흐름” 정보가 약함

충격 후 감쇠, 반복 패턴, 베어링 이상음의 주기성 같은 연속성 패턴을 충분히 못 담음

프레임 하나가 튀면 입력 자체가 흔들림


반대로 ring buffer 시퀀스 형태는 최근 여러 프레임의 특징 벡터를 연속으로 쌓아 넣는 구조입니다. 예를 들면:

프레임당 39차원

최근 8프레임 유지

최종 입력: 8 x 39 = 312 값


장점은 이렇습니다.

시간 변화 패턴을 반영 가능

LSTM/GRU/1D-CNN/Temporal CNN 같은 시계열 모델에 적합

단일 프레임 노이즈에 덜 민감

delta/delta-delta 외에도 실제 시간 순서를 유지함


단점은:

RAM 사용량 증가

모델 입력 크기 증가

초기 몇 프레임은 warm-up 필요

실시간 시스템에서 큐/복사 구조를 잘 설계해야 함


실무적으로는 이렇게 정리하면 됩니다.

클래식 분류기 / 빠른 추론 / 작은 모델이면 단일 39차원 벡터

패턴 지속시간이 중요 / AI 모델 사용 / 이상징후 temporal signature 중요이면 ring buffer 시퀀스


진동 이상탐지, 상태분류, 충격패턴 분류 쪽이면 저는 ring buffer 시퀀스 확장 쪽을 권장합니다.

2) 권장 확장안: “39차원 프레임 벡터”를 “ring buffer 시퀀스”로 확장

가장 무난한 구조는 아래입니다.

프레임 특징 벡터: 39차원

시퀀스 길이: 8 또는 16

저장 방식: 원형 버퍼(ring buffer)

모델 입력 시:

2D 텐서 해석: [time_steps][feature_dim]

또는 1D 평탄화: [time_steps * feature_dim]



예시:

feature_dim = 39

seq_len = 8

input_shape = [8][39]

평탄화 시 312


추천 기본값은:

경량 실시간 분류: seq_len = 8

패턴 안정성 우선: seq_len = 16


1.6kHz / FFT 256이면 프레임 길이는 대략 160ms입니다.
hop size를 128로 줄여 50% overlap을 쓰면 프레임 간격은 약 80ms가 됩니다. 이 경우:

seq_len = 8 → 약 640ms 문맥

seq_len = 16 → 약 1.28초 문맥


진동 이벤트 인식에는 꽤 적당한 길이입니다.

3) ring buffer 시퀀스 구조를 왜 추천하나

8) 메모리 관점 검토

예를 들어

SEQ_LEN = 8

FEATURE_DIM = 39

float 4바이트


필요 메모리:

8 * 39 * 4 = 1248 bytes


SEQ_LEN = 16이어도:

16 * 39 * 4 = 2496 bytes


즉, ring buffer 시퀀스 자체 메모리는 크지 않습니다.
실제로 더 무거운 건 FFT 버퍼, mel bank, 작업용 scratch buffer 쪽입니다.

그래서 시퀀스 확장은 부담이 적은 편입니다.
