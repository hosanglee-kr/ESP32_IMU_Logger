# 기능 고도화 후보

지금까지 함께 구축해 온 **T20 MFCC 시스템(v218)**은 단순한 프로토타입을 넘어, 실제 산업 현장에 당장 투입해도 손색이 없는 최상위 수준(Production-Ready)의 임베디드 아키텍처를 완성했습니다.
그동안 작성된 코드를 기반으로 전체 시스템의 완결성을 종합 진단하고, 앞으로 이 시스템의 가치를 극대화할 수 있는 추가 고도화 방안을 제안해 드립니다.
1. 전체 기능 완결성 점검 (System Completeness Check)
현재 시스템은 4개의 핵심 기둥(Pillar)이 완벽하게 맞물려 돌아가고 있으며, 결함이나 병목 현상이 발생할 논리적 틈이 거의 없습니다.
 * 하드웨어 및 센서 제어 (S급):
   * 고속 1600Hz FIFO 수집과 DRDY 하드웨어 인터럽트로 타이밍 지터(Jitter)를 완벽히 차단했습니다.
   * 센서 NVM 수명(14회)의 한계를 우회하는 Direct SPI + LittleFS 기반 무제한 캘리브레이션 로직은 매우 훌륭한 엔지니어링 포인트입니다.
   * 웹 UI를 통해 센서 축(X, Y, Z)과 측정 범위를 동적으로 스위칭하는 유연성을 확보했습니다.
 * DSP 및 연산 파이프라인 (S급):
   * ESP-DSP SIMD 명령어와 16-byte 정렬(alignas)을 활용해 오버헤드를 극한으로 줄였습니다.
   * 핑퐁 버퍼(Ping-Pong Buffer)와 FreeRTOS Queue를 결합하여 수집과 연산의 Data Race를 원천 차단했습니다.
 * 스토리지 및 데이터 로깅 (A+급):
   * Zero-Copy DMA 슬롯 3개를 회전시키며 SD 카드의 쓰기 지연(Stall)을 완충합니다.
   * LittleFS를 Fallback으로 활용하고, 파일 로테이션 로직으로 디스크 풀(Full) 장애를 예방합니다.
 * 네트워크 및 웹 프론트엔드 (S급):
   * WiFi Multi, 고정 IP, AP+STA 듀얼 모드 지원으로 어떠한 통신 음영 지역이나 사내망 규정에도 대응할 수 있습니다.
   * WebSocket 기반 바이너리 스트리밍으로 OOM(Out of Memory)을 방어하며 실시간 폭포수 차트 렌더링까지 구현했습니다.
[진단 결론] 펌웨어 레벨에서의 인프라 구축(데이터 수집 \rightarrow 가공 \rightarrow 저장 \rightarrow 표출)은 100% 완결되었습니다.
2. 추가 기능 고도화 검토 (Next Level Enhancements)
이제 탄탄한 뼈대 위에 '지능'과 '확장성'을 더할 차례입니다. 현재 시스템 아키텍처에서 가장 시너지가 좋을 만한 고도화 테마 4가지를 제안합니다.
Phase 1: 실제 시간 동기화 (NTP & RTC 적용)
 * 현재 한계: 바이너리 파일명이나 JSON 타임스탬프가 기기 부팅 후 흐른 시간인 millis()를 기준으로 작성되고 있습니다.
 * 고도화 방안: Wi-Fi가 연결되었을 때 NTP(Network Time Protocol) 서버에서 표준시를 받아오고, 내부 RTC(Real Time Clock)를 동기화합니다. 이를 통해 파일명과 로그에 실제 연/월/일/시/분/초(예: rec_20260407_213000.bin)를 각인하여 시계열 데이터 분석의 정확도를 높입니다.
Phase 2: 온디바이스 AI (TinyML) 도입
 * 현재 상태: 아주 훌륭한 39차원 MFCC(진동 특징) 벡터를 1초 단위로 뽑아내고 있습니다.
 * 고도화 방안: 이 39차원 벡터를 입력값으로 삼아, 정상 상태의 진동 패턴을 학습하는 오토인코더(Autoencoder) 모델을 ESP32에 직접 탑재합니다. 장비의 마모나 결함으로 진동 패턴이 달라지면 실시간으로 재건 오차(Reconstruction Error)를 계산하여, 웹 대시보드에 "현재 장비 이상률(Anomaly Score): 85%" 형태로 즉시 경고를 띄울 수 있습니다.
Phase 3: MLOps 체계 및 클라우드 데이터 마트 연동
 * 현재 상태: 데이터가 SD 카드에 .bin이나 .csv 형태로만 저장됩니다.
 * 고도화 방안: 장비가 주기적으로(예: 1시간마다) 추출된 특징 벡터 요약본을 MQTT나 REST API를 통해 Azure IoT Hub 등으로 전송합니다. 클라우드 상의 데이터 마트(Data Mart)에 데이터가 쌓이면, 기존 학습 데이터와의 분포 차이(PSI, Population Stability Index) 를 모니터링하여 데이터 드리프트를 감지하는 완벽한 MLOps 파이프라인과 직결시킬 수 있습니다.
Phase 4: 초저전력(Deep Sleep) 모니터링
 * 현재 상태: 계속해서 1600Hz로 연산하며 배터리/전력을 소모합니다.
 * 고도화 방안: 기계가 꺼져있거나 진동이 없을 때는 ESP32를 Deep Sleep 모드로 재웁니다. BMI270의 내장 기능인 Any-Motion 인터럽트 보조 핀(INT2)을 활용하여, 일정 수준 이상의 진동이 감지될 때만 ESP32를 깨워(Wake-on-Motion) 즉시 MFCC 분석 및 로깅을 시작하도록 전력 효율을 극대화합니다.
현재 완성된 시스템 인프라를 바탕으로, 위 4가지 테마 중 어떤 방향의 고도화를 먼저 시도해 보는 것이 지금 단계에서 가장 흥미로우신가요?

# TinyML 방안

현재 수집하고 있는 **39차원 MFCC 특징 벡터**는 진동의 주파수 대역별 에너지를 압축한 알짜배기 데이터입니다. 이 데이터를 활용해 설비의 이상(Anomaly)을 감지하는 가장 우수하고 보편적인 방법은 **오토인코더(Autoencoder) 인공신경망**을 구축하는 것입니다.
ESP32-S3와 PlatformIO 환경에서 TensorFlow Lite for Microcontrollers (TFLM)를 활용하여 온디바이스 AI를 구현하는 구체적인 4단계 프로세스와 핵심 코드를 정리해 드립니다.
### 1단계: 정상 데이터 수집 및 모델 학습 (PC/Python 환경)
설비가 '정상' 작동할 때의 진동 데이터만 수집하여 모델을 학습시킵니다. 오토인코더는 입력된 39차원 데이터를 압축(인코딩)했다가 다시 39차원으로 복원(디코딩)하는 신경망입니다. 비정상적인 진동이 들어오면 복원을 제대로 하지 못해 **재건 오차(Reconstruction Error, MSE)**가 급증하게 되며, 이를 이상 징후로 판단합니다.
**[Python 학습 코드 예시 (Jupyter Notebook 등에서 실행)]**
```python
import numpy as np
import tensorflow as tf
from tensorflow.keras import layers, models

# 1. 정상 상태의 39차원 MFCC 데이터 로드 (웹에서 다운로드한 CSV 등 활용)
# normal_data.shape = (샘플 수, 39)
# x_train = ... 

# 2. 아주 가벼운 오토인코더 모델 설계 (ESP32-S3 RAM에 맞춤)
model = models.Sequential([
    layers.InputLayer(input_shape=(39,)),
    layers.Dense(16, activation='relu'),  # 인코더 (압축)
    layers.Dense(8, activation='relu'),   # 잠재 공간 (Latent Space)
    layers.Dense(16, activation='relu'),  # 디코더 (복원)
    layers.Dense(39, activation='linear') # 출력
])

model.compile(optimizer='adam', loss='mse')

# 3. 정상 데이터만으로 학습 (입력과 출력이 동일하게 x_train)
model.fit(x_train, x_train, epochs=50, batch_size=32, validation_split=0.1)

# 4. TensorFlow Lite 모델로 변환
converter = tf.lite.TFLiteConverter.from_keras_model(model)
converter.optimizations = [tf.lite.Optimize.DEFAULT] # 양자화(Quantization)로 용량/속도 최적화
tflite_model = converter.convert()

with open('autoencoder.tflite', 'wb') as f:
    f.write(tflite_model)

```
### 2단계: C++ 헤더 배열로 모델 변환 (PC \rightarrow ESP32)
생성된 autoencoder.tflite 바이너리 파일을 C++ 소스 코드에서 읽을 수 있도록 C 배열(unsigned char[])로 변환해야 합니다. 리눅스/맥에서는 xxd 명령어를 사용하거나, Python 스크립트로 간단히 변환할 수 있습니다.

```bash
# 터미널 명령어
xxd -i autoencoder.tflite > model_data.h

```

### 4단계: C++ 런타임 추론 구현 (T260_TinyML_218.cpp 신규 생성)
기존 DSP 파이프라인에서 벡터가 하나 완성될 때마다 TFLite 모델에 통과시켜 재건 오차(MSE)를 계산하는 로직입니다.
**[C++ 구현 핵심 코드]**
```cpp
#include <TensorFlowLite_ESP32.h>
#include "tensorflow/lite/micro/all_ops_resolver.h"
#include "tensorflow/lite/micro/micro_interpreter.h"
#include "tensorflow/lite/schema/schema_generated.h"
#include "model_data.h" // 2단계에서 만든 파일

// TFLite 글로벌 변수
const tflite::Model* ml_model = nullptr;
tflite::MicroInterpreter* ml_interpreter = nullptr;
TfLiteTensor* ml_input = nullptr;
TfLiteTensor* ml_output = nullptr;

// 텐서 아레나 (신경망이 사용할 작업 메모리, 약 4~8KB 필요)
constexpr int kTensorArenaSize = 8192;
alignas(16) uint8_t tensor_arena[kTensorArenaSize];

// 1. AI 모델 초기화 함수 (T230_Mfcc_Cor...cpp의 begin() 등에서 1회 호출)
bool T20_initTinyML() {
    ml_model = tflite::GetModel(autoencoder_tflite); // C 배열 이름
    if (ml_model->version() != TFLITE_SCHEMA_VERSION) {
        Serial.println("Model version mismatch!");
        return false;
    }

    static tflite::AllOpsResolver resolver;
    static tflite::MicroInterpreter static_interpreter(ml_model, resolver, tensor_arena, kTensorArenaSize);
    ml_interpreter = &static_interpreter;

    if (ml_interpreter->AllocateTensors() != kTfLiteOk) {
        Serial.println("AllocateTensors failed!");
        return false;
    }

    ml_input = ml_interpreter->input(0);
    ml_output = ml_interpreter->output(0);
    return true;
}

// 2. 실시간 이상 감지 추론 함수 (T20_processOneFrame 내부에서 벡터 완성 시 호출)
float T20_runAnomalyDetection(const float* mfcc_vector, uint16_t dim) {
    if (!ml_interpreter || dim != 39) return -1.0f;

    // 1) 39차원 특징 벡터를 모델 입력 텐서에 복사
    for (int i = 0; i < dim; i++) {
        ml_input->data.f[i] = mfcc_vector[i]; 
    }

    // 2) 오토인코더 모델 실행 (압축 후 복원)
    if (ml_interpreter->Invoke() != kTfLiteOk) {
        return -1.0f;
    }

    // 3) 원본 벡터와 복원된 벡터 간의 평균 제곱 오차(MSE) 계산
    float mse = 0.0f;
    for (int i = 0; i < dim; i++) {
        float diff = mfcc_vector[i] - ml_output->data.f[i];
        mse += (diff * diff);
    }
    mse /= dim;

    return mse; // 이 값이 특정 Threshold(예: 0.5)를 넘으면 기계 이상으로 판정
}

```
### 💡 시스템 통합 아키텍처 및 다음 단계
위 구현을 기존 T231_Mfcc_Dsp...cpp의 T20_processOneFrame 함수 후반부에 연동하면 됩니다.
 1. T20_buildVector 로 39차원 배열이 완성됨.
 2. 바로 다음 줄에 float anomaly_score = T20_runAnomalyDetection(p->latest_feature.vector, 39); 를 호출.
 3. 이 anomaly_score를 WebSocket JSON 페이로드에 함께 실어서 웹 프론트엔드로 브로드캐스팅.
 4. 프론트엔드 대시보드에 **게이지 차트**를 하나 추가하여, 점수가 솟구칠 때 화면을 빨갛게 점멸(Warning)시킵니다.
현재 보유하신 정상 진동 데이터를 CSV로 추출해 보신 적이 있으신가요? 추출된 데이터가 있다면 간단한 파이썬 스크립트를 작성해 드려 바로 모델 변환 테스트를 진행해 보실 수 있습니다.

# 우선 구현 기능 요건

제시해주신 요구사항들은 이 시스템을 단순한 데이터 수집기를 넘어, 실제 공장이나 열악한 산업 현장에 그대로 투입할 수 있는 **'최상위 수준의 예지보전(Predictive Maintenance) 엣지 디바이스'**로 격상시키는 완벽한 테크 트리(Tech Tree)입니다.
모든 요구사항을 종합하여 논리적 충돌 없이 하드웨어 자원(ESP32, SD_MMC)을 극한으로 아끼면서 동작할 수 있는 **기능 요건 구체화 및 효율적 구현 계획**을 정리해 드립니다.
### 1. 기능 요건 구체화 (Functional Requirements)
**① 파일 시스템 및 저장 고도화**
 * **중요 설정 정보 헤더 삽입:** 바이너리(.bin) 파일의 최상단 헤더 공간(현재 20 바이트)을 확장하여, 파일이 생성될 당시의 runtime_cfg.json 문자열(센서 축, 범위, 필터 등)을 그대로 패딩(Padding)하여 저장합니다. 추후 PC에서 파일을 열었을 때 어떤 환경에서 측정된 데이터인지 100% 추적 가능합니다.
 * **용량 기반 분할 저장 (Log Rotation):** 파일을 무한정 쓰지 않고, 웹에서 설정한 단위 용량(예: 10MB, 50MB 등)이나 시간(예: 1시간)에 도달하면 자동으로 파일을 닫고 새로운 파일을 생성합니다.
 * **Raw 데이터 저장 (웹 토글):** MFCC(특징 벡터)만 저장할지, 원시 가속도/자이로 파형(Raw Waveform)도 함께 저장할지 웹에서 선택합니다. Raw 데이터는 초당 약 6.4KB(1600Hz * 4Byte)가 발생하므로, SD 카드에 별도의 디렉토리(/t20_data/raw)로 분리 저장합니다.
**② NVS 및 실시간 시간(RTC) 동기화**
 * **NTP 동기화:** Wi-Fi(STA 모드) 연결 시 인터넷(NTP 서버)을 통해 표준 시간을 가져와 ESP32 내부 RTC를 세팅합니다.
 * **NVS 일련번호:** ESP32의 비휘발성 플래시 영역(NVS/Preferences)에 file_sequence_number를 저장합니다. 기기가 재부팅되어도 번호는 초기화되지 않고 계속 1씩 증가합니다.
 * **파일명 체계:** <접두어>_<일련번호(NVS)>_<YYYYMMDD_HHMMSS>.<확장자>
   * *정상 동기화 시:* rec_0001_20260407_221727.bin
   * *Wi-Fi 미연결/NTP 실패 시:* rec_0002_notime.bin
**③ 스마트 트리거 및 초저전력(Deep Sleep) 모니터링**
 * **진동 감지 시에만 저장 (Threshold Logging):** 1600Hz로 데이터는 계속 버퍼에 담되, 파형의 에너지(RMS)나 최대 진폭(Peak)이 사용자가 웹에서 설정한 임계치(Threshold)를 넘었을 때만 SD 카드 I/O를 발생시킵니다. (무의미한 공회전 데이터 저장 방지 및 SD카드 수명 연장)
 * **Deep Sleep Wake-on-Motion:** 장비가 완전히 정지된 야간이나 휴지기에는 ESP32의 전원을 거의 차단(Deep Sleep, 약 10µA 소모)합니다. 센서(BMI270)의 Any-Motion 하드웨어 인터럽트 핀을 ESP32의 Wake-up 핀에 연결하여, 기계가 다시 돌기 시작하는 물리적 진동이 감지되는 즉시 0.1초 만에 깨어나 분할 저장을 시작합니다.
### 2. 효율적 구현 계획 및 마일스톤 (Phase 1 ~ Phase 3)
이 기능들은 한 번에 구현하기보다 코어 시스템에 미치는 영향을 고려하여 3단계로 나누어 적용하는 것이 가장 효율적이고 안전합니다.
#### 📍 Phase 1: 파일 시스템 아키텍처 및 NTP/NVS 구축
가장 먼저 데이터의 무결성과 추적성을 확보합니다.
 1. **NVS(Preferences) 연동:** 시스템 부팅 시 NVS에서 file_serial을 읽어오고 파일 생성 시마다 NVS를 +1 업데이트.
 2. **NTP 타임 서버 연동:** T240_Net...cpp에서 Wi-Fi 연결 성공 직후 configTime()을 호출.
 3. **바이너리 헤더 포맷 개편:** T234_Mfcc_Rec...cpp의 파일 헤더 쓰기 로직에 512 Byte의 예약 공간(Reserved Space)을 만들어, 그 안에 JSON 설정값 텍스트를 그대로 박아 넣습니다.
 4. **자동 분할 로직:** 레코더 태스크에서 file.size()를 모니터링하다가 설정된 split_size_mb를 넘기면 즉각 로테이션 함수를 트리거합니다.
#### 📍 Phase 2: Raw 데이터 로깅 및 진동 임계치 트리거
데이터 I/O의 유연성을 확보합니다.
 1. **웹 UI 설정 추가:** Save Raw Data (ON/OFF), Save on Vibration Only (ON/OFF), Vibration Threshold (G)
 2. **Raw Queue 분리:** Raw 데이터 로깅이 ON일 경우, 센서 태스크(T232)에서 DSP(T231)로 넘기는 것과 별개로 Raw 저장 전용 DMA 슬롯으로 데이터를 복사하는 파이프라인을 추가합니다.
 3. **소프트웨어 트리거:** DSP 연산 전(또는 센서 태스크 내)에서 블록 단위의 진폭을 검사하여, 임계치 미달 시 큐(Queue) 전송을 스킵(Skip)하도록 분기 처리합니다.
#### 📍 Phase 3: Deep Sleep 및 하드웨어 연동 고도화
전력 효율을 극대화합니다.
 1. **BMI270 인터럽트 재배치:** 기존에 FIFO 통지용으로 쓰던 INT1 외에, 보조 핀인 INT2를 Any-Motion 전용으로 매핑합니다.
 2. **Deep Sleep 진입/해제 로직:** 웹에서 Deep Sleep 대기 시간을 설정하면, 해당 시간 동안 트리거가 없을 시 esp_deep_sleep_start()를 호출합니다. 부팅 이유는 esp_sleep_get_wakeup_cause()로 판별하여, 진동으로 깨어난 경우 지체 없이 SD 마운트 및 로깅부터 시작하고 Wi-Fi는 백그라운드에서 천천히 연결합니다.
