이번 리팩토링에서 한 작업은 단순 코드 정리가 아니라, **“임베디드 시스템 전반에 적용 가능한 구조 개선 패턴”**입니다.
다른 프로젝트에도 그대로 적용할 수 있도록 일반화해서 정리해드리면 아래와 같습니다.


---

1. 상태 관리 패턴 일반화

❌ 기존 문제 패턴

#define XXX_STATE_IDLE

모듈마다 동일 상태 반복 정의

상태 의미는 같지만 이름만 다름


✅ 개선 패턴

① 공통 상태 enum

typedef enum {
    STATE_IDLE = 0,
    STATE_READY,
    STATE_RUNNING,
    STATE_DONE,
    STATE_ERROR
} State_t;

② 모듈별 상태 구조체

typedef struct {
    State_t init;
    State_t exec;
    State_t io;
} ModuleState_t;

핵심 원칙

상태는 공통화

상태 변수는 분리

의미는 재사용



---

2. 매크로 → 타입 기반 설계

❌ 문제

#define OK 1
#define FAIL 0
#define SAMPLE_RATE 1600

✅ 개선

① 결과 타입

typedef enum {
    RESULT_FAIL = 0,
    RESULT_OK
} Result_t;

② 상수는 const

static const float SAMPLE_RATE = 1600.0f;

핵심 원칙

매크로 대신 타입 사용

컴파일러가 검증 가능하도록



---

3. Magic Number 제거 패턴

❌ 문제

buffer[16];
frame[1024];

✅ 개선

구조체로 그룹화

typedef struct {
    uint16_t queue_depth;
    uint16_t frame_size;
} SystemLimits_t;

핵심 원칙

숫자는 의미를 가져야 한다

관련 값은 묶는다



---

4. ISR / Queue 구조 패턴

❌ 문제

ISR에서 직접 처리

전역 변수 남발

race condition 위험


✅ 개선

Ring Buffer

typedef struct {
    volatile uint8_t w;
    volatile uint8_t r;
    uint8_t buf[N];
} RingBuffer_t;

핵심 원칙

ISR은 최소 작업만

데이터 전달은 Queue

lock-free 구조 우선



---

5. 상태 디버깅 패턴

❌ 문제

printf("%d\n", state);

✅ 개선

const char* StateToStr(State_t s);

핵심 원칙

상태는 사람이 읽을 수 있어야 한다

로그는 디버깅 도구다



---

6. 상태 머신 구조화

❌ 문제

if-else 난무

흐름 추적 어려움


✅ 개선

void StateMachine(Module_t* m)
{
    switch (m->state) {
        case STATE_IDLE: ...
        case STATE_READY: ...
    }
}

핵심 원칙

흐름은 함수로 만든다

상태 전이는 명확하게



---

7. 구조체 중심 설계

❌ 문제

전역 변수 많음

모듈 경계 불명확


✅ 개선

typedef struct {
    Config_t config;
    State_t state;
    Buffer_t buffer;
} Module_t;

핵심 원칙

데이터 + 상태 + 설정 = 하나의 객체

C에서도 “객체지향적 구성”



---

8. 명명 규칙 표준화

추천 규칙

종류	규칙

구조체	ST_XXX
enum	EM_XXX
전역 상수	G_XXX
함수	Module_Action()


핵심 원칙

이름만 보고 역할이 보여야 한다

프로젝트 전체 일관성 유지

핵심 원칙

10. 메모리 설계 패턴

❌ 문제

MAX 배열 고정

RAM 낭비


✅ 개선 옵션

① 길이 + 고정 배열

uint16_t len;
float data[MAX];

② 포인터 기반

float* data;

핵심 원칙

“최대 크기”와 “실제 사용 크기” 분리

🔥 핵심 설계 철학 (요약)

1. 반복은 추상화하라

→ 상태, 결과, 구조

2. 숫자는 숨기지 말고 구조화하라

→ Magic Number 제거

3. 흐름은 코드가 아니라 구조로 표현하라

→ 상태머신

4. ISR은 최소화하라

→ Queue 기반 설계

5. C에서도 객체처럼 설계하라

→ struct 중심
