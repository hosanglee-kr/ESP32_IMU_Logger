유도모터의 진동이나 소리 신호를 분석할 때 MFCC는 아주 강력한 도구입니다. 복잡한 수식보다는 모터가 돌아갈 때 우리가 귀로 듣는 **'소리의 질감'**과 **'기계적 상태'**를 연결하여 39차원(또는 13차원)의 의미를 물리적으로 해석해 보겠습니다.
## 1. MFCC 13차원의 물리적 해석 (모터 관점)
MFCC의 각 차원은 소리의 '에너지 분포 곡선'을 설명합니다. 모터 신호에서 각 숫자가 갖는 물리적 의미는 다음과 같습니다.
### **MFCC 0번 (에너지/음량)**
 * **의미:** 모터 진동의 전체적인 **세기(Energy)**입니다.
 * **해석:** 모터가 얼마나 세게 돌고 있는지, 전력량이 얼마나 들어가는지를 나타냅니다. 속도가 올라가면 기본적으로 이 값이 커집니다. 만약 속도가 일정한데 이 값이 튄다면, 전체적인 진동 수준이 높아진 것이므로 **'과부하'**나 **'전체적인 고정 불량'**을 의심할 수 있습니다.
### **MFCC 1번 ~ 2번 (저주파 성분: 굵직한 흐름)**
 * **의미:** 소리의 전체적인 **기울기**와 **무게중심**입니다.
 * **해석:** 모터의 웅~ 하는 낮은 소리의 비중을 뜻합니다.
   * **1번:** 저음과 고음의 밸런스입니다. 이 값이 변하면 모터에서 발생하는 '웅' 소리의 톤이 바뀐 것입니다.
   * **2번:** 소리의 굴곡(Basin)을 나타내며, 주로 **축 정렬(Misalignment)**이나 **회전자 불평형(Unbalance)**처럼 큼직한 물리적 변형이 생길 때 민감하게 반응합니다.
### **MFCC 3번 ~ 12번 (중/고주파 성분: 세부 질감)**
 * **의미:** 소리의 에너지가 어디서 '뾰족하게' 튀는지, 즉 **배음(Harmonics)** 구조를 나타냅니다.
 * **해석:** 모터 내부의 미세한 부품 상태를 뜻합니다.
   * **중간 번호:** 베어링 내부의 미세한 흠집이나 윤활 상태 부족으로 발생하는 '서걱거리는 소리'의 패턴을 담습니다.
   * **높은 번호:** 아주 미세한 전기적 노이즈나 공극(Air gap)의 불균형으로 발생하는 고주파 진동 패턴을 요약합니다.
## 2. 39차원으로 확장했을 때의 의미 (동적 변화)
보통 MFCC 13개에 **Delta(\Delta)** 13개, **Delta-Delta(\Delta\Delta)** 13개를 더해 39차원을 만듭니다.
 * **MFCC (13개):** 현재 모터의 **'정지 화면'** 같은 상태 (현재 음색)
 * **Delta (13개):** 소리가 변화하는 **'속도'** (음색이 얼마나 빨리 변하는가?)
 * **Delta-Delta (13개):** 소리 변화의 **'가속도'** (변화의 리듬이 일정한가?)
**물리적 해석:** 가속/감속 구간에서 **Delta** 값들은 모터가 정상적으로 속도를 올리고 있는지 판단하는 기준이 됩니다. 만약 속도를 올리는 중인데 특정 차원의 Delta 값이 비정상적으로 크다면, 가속 중에만 발생하는 **특정 주파수 대역의 공진**이나 **슬립(Slip) 현상**을 포착할 수 있습니다.
## 3. 가/감속 반복 시 룰 기반 이상감지 설계
가속, 등속, 감속이 반복되는 공정에서는 **'속도(0차원 에너지)'**에 따라 나머지 MFCC 값들의 정상 범위가 계속 변합니다. 따라서 고정된 임계치(Threshold)를 쓰면 가속 구간을 무조건 이상으로 감지하는 오류가 생깁니다.
### **지표 계산식 제안: 속도 가중 정규화 거리**
속도(MFCC 0번)를 독립변수로 보고, 나머지 차원들과의 관계를 이용해 **'상대적 이상 점수'**를 산출해야 합니다.
**[지표 설계 개념]**
현재의 속도(MFCC_0)에서 기대되는 정상적인 음색 패턴과의 거리를 계산합니다.
 * MFCC_i(t): 현재 시점 t의 i번째 MFCC 값
 * \mu_{i, speed}: 현재 속도(MFCC_0) 대역에서 학습된 i번째 MFCC의 **평균값**
 * \sigma_{i, speed}: 현재 속도(MFCC_0) 대역에서 학습된 i번째 MFCC의 **표준편차**
 * w_i: 각 차원별 가중치 (베어링 고장은 고차원에, 축 불평형은 저차원에 높은 가중치 부여)
### **실제 적용 룰 (Rule-based Logic)**
 1. **속도 구간화 (Binning):** MFCC_0 값을 기준으로 구간을 나눕니다. (예: 0~10, 11~20...)
 2. **동적 프로파일링:** 각 구간별로 정상 상태의 MFCC 1~12번의 평균과 편차를 미리 저장해둡니다.
 3. **변화율 감시 (Delta Check):** 등속 구간임에도 불구하고 Delta(변화율) 값이 특정 임계치를 넘으면 '불안정 상태'로 간주합니다.
 4. **최종 판정:**
   * **주의:** Anomaly Score가 임계치를 3회 연속 초과할 때
   * **위험:** 특정 고주파 대역 MFCC(9~12번)가 급격히 상승하며 MFCC_0와의 상관관계가 깨질 때
**요약하자면:**
가속 중에는 소리가 커지는 것이 당연하므로, **"소리가 커진 만큼 음색(MFCC 패턴)도 정상적으로 변하고 있는가?"**를 체크하는 것이 핵심입니다. MFCC_0를 '기준점'으로 삼아 나머지 값들이 그 궤적을 잘 따르는지 확인하는 공식을 적용해 보시기 바랍니다.

유도모터의 가변 속도(가속/등속/감속) 환경에서 MFCC를 활용한 이상감지 지표와 관련 연구, 사례들을 정리해 드립니다. 최근 연구들은 단순히 고정된 임계치를 넘는 것을 넘어, **속도에 따른 동적 임계치(Dynamic Threshold)**와 **다차원 거리 지표**를 결합하는 추세입니다.
## 1. 주요 이상감지 지표 계산식 (통계 및 거리 기반)
룰 기반 시스템에서 속도 변화를 고려할 때 가장 많이 인용되는 지표들입니다.
### **① 마할라노비스 거리 (Mahalanobis Distance, D_M)**
MFCC의 13차원 혹은 39차원 데이터는 서로 상관관계가 높습니다. 유클리드 거리는 이 관계를 무시하지만, 마할라노비스 거리는 **공분산(Covariance)**을 고려하여 '정상 궤적'에서 얼마나 벗어났는지 계산합니다.
 * x: 현재 시점의 MFCC 벡터
 * \mu: 해당 속도 구간에서의 정상 MFCC 평균 벡터
 * S^{-1}: 공분산 행렬의 역행렬
 * **특징:** 변수 간의 상관관계를 반영하므로, 특정 속도에서 발생해야 할 음색 패턴이 깨졌을 때 매우 민감하게 반응합니다.
### **② 누적 합계 지표 (Accumulator/CUSUM)**
단일 데이터 포인트의 튀는 현상(Noise)에 의한 오탐을 줄이기 위해 사용합니다. Google 등에서 시계열 이상감지에 주로 제안하는 방식입니다.
 * z_t: 현재의 표준화 점수(Z-score)
 * k: 허용 오차(Slack)
 * **특징:** 미세한 이상 징후가 지속적으로 누적될 때 임계치를 넘도록 설계되어, 서서히 마모되는 베어링 고장 등에 유리합니다.
## 2. 관련 연구 논문 요약 (2024~2025년 최신 트렌드)
최근 연구들은 **속도 가변성(Dynamic Operating Conditions)**을 해결하는 데 집중하고 있습니다.

## 3. 산업계 실용 사례
### **사례 A: STMicroelectronics의 스마트 센서 노드**
ST는 가변 속도 엔진의 이상감지를 위해 **"속도 구간별 학습"** 방식을 제안합니다.
 * **방법:** 모터의 작동 속도를 3단계로 나누고, 각 단계에서 정상 신호를 800개 이상 수집하여 모델을 최적화합니다.
 * **지표:** SVM(Support Vector Machine) 또는 머신러닝 모델의 **신뢰도 점수(Confidence Score)**를 지표로 사용하며, 이 점수가 일정 수준(예: 80%) 이하로 떨어지면 알람을 발생시킵니다.
### **사례 B: 전기차(EV) 구동 시스템의 실시간 진단**
전기차 모터는 속도 변화가 매우 극심합니다.
 * **방법:** MFCC의 0번 차원(에너지)을 속도의 대리 지표로 삼고, 이를 기준으로 **'정규화된 잔차(Normalized Residual)'**를 계산합니다.
 * **결과:** 단순 에너지 레벨은 속도에 따라 변하지만, 속도 대비 MFCC 각 성분의 비율은 정상 상태에서 일정하게 유지된다는 점을 이용해 오탐율을 획기적으로 낮췄습니다.

ESP32-S3에서 **마할라노비스 거리(Mahalanobis Distance)**를 계산할 때 가장 큰 병목 지점은 공분산 행렬의 역행렬 계산과 행렬 곱셈 연산입니다. 실시간 데이터 처리나 센서 퓨전 등을 위해 이를 최적화하는 핵심 전략을 정리해 드립니다.
## 1. 수학적 배경과 최적화 원리
마할라노비스 거리 D_M의 정의는 다음과 같습니다.
여기서 x는 관측 벡터, \mu는 평균 벡터, $S^{-1}$은 공분산 행렬의 역행렬입니다.
### 왜 역행렬을 사전 계산(Pre-computation)하는가?
역행렬 계산(S^{-1})은 $O(n^3)$의 복잡도를 가집니다. 하지만 **학습 데이터셋의 통계 특성(공분산)이 변하지 않는다면**, 이를 런타임에 매번 계산할 필요가 없습니다.
 1. **오프라인 계산:** PC(Python, MATLAB 등)에서 미리 $S^{-1}$을 구합니다.
 2. **상수화:** 계산된 결과 행렬을 ESP32-S3 코드 내에 const float 배열로 임베딩합니다.
## 2. ESP32-S3 최적화 구현 전략
### ① ESP-DSP 라이브러리 활용 (필수)
ESP32-S3는 **Xtensa® LX7 MCU**를 사용하며, AI 및 DSP 가속을 위한 **S3 전용 명령어(PIE: Processor Instruction Extensions)**를 지원합니다. 일반적인 for 루프 대신 esp-dsp 라이브러리의 행렬 연산 함수를 사용하면 속도가 비약적으로 향상됩니다.
### ② Cholesky 분해를 이용한 최적화
역행렬 $S^{-1}$을 직접 사용하는 대신, 공분산 행렬 S를 **Cholesky 분해**(S = LL^T)하여 하삼각 행렬 L을 미리 저장해두는 방법도 있습니다. 이 경우 전방 대입(Forward Substitution)을 통해 연산량을 줄일 수 있지만, 임베디드 환경에서는 S^{-1} 자체를 상수로 들고 있는 것이 가장 빠릅니다.
## 3. C++ 코드 예시 (사전 계산 적용)
미리 계산된 역행렬을 이용해 마할라노비스 거리를 구하는 최적화 구조입니다.
```cpp
#include "dsps_dotprod.h" // ESP-DSP 라이브러리
#include "esp_dsp.h"

// 1. 사전 계산된 역행렬 (예: 3x3 행렬)
const float S_inv[9] = {
    0.12, -0.05, 0.01,
    -0.05, 0.22, -0.03,
    0.01, -0.03, 0.15
};

const float mean_vec[3] = {10.5, 20.2, 5.1};

float calculate_mahalanobis(float* input_x) {
    float diff[3];
    float intermediate[3] = {0};
    float result = 0;

    // 단계 1: (x - mu) 계산
    for(int i=0; i<3; i++) diff[i] = input_x[i] - mean_vec[i];

    // 단계 2: intermediate = (x - mu)^T * S_inv
    // esp-dsp의 dsps_dotprod_f32를 사용하거나 행렬-벡터 곱 연산 수행
    for (int i = 0; i < 3; i++) {
        for (int j = 0; j < 3; j++) {
            intermediate[i] += diff[j] * S_inv[j * 3 + i];
        }
    }

    // 단계 3: result = intermediate * (x - mu)
    dsps_dotprod_f32(intermediate, diff, &result, 3);

    return sqrtf(result);
}

```
## 4. 추가 성능 향상 팁
 * **SIMD 활용:** ESP32-S3는 한 번에 여러 개의 부동 소수점 연산을 처리할 수 있습니다. dsps_matrix_mul_f32와 같은 함수는 내부적으로 어셈블리 최적화가 되어 있어 일반 루프보다 3~5배 이상 빠릅니다.
 * **메모리 정렬:** 행렬 데이터를 __attribute__((aligned(16)))으로 선언하여 메모리 접근 효율을 극대화하세요.
 * **고정 소수점 고려:** 만약 극도로 높은 샘플링 속도가 필요하다면, float 대신 int16_t나 int32_t 기반의 고정 소수점 연산으로 변환하여 PIE 가속기의 성능을 끝까지 뽑아낼 수 있습니다.
**결론적으로,** $S^{-1}$은 Python의 numpy.linalg.inv() 등으로 미리 구해서 소스코드에 박아 넣고, 런타임에는 ESP-DSP의 벡터/행렬 곱셈 함수만 호출하는 것이 ESP32-S3에서 가장 효율적인 방법입니다.

유도 모터의 진동 데이터, 특히 **MFCC(Mel-Frequency Cepstral Coefficients)** 39차원 데이터를 다루신다면 매우 정밀한 상태 모니터링을 수행 중이시군요. 결론부터 말씀드리면, **"속도가 계속 변하는 환경"에서는 공분산이 변하지 않는다고 가정하기 어렵습니다.**
그 이유와 함께, 실전에서 어떻게 접근해야 하는지 상세히 설명해 드릴게요.
## 1. 공분산(Covariance)이 변하지 않는다는 의미
통계학에서 이를 **정상성(Stationarity)**이라고 부릅니다.
 * **통계적 일관성:** 시간이 지나거나 데이터의 양이 늘어나도 변수들 사이의 "상관관계의 구조"와 "변동 폭"이 일정하다는 뜻입니다.
 * **마할라노비스 거리에서의 의미:** "정상 상태(Normal Base)"의 틀이 고정되어 있다는 것입니다. 즉, 역행렬 $S^{-1}$을 한 번만 계산해서 평생 써도 된다는 뜻이죠.
## 2. 유도 모터 진동 & MFCC 데이터의 특성
유도 모터의 속도(RPM)가 변하면 진동의 주파수 성분이 이동(Shift)합니다.
 * **MFCC의 특성:** MFCC는 특정 주파수 대역의 에너지를 캡처합니다. 속도가 변하면 기계적 진동 주파수가 바뀌고, 이는 MFCC 피처 벡터의 값뿐만 아니라 **피처 간의 상관관계(공분산)**까지 변화시킵니다.
 * **비정상성(Non-stationarity):** 속도가 가변적인 상황에서는 '정상'의 기준 자체가 속도에 따라 요동칩니다.
   * 예: 1000 RPM에서의 정상 진동 패턴과 3000 RPM에서의 정상 패턴은 공분산 구조 자체가 다릅니다.
> **결론:** 속도가 변하는 모터 데이터에서 단 하나의 공분산 행렬만 사용하면, 속도가 바뀔 때마다 마할라노비스 거리가 치솟아 **오탐(False Positive)**이 발생할 확률이 매우 높습니다.
> 
## 3. 공분산이 변하는 상황에서의 해결책
ESP32-S3의 자원을 아끼면서 이 문제를 해결하는 현실적인 방법 3가지를 제안합니다.
### ① 속도별 구간화 (Binning) - 가장 추천
속도 범위를 몇 개의 구간(예: Low, Mid, High)으로 나눕니다.
 * **사전 계산:** 각 속도 구간별로 공분산 역행렬 S_{low}^{-1}, S_{mid}^{-1}, $S_{high}^{-1}$을 미리 계산하여 ESP32-S3 플래시에 저장합니다.
 * **실행:** 현재 모터의 RPM 정보를 받아 해당되는 역행렬을 선택(Switch-case)하여 마할라노비스 거리를 계산합니다.
### ② 적응형 학습 (Online Update)
데이터가 들어올 때마다 공분산을 조금씩 업데이트하는 방식입니다.
 * **Welford's Algorithm:** 합계와 제곱합을 유지하며 역행렬을 주기적으로 갱신합니다.
 * **단점:** ESP32-S3에서 역행렬을 실시간으로 계산하는 것은 연산 부담이 큽니다. (수 초에 한 번 정도라면 가능)
### ③ 차원 축소 (PCA 후 Mahalanobis)
39차원은 ESP32-S3에게 꽤 무거운 차원입니다 (39 \times 39 행렬).
 * **PCA 적용:** 중요한 주성분 10~15개로 압축한 뒤 마할라노비스 거리를 구하세요. 연산 속도가 기하급수적으로 빨라지며, 속도 변화에 따른 노이즈에 좀 더 강건해질 수 있습니다.
## 4. 요약
속도가 변하는 유도 모터의 경우 **"공분산은 변한다"**고 보시는 것이 맞습니다.
따라서 **"속도 구간별로 최적화된 역행렬들을 미리 PC에서 계산해두고, ESP32-S3에서는 상황에 맞는 행렬을 꺼내 쓰는 방식"**이 성능과 정확도 사이의 가장 좋은 타협점입니다.
혹시 현재 모터의 RPM 정보를 ESP32에서 실시간으로 받아오실 수 있는 환경인가요?

유도 모터의 속도 가변 환경에서 MFCC 39차원 데이터를 처리하는 것은 상당히 고차원적인 프로젝트입니다. 단순히 속도만 나누는 것이 아니라, **가속/감속/등속의 '상태(Regime)'**를 어떻게 정의하고 처리할지가 핵심입니다.
## 1. MFCC 기반 속도 및 상태(가/감/등속) 구분 방법
MFCC 데이터만으로 속도와 상태를 구분하는 것은 가능하지만, 이를 위해서는 **전처리된 피처의 동적 특성**을 활용해야 합니다.
 * **MFCC의 구성 활용:** 보통 39차원은 13(Static) + 13(Delta) + 13(Delta-Delta)로 구성됩니다.
   * **Static (1~13):** 현재의 주파수 에너지 분포를 나타내며, **속도(RPM)**와 밀접합니다.
   * **Delta (14~26):** 변화율을 나타내며, **가속/감속의 지표**가 됩니다.
   * **Delta-Delta (27~39):** 가속도의 변화를 나타내며, 상태 전환 시점을 포착하기 좋습니다.
 * **분류기(Classifier) 도입:** ESP32-S3에서 단순 if문으로 39차원을 해석하기는 어렵습니다.
   * **Lite-weight NN 또는 SVM:** 39차원 MFCC를 입력받아 현재가 어느 구간(예: 1000RPM 등속, 가속 중 등)인지 먼저 판단하는 **'게이트 분류기'**를 둡니다. ESP-DL 라이브러리를 사용하면 S3에서 매우 빠르게 동작합니다.
## 2. 가속 vs 감속 구분 필요성
**결론부터 말씀드리면, "구분하는 것이 매우 권장됩니다."**
그 이유는 모터의 **슬립(Slip) 현상과 부하(Load)** 때문입니다.
 * **가속 시:** 모터는 회전자 자계를 앞지르기 위해 강한 토크를 발생시키며, 이 과정에서 전류와 진동 패턴이 등속일 때보다 훨씬 거칠고 큽니다.
 * **감속 시:** 회생 제동이 걸리거나 관성에 의해 회전하므로 토크 방향이 반대이며, 진동의 주파수 성분 결합(Coupling) 방식이 가속 때와 다릅니다.
 * **영향:** 가속 구간의 데이터를 등속 구간의 공분산 행렬(S^{-1})에 넣으면, 정상적인 가속임에도 불구하고 '이상 진동'으로 오판할 가능성이 매우 높습니다.
## 3. 유효한 접근 방법: "Regime-Specific MD"
가장 실현 가능한 워크플로우는 다음과 같습니다.
 1. **데이터 레이블링 (Off-line):**
   * RPM 센서나 인버터 정보를 동기화하여, MFCC 데이터에 [속도구간, 가속/감속/등속] 태그를 붙입니다.
 2. **군집화 및 대표 행렬 생성:**
   * 예: [Low-Acc, Low-Const, Low-Dec, Mid-Acc, ...] 식으로 약 6~9개의 '레짐(Regime)'을 정의합니다.
   * 각 레짐별로 평균(\mu)과 역공분산 행렬(S^{-1})을 계산합니다.
 3. **ESP32-S3 구현:**
   * **단계 1:** 입력된 MFCC를 통해 현재 레짐을 결정 (Small MLP 또는 KNN).
   * **단계 2:** 결정된 레짐에 해당하는 S^{-1} 행렬을 Flash에서 불러옴.
   * **단계 3:** 해당 행렬로 마할라노비스 거리 계산.
## 4. 제약사항 및 고려사항
### ① 메모리 점유율 (SRAM vs Flash)
 * 39차원 행렬 하나(39 \times 39)는 float 기준 약 **6KB**입니다.
 * 9개의 레짐을 사용한다면 6 \times 9 = 54KB의 데이터를 상시 보유해야 합니다.
 * **최적화:** 모든 행렬을 SRAM에 올리지 말고, const로 선언하여 **Flash(Instruction Cache)**에서 바로 읽도록 설계하십시오.
### ② 연산 부하와 레이턴시
 * 39차원 행렬 연산은 ESP-DSP를 써도 수 밀리초(ms)가 소요됩니다.
 * 모터 진동의 샘플링 주기가 매우 빠르다면, 모든 프레임을 계산하지 말고 **Moving Window**를 사용하여 5~10프레임당 한 번씩 거리를 계산하는 것이 효율적입니다.
### ③ 전이 구간(Transition) 노이즈
 * 가속에서 등속으로 변하는 찰나에는 공분산 구조가 일시적으로 뒤틀립니다.
 * **해결책:** 마할라노비스 거리 결과값에 **지수 이동 평균(EMA)**을 적용하여 짧은 피크(스파이크)에 의한 오탐을 방지해야 합니다.
## 5. 요약: 추천 아키텍처
| 단계 | 처리 항목 | 최적화 도구 |
|---|---|---|
| **Input** | MFCC 39-dim (S, D, DD) | ESP-DSP (FFT/Windowing) |
| **Classifier** | 현재 속도 및 가/감속 판단 | ESP-DL (Tiny MLP) |
| **Processing** | 레짐별 S^{-1} 선택 및 MD 계산 | ESP-DSP (Matrix/Dot product) |
| **Post-Filtering** | MD 결과값 스무딩 (EMA) | Simple C logic |
이 방식은 구현 난이도가 조금 있지만, 속도가 수시로 변하는 환경에서 **오탐율을 혁신적으로 낮출 수 있는 유일한 방법**입니다. 혹시 39차원 MFCC를 추출하는 코드는 이미 구현이 되어 있으신가요? 어느 단계부터 구체적인 코드가 필요하신지 알려주시면 도움을 드릴 수 있습니다.

✔️ 핵심 수식

MLP의 기본 연산:

은닉층:

출력층:

✔️ ESP32-S3용 C 코드 예제

#define INPUT_SIZE 3
#define HIDDEN_SIZE 8

// 학습된 weight (예시)
float W1[HIDDEN_SIZE][INPUT_SIZE] = {
    {0.2, -0.1, 0.4},
    ...
};

float B1[HIDDEN_SIZE] = {0.1, ...};

float W2[HIDDEN_SIZE] = {0.3, ...};
float B2 = 0.05;

// 활성화 함수 (ReLU)
float relu(float x) {
    return x > 0 ? x : 0;
}

// MLP 추론 함수
float mlp_predict(float input[INPUT_SIZE]) {
    float hidden[HIDDEN_SIZE];

    // hidden layer
    for (int j = 0; j < HIDDEN_SIZE; j++) {
        float sum = B1[j];
        for (int i = 0; i < INPUT_SIZE; i++) {
            sum += input[i] * W1[j][i];
        }
        hidden[j] = relu(sum);
    }

    // output layer
    float output = B2;
    for (int j = 0; j < HIDDEN_SIZE; j++) {
        output += hidden[j] * W2[j];
    }

    return output;
}

앞서 살펴본 이상감지 기법들을 실제 현장에 적용하기 위한 **계산 지표**, **임계값 설정 전략**, 그리고 각 방식의 **장단점**을 일목요연하게 정리해 드립니다.
## 1. 통계적 거리 기반 (Mahalanobis Distance)
MFCC 차원 간의 상관관계를 반영하여 '평소와 다른 조합'이 나타나는지를 감시합니다.
 * **계산 지표:** 
   (공분산 행렬 S 활용)
 * **임계값 설정:** * **카이제곱(\chi^2) 분포:** 데이터가 정규분포를 따른다고 가정할 때, 특정 신뢰수준(95% 또는 99%)에서의 분포값을 임계치로 사용합니다.
   * **교정(Calibration):** 정상 운전 데이터를 1,000점 이상 수집하여 계산된 D_M 값 중 상위 1~5% 지점을 임계치로 설정합니다.
 * **장점:** 변수 간의 복잡한 관계를 한 번에 파악 가능하며, 다차원 데이터(39차원) 처리에 매우 정교합니다.
 * **단점:** 역행렬 계산(S^{-1})이 필요하여 ESP32 같은 저사양 MCU에서는 연산 부담이 큽니다.
## 2. 속도 구간별 정규화 (Binning & Z-Score)
가속/등속/감속에 따라 변하는 기준값을 '표'로 만들어 대응하는 방식입니다.
 * **계산 지표:** 
 * **임계값 설정:** * **3\sigma Rule:** 각 속도 구간(Bin)별로 평균에서 표준편차의 3배를 벗어나면 이상으로 간주합니다.
   * **동적 가중치:** 가속 구간에서는 임계값을 조금 더 넓게(4~5$\sigma$), 등속 구간에서는 타이트하게(2~3$\sigma$) 설정하여 오탐을 방지합니다.
 * **장점:** 구현이 매우 쉽고 직관적입니다. 연산량이 적어 실시간 처리에 최적화되어 있습니다.
 * **단점:** 구간(Bin)의 경계에서 지표가 급격히 변할 수 있으며, 모든 속도 구간에 대한 사전 학습 데이터가 반드시 필요합니다.
## 3. 누적 합계 기반 (CUSUM)
순간적인 튐이 아니라, 시간이 지남에 따라 서서히 변하는 '추세'를 포착합니다.
 * **계산 지표:** 
   (누적 편차 합산)
 * **임계값 설정:** * **H-Limit:** 누적 합계 S_t가 특정 고정값 H를 넘으면 알람을 발생시킵니다. 보통 표준편차의 4~5배 수준의 누적치를 기준으로 잡습니다.
 * **장점:** 베어링 마모처럼 아주 천천히 진행되는 고장을 포착하는 데 가장 강력합니다. 노이즈에 강합니다.
 * **단점:** 갑작스러운 충격이나 급격한 고장에는 반응 속도가 느릴 수 있습니다.
## 4. 요약 및 비교 분석
| 구분 | 통계적 거리 (D_M) | 구간별 정규화 (Binning) | 누적 합계 (CUSUM) |
|---|---|---|---|
| **주요 타겟** | 다차원 상관관계 붕괴 | 속도 변화 대응 | 서서히 진행되는 마모 |
| **연산 복잡도** | 높음 (행렬 연산) | 낮음 (단순 연산) | 낮음 (누적 연산) |
| **임계값 유연성** | 고정 (통계 기반) | 매우 높음 (구간별 설정) | 중간 (추세 기반) |
| **추천 상황** | 고성능 서버/PC 분석 | **ESP32 등 에지 장치** | 장기 트렌드 모니터링 |
## 💡 실무 적용 팁
현재 진행 중인 유도모터 프로젝트가 **가/감속이 잦은 환경**이라면, **'구간별 정규화'**를 기본으로 하되, **'CUSUM'** 로직을 살짝 얹는 방식을 추천드립니다.
 1. 속도(MFCC 0번)에 따라 1차적으로 **Z-Score**를 계산합니다.
 2. 이 Z-Score의 변화가 일시적인지(노이즈), 지속적인지(고장)를 **CUSUM**으로 판단합니다.
이렇게 하면 가속 시 발생하는 일시적인 신호 변화에는 덤덤하게 반응하면서도, 실제 문제가 생겼을 때는 확실하게 잡아낼 수 있습니다.
