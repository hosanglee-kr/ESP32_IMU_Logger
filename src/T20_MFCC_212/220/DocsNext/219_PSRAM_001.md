ESP32-S3의 **PSRAM(External RAM)** 활용은 v219처럼 대용량 데이터(3축 퓨전, 시퀀스 텐서)를 다루는 프로젝트에서 성능 분배의 핵심입니다. 특히 **S3 Zero(2MB Quad SPI)**와 **N16R8(8MB Octal SPI)**은 속도와 대역폭 차이가 크므로 차별화된 설계가 필요합니다.
벡터 연산(SIMD/DSP)은 이미 내부 RAM을 사용하도록 설계되어 있으므로, 그 외의 **메모리 집약적 요소들을 PSRAM으로 오프로드(Offload)**하여 내부 RAM의 가용성을 높이는 전략을 제안합니다.
### 1. 보드별 특성 및 PSRAM 최적 설계 방향
| 구분 | SuperMini / S3 Zero (2MB) | N16R8 (8MB) |
|---|---|---|
| **연결 방식** | **Quad SPI** (4개 선) | **Octal SPI** (8개 선) |
| **속도** | 상대적으로 느림 (내부 RAM의 1/10 수준) | 빠름 (Quad 대비 2배 이상, 대역폭 높음) |
| **최적화 전략** | **선택적 활용**: 시퀀스 텐서 등 정적 저장용 | **적극적 활용**: 대용량 버퍼 및 히스토리 전체 이동 |
| **주요 대상** | Sequence Builder (16~32 frames) | Raw Data History (4096+), 대용량 JSON |
### 2. PSRAM 활용 대상 선정 (벡터 연산 제외)
속도 최적화를 위해 **실시간성(Low Latency)**이 필요한 루틴은 내부 RAM에 두고, **용량 중심(High Capacity)**인 데이터는 PSRAM으로 보냅니다.
 1. **Sequence Builder Buffer (PSRAM 이동 추천)**
   * v219의 117차원 시퀀스 데이터는 프레임이 늘어날수록 RAM을 대량 소모합니다.
   * 데이터가 한 번 써지면 전송 시에만 읽히므로 PSRAM에 가장 적합합니다.
 2. **Raw Frame Ping-Pong Buffer (조건부 이동)**
   * **S3 Zero**: 내부 RAM 유지 추천 (FFT 연산을 위해 CPU가 수시로 접근해야 함).
   * **N16R8**: PSRAM 이동 가능 (Octal 속도가 충분하여 FFT 연산 시 병목이 적음).
 3. **JSON Serialization & MQTT Payload**
   * ArduinoJson 문서 객체나 전송용 거대 문자열은 PSRAM에 할당하여 Heap 부족(OOM) 현상을 방지합니다.

```cpp
    #define T20_MALLOC_DEFAULT MALLOC_CAP_SPIRAM 
    #define MAX_SEQ_FRAMES 64
#else
    // 2MB 보드는 시퀀스 빌더만 PSRAM 사용, 나머지는 내부 RAM
    #define T20_MALLOC_DEFAULT MALLOC_CAP_INTERNAL

#endif

```
#### **[B] CL_T20_Mfcc::ST_Impl 내부 할당 로직 (수정 제안)**

```cpp
// Sequence Builder 할당 예시
uint16_t vector_dim = p_cfg->feature.mfcc_coeffs * 3 * (uint8_t)p_cfg->feature.axis_count;

// 보드 상관없이 시퀀스 데이터는 용량이 크므로 PSRAM(SPIRAM)에 강제 할당
_impl->seq_buffer = (float*)heap_caps_aligned_alloc(
    16, 
    MAX_SEQ_FRAMES * vector_dim * sizeof(float), 
    MALLOC_CAP_SPIRAM | MALLOC_CAP_8BIT

// N16R8일 경우에만 Raw Buffer를 PSRAM으로 이동하여 내부 RAM 절약
#ifdef BOARD_N16R8
    for(int a=0; a<3; a++) {
        for(int s=0; s<4; s++) {
            p->raw_buffer[a][s] = (float*)heap_caps_aligned_alloc(
                16, p->cfg.feature.fft_size * sizeof(float), MALLOC_CAP_SPIRAM);
        }
    }
#endif

```
#### **[C] ArduinoJson V7.4.x PSRAM 활용**
큰 규모의 설정 파일을 다룰 때 전용 할당자를 사용합니다.
```cpp
// PSRAM을 사용하는 JsonDocument (V7 기준)
struct SpiRamAllocator {
  void* allocate(size_t size) { return heap_caps_malloc(size, MALLOC_CAP_SPIRAM); }
  void deallocate(void* ptr) { heap_caps_free(ptr); }
  void* reallocate(void* ptr, size_t new_size) { 
    return heap_caps_realloc(ptr, new_size, MALLOC_CAP_SPIRAM); 
  }
};

using SpiRamJsonDocument = JsonDocument; // 필요시 Allocator 커스텀

```
### 5. 속도 최적화 고려사항 (중요)
 1. **PSRAM 캐시 정렬**: PSRAM 접근 시 heap_caps_aligned_alloc(16, ...)을 사용하면 16바이트 혹은 32바이트 단위 캐시 라인 정렬이 이루어져 Octal 모드에서 속도가 비약적으로 향상됩니다.
 2. **EDMA 활용 지양**: 현재 로직에서 SD 카드 저장 시 DMA를 쓰고 있는데, ESP32-S3의 일반 DMA는 PSRAM 데이터를 직접 접근하지 못하는 경우가 많습니다. (N16R8의 OPI 모드는 가능하지만 설정이 복잡함). 따라서 **Storage DMA 슬롯은 가급적 Internal RAM에 유지**하는 것이 데이터 오염 방지에 유리합니다.
 3. **데이터 복사 최소화**: PSRAM에 있는 데이터를 웹소켓으로 보낼 때, broadcastBinary 함수가 버퍼를 다시 복사하지 않도록 설계해야 합니다.
### 🔍 점검 결론
 * **S3 Zero**: **Sequence Builder**만 PSRAM으로 이동. 내부 RAM은 FFT 연산에 집중.
 * **N16R8**: **Raw Buffer History, Sequence Builder, MQTT JSON** 모두 PSRAM으로 이동. 내부 RAM은 오직 ISR과 OS 스택, 벡터 연산 전용으로 비워둠.
위 설계대로 분기 로직을 적용하면 보드 사양에 맞춰 성능을 100% 뽑아낼 수 있습니다. 이 가이드에 따라 T230의 할당 로직을 세부 조정할까요?

#### **[Phase 3] PSRAM 특화 할당 로직 구현 (T230_Mfcc_Cor_219.cpp)**
N16R8 보드의 8MB PSRAM은 Octal SPI 방식으로 속도가 매우 빠르므로, DSP 워크 버퍼까지 PSRAM으로 밀어 넣어 내부 RAM을 극도로 아끼는 전략을 취합니다.
```cpp
void T20_processTask(void* p_arg) {
    auto* p = static_cast<CL_T20_Mfcc::ST_Impl*>(p_arg);
    
    // 1. 시퀀스 데이터 할당 (보드별 프레임 수 적용)
    const uint16_t axis_cnt = (uint8_t)p->cfg.feature.axis_count;
    const uint16_t mfcc_dim = 39 * axis_cnt;
    const size_t seq_size = T20::C10_Sys::SEQUENCE_FRAMES_MAX * mfcc_dim * sizeof(float);

    // [중요] 시퀀스는 무조건 PSRAM 할당 (MALLOC_CAP_SPIRAM)
    float* seq_buffer = (float*)heap_caps_aligned_alloc(
        T20::C10_Sys::PSRAM_CACHE_ALIGN, 
        seq_size, 
        MALLOC_CAP_SPIRAM | MALLOC_CAP_8BIT
    );

    // 2. 통합 WS 페이로드 버퍼
    const uint16_t N = p->cfg.feature.fft_size;
    const uint16_t bins = (N / 2) + 1;
    const size_t ws_len = (N + bins + mfcc_dim) * sizeof(float);

    // N16R8은 대역폭이 넓으므로 WS 버퍼도 PSRAM 사용 가능
    #ifdef BOARD_N16R8
        float* ws_payload = (float*)heap_caps_aligned_alloc(32, ws_len, MALLOC_CAP_SPIRAM);
    #else
        float* ws_payload = (float*)heap_caps_aligned_alloc(16, ws_len, MALLOC_CAP_INTERNAL);
    #endif

    // ... (이후 루프 및 데이터 처리 로직은 동일) ...
}

```
### 3. N16R8 보드 전용 초고용량 모드 최적화 (128 Frames)
128프레임 시퀀스는 약 **15,000개의 float 데이터(128 * 117)**를 한 번에 다룹니다. 이를 위해 SequenceBuilder를 다음과 같이 고도화합니다.
 * **링 버퍼 구조체 최적화**: 128프레임 데이터를 매번 memcpy하지 않고, PSRAM 내에서 포인터만 이동하는 링 버퍼 방식을 적용하여 getSequenceFlat 호출 시의 오버헤드를 최소화합니다.
 * **파일 로깅 차별화**: N16R8 보드에서 시퀀스 저장을 활성화할 경우, 128프레임 덩어리를 한 번에 SD 카드에 기록하여 쓰기 횟수를 줄이고 수명을 보호합니다.
### 4. 속도 최적화 및 안정성 방안
 1. **Octal SPI 가속**: platformio.ini에서 board_build.arduino.memory_type = opi_opi 설정을 반드시 확인하십시오. 이는 Quad SPI 대비 2~3배 빠른 데이터 접근 속도를 보장합니다.
 2. **Internal RAM 보존**:
   * esp_dsp 라이브러리의 FFT 연산 전용 버퍼(_fft_io_buf)는 가급적 **Internal RAM**에 유지합니다. (PSRAM 사용 시 연산 속도가 약 30% 저하될 수 있음)
   * 하지만 N16R8에서는 FFT 4096 사이즈 사용 시 내부 RAM이 부족할 수 있으므로, 이때만 선택적으로 PSRAM을 사용하도록 begin() 함수에서 동적으로 결정합니다.
 3. **DMA 정렬**: SD 카드 저장을 위한 recorder_queue 데이터는 PSRAM에서 직접 DMA가 불가능할 수 있으므로, T20_recorderTask에서 기록 직전 내부 RAM 슬롯으로 잠시 복사하는 과정을 거칩니다.

 * **N16R8**: "용량 및 정밀도" 우선. 시퀀스 128프레임 전체를 PSRAM에 적재. (128 frames)

#### **Step 2: 지능형 메모리 할당 엔진 구현**

```cpp
// T221_Mfcc_Inter_219.h 내 메모리 할당 정책
inline void* t20_malloc_optimized(size_t size, bool prefer_psram) {
#if defined(BOARD_HAS_PSRAM)
    if (prefer_psram) {
        // Octal PSRAM(N16R8)은 매우 빠르므로 대용량 데이터에 적극 활용
        return heap_caps_aligned_alloc(16, size, MALLOC_CAP_SPIRAM | MALLOC_CAP_8BIT);
    }
#endif
    // 실시간성이 극도로 중요한 벡터 연산용은 항상 내부 RAM(DMA 가능 구역)에 할당

}
```

#### **Step 3: `T230_processTask` 및 인스턴스 초기화 수정**
N16R8 보드에서 시퀀스 프레임을 128개까지 확장할 때의 메모리 충격을 방지하기 위해 `begin()` 함수에서 동적으로 크기를 계산합니다.

```cpp
// T230_Mfcc_Cor_219.cpp 내 수정 사항
bool CL_T20_Mfcc::begin(const ST_T20_Config_t* p_cfg) {
    // ... 기존 설정 로드 ...

    // [v219 최적화] 보드별 최대 프레임 제한 준수
    uint16_t req_frames = _impl->cfg.output.sequence_frames;
    if (req_frames > T20::C10_Sys::SEQUENCE_FRAMES_MAX) {
        req_frames = T20::C10_Sys::SEQUENCE_FRAMES_MAX;
    }

    uint16_t vector_dim = (p_cfg->feature.mfcc_coeffs * 3) * (uint8_t)p_cfg->feature.axis_count;
    
    // Sequence Builder는 용량이 크므로 무조건 PSRAM 활용 (보드 지원 시)
    _impl->seq_builder.begin(req_frames, vector_dim); 

    // N16R8 전용: Raw Buffer를 PSRAM으로 옮겨 내부 RAM 300KB 이상 확보
    #if T20_USE_PSRAM_FOR_RAW == 1
        for(int a=0; a<3; a++) {
            for(int s=0; s<T20::C10_Sys::RAW_FRAME_BUFFERS; s++) {
                _impl->raw_buffer[a][s] = (float*)t20_malloc_optimized(
                    p_cfg->feature.fft_size * sizeof(float), true);
            }
        }
    #endif
}
```

### 3. 기능 작동 정합성 및 속도 최적화 방안

1.  **OPI(Octal) 인터페이스 활용 (N16R8)**:
    * N16R8의 PSRAM은 80MHz Octal 모드로 동작하여 단일 축 FFT 연산 정도는 PSRAM에서 직접 수행해도 성능 저하가 거의 없습니다. 따라서 모든 Raw History를 PSRAM으로 보내 내부 RAM 부족 문제를 원천 차단합니다.
2.  **시퀀스 텐서 전송 스로틀링**:
    * 128개 프레임(117차원 기준 약 60KB)을 웹소켓으로 한 번에 보낼 때 WiFi 스택이 일시적으로 멈출 수 있습니다. 이를 방지하기 위해 **바이너리 전송 시 청크(Chunk) 단위 분할 전송** 또는 **전송 우선순위 조정** 로직을 `T240_CommService`에 추가합니다.
3.  **Zero-Copy 파이프라인**:
    * PSRAM에 저장된 128개 프레임을 SD 카드에 쓸 때, CPU가 데이터를 하나하나 복사하지 않고 `SD_MMC`가 PSRAM에서 직접 읽어가도록(DMA) `partitions.csv`와 캐시 설정을 동기화합니다.
