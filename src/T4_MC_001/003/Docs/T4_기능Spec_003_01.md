

제시해주신 **[최종 완결판 기능 명세서 (v2.0)]**와 현재까지 우리가 작성한 실제 C++ 시스템 코드(T410 ~ T480)를 바닥부터 다시 1:1로 교차 검증했습니다.
정신을 바짝 차리고 명세서의 글귀 하나하나를 코드의 변수/함수와 대조해 본 결과, 훌륭하게 구현된 뼈대 이면에 **실제 공장 환경의 MLOps 및 데이터 파이프라인 연동 시 치명적인 구멍이 될 수 있는 4건의 명백한 기능 누락과 1건의 설계 우회(축소)**를 적발했습니다.
단순한 오타 수준이 아닙니다. 아래 사항들을 점검하시고 보완 결정을 내리셔야 합니다.
### 🚨 스펙 대비 기능 축소 및 누락 적발 리포트
#### 1. [기능 누락] 120Hz 고조파(Harmonics) Notch 필터 증발
 * **명세서 (Phase 2):** "Notch Filter: 60Hz/120Hz 및 주요 전원 하모닉스 제거 (IIR Biquad)"
 * **코드 현실 (T430_DspEng):** 현재 코드는 SmeaConfig::Dsp::NOTCH_FREQ_HZ = 60.0f 단 한 개의 대역에 대해서만 Biquad 필터를 생성하고 적용하고 있습니다.
 * **문제점:** 공장 설비나 모터 주변에서는 60Hz 교류 전원의 1차 고조파인 120Hz 노이즈가 매우 강하게 유입됩니다. 60Hz만 깎아내고 120Hz를 방치하면 피처 추출 시 에일리어싱이나 저주파 왜곡이 발생합니다.
 * **보완책:** T430 엔진에 120Hz용 필터 계수 배열과 딜레이 라인을 추가로 선언하고, 60Hz 처리 직후에 직렬(Cascade)로 연결하여 120Hz 대역까지 깎아내야 합니다.
#### 2. [기능 누락] 주파수 영역 'Band RMS' 피처 추출 누락
 * **명세서 (Phase 3):** "주파수 영역: Spectral Centroid, Band RMS, N-top Peaks(주파수/진폭)"
 * **코드 현실 (T420_Types & T440_FeatExtra):** 전체 주파수 영역의 에너지를 보는 Spectral Centroid와 N-top Peaks는 완벽하게 복원되었으나, 정작 **Band RMS (특정 주파수 대역별 에너지 크기)**를 저장할 구조체 멤버나 계산 로직이 아예 존재하지 않습니다.
 * **문제점:** 기계 설비 모니터링 시 베어링 결함이나 특정 마찰음은 전체 RMS가 아닌 특정 주파수 대역(예: 2kHz~5kHz)의 RMS 변화로만 감지되는 경우가 많습니다. Band RMS 누락은 치명적인 진단력 하락을 야기합니다.
#### 3. [기능 축소] STA/LTA 기반 '동적 구간 슬라이싱' 미구현 (고정 길이 녹음)
 * **명세서 (Phase 1):** "Active Signal: 에너지 기반 자동 검출(STA/LTA)을 통한 실제 작동 구간 슬라이싱"
 * **코드 현실 (T450_FsmMgr):** 현재 시스템은 불량이 감지되면 타이머(v_recordStartMs)를 돌려 무조건 SmeaConfig::VALID_END_SEC (0.5초) 만큼만 고정된 길이로 저장하고 세션을 닫습니다. STA/LTA는 불량을 '판정'하는 용도로만 쓰일 뿐, 소리가 지속되는 '구간'을 동적으로 잘라내지 않습니다.
 * **문제점:** 실제 현장에서는 타격음이 0.1초 만에 끝날 수도, 1초 동안 지속될 수도 있습니다. 고정 길이 저장은 불필요한 공백 데이터를 남기거나 긴 사고 데이터를 잘라먹는 기능 축소입니다.
#### 4. [설계 우회] 특징량 파일 저장 포맷 (JSON -> BIN)
 * **명세서 (Phase 4):** "저장 방식: SD카드에 최종 정제된 Raw PCM 및 모든 피처를 담은 JSON 기록"
 * **코드 현실 (T460_Storage):** 우리는 SD 카드의 I/O 병목과 병목으로 인한 캡처 태스크의 프레임 드랍을 막기 위해, 구조체(FeatureSlot)를 C++ 메모리 덤프 방식인 이진 파일(.bin)로 제자리 복사(Zero-copy) 해버렸습니다.
 * **문제점:** 임베디드 단의 성능은 극한으로 최적화되었지만, 명세서가 요구했던 "데이터 엔지니어링 및 분석의 용이성(JSON)"을 희생한 설계 우회(Trade-off)입니다. 외부 클라우드나 분석 플랫폼으로 이 데이터를 가져갔을 때, .bin 파일을 파싱하려면 C++ 구조체와 정확히 일치하는 별도의 Python 파서(Parser)를 구축해야만 하는 부담이 생깁니다.
#### 5. [파라미터 불일치] 노이즈 프로파일링 학습 구간 (150ms vs 1초)
 * **명세서 (Phase 1):** "Noise Profile: 시작 신호 후 0~150ms (ANC용 배경 소음 프로파일링)"
 * **코드 현실 (T410_Config):** SmeaConfig::Dsp::NOISE_LEARN_FRAMES = 100으로 설정되어 있습니다. 홉 크기가 10ms이므로 100 프레임은 곧 **1000ms(1초)**에 해당합니다.
 * **문제점:** 150ms만 배경 소음을 빠르게 학습하고 곧바로 감시 모드로 들어가야 하는데, 현재는 무려 1초 동안 학습을 진행하므로 초기 1초 동안의 기계음이 백그라운드 노이즈로 잘못 학습되어 ANC가 핵심 신호마저 다 깎아버릴 위험이 있습니다. 이 값은 15로 수정되어야 명세서와 일치합니다.
### 💡 보완 권고 사항
시스템의 뼈대는 완벽하므로 대대적인 구조 변경은 필요하지 않습니다.
다만, 120Hz 노치 필터와 Band RMS는 **[DSP 엔진 및 피처 추출기]**에 즉시 코드를 추가해야 합니다. 또한 .bin 파일 포맷 건은 임베디드 성능을 위해 현재 방식을 유지하되, 추후 서버/PC 단에서 이 .bin 데이터를 읽어 데이터 파이프라인에 태울 수 있는 가벼운 디코더(Decoder) 스크립트를 마련해 두는 방향으로 전략을 수정하는 것이 현명합니다.



제대로 짚으셨습니다. 컴파일 오류와 메모리 누수 같은 '치명적인 시스템 에러'를 잡는 데 몰두한 나머지, **명세서(Spec)에 적힌 수식과 도메인 로직이 코드에 정확히 1:1로 매핑되었는지**에 대한 딥다이브 점검이 부족했습니다.
바닥부터 수식과 상태 머신을 긁어내듯 재점검한 결과, **42kHz 고음질 스펙을 무용지물로 만들고 딥러닝(ML) 판정력을 떨어뜨리는 4가지의 '조용한 기능 축소(Silent Reduction)'**를 추가로 적발해 냈습니다.
이것들은 에러를 뿜지 않아서 더 위험한 결함들입니다.
### 🚨 스펙 대비 조용한 기능 축소/누락 (심층 적발 리포트)
#### 1. [스펙 기만] FIR LPF (저역통과필터) 컷오프 주파수 오설정
 * **명세서 (Phase 2):** "FIR Band-pass: 100Hz ~ 20kHz 대역 정제"
 * **코드 현실 (T410_Config):** SmeaConfig::Dsp::FIR_LPF_CUTOFF = 3800.0f; 로 설정되어 있습니다.
 * **치명적 문제:** 샘플링 레이트를 42kHz로 올린 이유는 Nyquist 정리에 따라 최대 21kHz 대역의 초고주파 마찰음과 고차 하모닉스를 분석하기 위함입니다. 그런데 정작 전처리 DSP 엔진에서 **3.8kHz 이상의 소리를 모조리 잘라내어 버리고(Cutoff) 있습니다.** 이는 42kHz 스펙을 완전히 기만하는 설정 오류입니다.
#### 2. [수학적 축소] MFCC Delta (동적 특징) 공식 간소화
 * **명세서 (Phase 3):** Delta 계산 시 N=2 (현재 프레임 전후 2프레임)를 참조하는 정밀한 가중합 수식을 요구했습니다.
 * **코드 현실 (T440_FeatExtra):** (v_mfccHistory[4][i] - v_mfccHistory[2][i]) / 2.0f; 즉, N=1 (전후 1프레임)의 단순 차분으로 공식을 임의로 간소화(축소)하여 구현했습니다.
 * **치명적 문제:** N=1 단순 차분은 공장 소음 환경에서 튀는 값에 매우 취약합니다. 명세서대로 N=2 수식을 적용해야 시간적 흐름의 노이즈가 평활화(Smoothing)되어 ML 모델의 판정 정확도가 올라갑니다.
#### 3. [시나리오 증발] 3회 반복 수집 (워밍업 및 교차 검증) 시퀀스 누락
 * **명세서 (Phase 1):** "DC 4V 이상 감지 시 즉시 수집 시작 (3회 반복). 1회차: 워밍업 및 배경 노이즈 캡처, 2~3회차: 실제 분석 대상 (교차 검증)."
 * **코드 현실 (T450_FsmMgr):** 현재 상태 머신은 트리거가 들어오면 단순히 MONITORING으로 한 번 진입하고, 불량이 나면 RECORDING으로 빠지는 **'단발성(1-shot)' 구조**입니다. 3회 반복에 대한 루프 제어나 1회차를 노이즈 프로파일링으로 활용하는 시나리오가 코드에서 완전히 증발했습니다.
 * **결과:** 구조체에 정의해 둔 uint8_t trial_no; 변수는 값이 한 번도 할당되지 않은 채 고아(Orphan) 상태로 방치되어 0만 기록됩니다.
#### 4. [판정 누락] Level 1 (Test NG - 신호 누락) 검문소 누락
 * **명세서 (Phase 4):** "Level 1 (Test NG): 트리거 미발생 또는 신호 누락"
 * **코드 현실 (T450_FsmMgr):** runHybridDecision() 함수에는 RULE_NG와 PASS만 존재합니다. 마이크 하드웨어 에러나 I2S 신호 누락 등으로 인해 소리가 아예 들어오지 않는 치명적 상황(RMS/Energy가 비정상적으로 0에 수렴하는 경우)에 대한 하한선 검사가 누락되었습니다.

#### ⚠️ 아직 코드에 반영해야 할 스펙 축소/누락 사항
*이 항목들은 "컴파일이나 시스템 다운"을 유발하진 않지만, **기계 결함 판정력을 떨어뜨리고 명세서를 위반하는 도메인 로직들**입니다. 이 부분들의 패치를 진행해야 합니다.*
 1. **[스펙 기만] 42kHz 대역폭 절단 (FIR LPF 컷오프):**
   * **문제:** 42kHz로 샘플링해 놓고 DSP 엔진에서 3.8kHz 이상을 다 잘라버리고 있습니다 (FIR_LPF_CUTOFF = 3800.0f).
   * **조치 필요:** 컷오프를 20,000.0f (20kHz)로 개방해야 합니다.
 2. **[기능 누락] 120Hz 주요 전원 고조파(Harmonics) 노치 필터 누락:**
   * **문제:** 60Hz 전원 노이즈만 잡고, 더 치명적인 1차 고조파인 120Hz 필터링이 누락되었습니다.
   * **조치 필요:** T430_DspEng에 120Hz Biquad Notch 필터 직렬(Cascade) 추가.
 3. **[수학 축소] MFCC Delta N=2 정밀 공식 간소화:**
   * **문제:** 명세서는 전후 2프레임(N=2)의 가중합을 요구했으나, 코드는 단순 전후 1프레임 차분으로 축소 구현되었습니다.
   * **조치 필요:** T440의 Delta 및 Delta-Delta 계산식에 N=2 수식 적용.
 4. **[기능 누락] 주파수 영역 'Band RMS' 피처 누락:**
   * **문제:** 베어링 결함 등 특정 주파수 대역의 에너지를 보는 Band RMS 지표가 구조체와 추출기에서 증발했습니다.
   * **조치 필요:** FeatureSlot에 배열 추가 및 T440 추출 로직 구현.
 5. **[기능 축소] 3회 반복 수집(워밍업 -> 교차 검증) 시나리오 증발:**
   * **문제:** 트리거 발생 시 1회성으로 수집하고 끝나는 단발성 구조입니다. (1회차 배경 학습, 2/3회차 본 검사 누락).
   * **조치 필요:** FSM 상태 머신에 시퀀스 루프 카운터(trial_no) 도입 여부 결정.
 6. **[판정 누락] Level 1 (Test NG - 마이크/신호 단선) 방어 누락:**

void T440_FeatureExtractor::computeBandRMS(SmeaType::FeatureSlot& p_slot) {
    float v_binRes = (float)SmeaConfig::SAMPLING_RATE / SmeaConfig::FFT_SIZE;
    uint16_t v_maxBin = (SmeaConfig::FFT_SIZE / 2) + 1;

    for (int b = 0; b < SmeaConfig::Feature::BAND_RMS_COUNT; b++) {
        float v_startHz = SmeaConfig::Feature::BAND_RANGES[b][0];
        float v_endHz = SmeaConfig::Feature::BAND_RANGES[b][1];
        
        int v_startBin = (int)(v_startHz / v_binRes);
        int v_endBin = (int)(v_endHz / v_binRes);
        
        float v_sumSq = 0.0f;
        int v_count = 0;
        
        for (int i = v_startBin; i <= v_endBin && i < v_maxBin; i++) {
            v_sumSq += v_powerSpectrum[i]; 
            v_count++;
        }
        p_slot.band_rms[b] = v_count > 0 ? sqrtf(v_sumSq / v_count) : 0.0f;
    }
}
