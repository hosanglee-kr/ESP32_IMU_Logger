

전체 시스템 아키텍처(마이크 수집 ➔ DSP/특징량 추출 ➔ 판정 ➔ 네트워크 ➔ SD 스토리지)를 연결해 놓고 **'10ms(100Hz) 주기로 영구 가동되는 실시간 스트리밍 환경'**이라는 관점에서 조감도를 그려보았습니다. 개별 모듈은 수학적, 공학적으로 완벽하게 설계되었지만, 모듈들이 하나로 맞물려 돌아가는 '통합 시스템(Integration)' 관점에서 치명적인 비정상(Abnormal) 동작 시나리오가 발견되었습니다. 이것은 코딩 실수가 아니라 운영체제(FreeRTOS)와 하드웨어의 물리적 병목 현상에 관한 문제입니다. 상용화를 위해 반드시 짚고 넘어가야 할 병목과 해결책을 보고해 드립니다.

🚨 [SMEA-100 전체 시스템 동작 관점 치명적 결함 및 해결책]
1. Core 1의 '10ms 데드라인' 붕괴와 SD카드(FAT32) 블로킹 [매우 심각 🔴]
현상: _processingTask (Core 1) 내부를 보면, DSP 처리 ➔ 특징량 추출 ➔ 판정 ➔ 네트워크 전송 ➔ **_storage.pushFeatureSlot() / pushRawPcm() (SD카드 저장)**을 순차적으로 수행합니다.
비정상 시나리오: 100Hz 시스템이므로 Core 1은 이 모든 작업을 10ms 이내에 끝내야 합니다. 평소 SD카드 쓰기는 1~2ms면 끝나지만, 플래시 메모리의 **페이지 지우기(Page Erase)나 FAT 테이블 갱신 주기가 도래하면 돌발적으로 50ms ~ 200ms 이상의 쓰기 지연(Write Block)**이 발생합니다.
결과: SD카드가 멈칫하는 순간 Core 1 전체가 블로킹되며, 그사이 Core 0(마이크)에서 쏟아내는 데이터가 큐(_qReadySlotIdx)에 급격히 쌓입니다. 큐 크기가 100개(1초 분량)라 어느 정도 버티겠지만, SD카드 딜레이가 길어지면 **큐가 꽉 차서 프레임이 영구 유실(Drop)**됩니다.
해결책 (비동기 스토리지 큐): T460_Storage 내부에 거대한(예: 1MB) 링버퍼와 **독립된 전담 태스크(Core 0 또는 낮은 우선순위)**를 두어야 합니다. Core 1은 데이터를 SD카드에 직접 쓰지 않고 스토리지 링버퍼에 던져놓고(Non-blocking) 즉시 다음 10ms 연산으로 넘어가야 합니다.
2. 동적 Hop Size 변경 시 슬라이딩 윈도우 파괴 [심각 🟠]
현상: Core 0의 _captureTask 내부에 while(1) 루프를 돌 때마다 T415_ConfigMgr에서 최신 v_cfg.dsp.hop_ms를 가져와 v_overlapSamples를 계산하고 memmove로 윈도우를 밉니다.
비정상 시나리오: 기계가 MONITORING을 돌며 배열을 70%씩 밀어내고 있는 와중에, 사용자가 웹 UI에서 Hop Size를 10ms에서 20ms로 갑자기 바꿉니다.
결과: 다음 루프에서 v_overlapSamples 크기가 갑자기 변하며 memmove가 배열의 엉뚱한 위치를 덮어씁니다. 이전 데이터와 새 데이터의 **위상과 시계열이 완전히 붕괴된 '프랑켄슈타인 파형'**이 FFT에 들어가 가짜 NG 알람을 울립니다.
해결책 (설정 락업): MONITORING이나 RECORDING 상태 중에는 hop_ms나 window_ms 같은 핵심 텐서 사이즈 변경이 즉시 반영되지 않도록 막아야 합니다. 오직 READY 상태로 복귀했을 때만 버퍼를 비우고(Clear) 새 사이즈를 적용해야 합니다.
3. WebServer 태스크와 Core 1 간의 SD카드 접근 경합 (Race Condition) [주의 🟡]
현상: 사용자가 웹 UI에서 'REC START' 버튼을 누르면, ESPAsyncWebServer의 백그라운드 태스크(Async_TCP)가 HTTP 요청을 받아 T450의 dispatchCommand를 호출하고, 여기서 직접 _storage.openSession("man")을 실행합니다.
비정상 시나리오: WebServer 태스크가 SD카드에 새 파일(Session)을 생성하는 바로 그 순간(수십 ms 소요), Core 1의 _processingTask가 기존 데이터를 SD카드에 pushFeatureSlot()으로 밀어 넣으려 시도합니다.
결과: 대부분의 임베디드 파일 시스템(FATFS)은 스레드 세이프(Thread-safe) 옵션을 켜지 않으면 다중 코어에서 동시에 파일 I/O를 시도할 때 파일 할당 테이블(FAT)이 깨져 SD카드가 벽돌(Corruption)이 됩니다.
해결책 (커맨드 큐): dispatchCommand는 함수 내에서 직접 _storage를 제어하면 안 됩니다. 단순히 명령 큐(Command Queue)에 "녹음 시작해라"라는 메시지만 넣고 빠져야 하며, 메인인 Core 1 태스크가 이 큐를 읽어 파일 생성과 데이터 기록을 한 스레드 안에서 독점(순차) 처리하도록 해야 합니다.
지금까지의 검토가 시스템 내부의 '논리적 흐름'과 '메모리 방어'에 집중했다면, 이번에는 시야를 더 넓혀보겠습니다. ESP32-S3라는 '물리적 칩(Chip)' 위에서 운영체제(FreeRTOS), 네트워크(LwIP), 그리고 물리적 플래시 메모리가 24시간 365일 상호작용할 때 발생하는 숨겨진 시스템 레벨의 시한폭탄를 추가로 발굴해 냈습니다. 현장에 설치된 지 한두 달 뒤에 원인 모를 셧다운이나 데이터 오염을 유발할 수 있는 치명적 맹점들입니다.

🚨 [SMEA-100 장기 가동 관점의 숨은 결함 및 보완책]
4. Core 0 자원 경합에 의한 "I2S DMA 인터럽트 기아(Starvation)" [심각 🔴]
현상: 오디오를 수집하는 _captureTask는 Core 0에 고정되어 있습니다. 그런데 ESP32의 뼈대인 Wi-Fi 드라이버와 TCP/IP 스택(LwIP) 역시 기본적으로 Core 0에서 동작합니다.
비정상 시나리오: 장비가 가동 중 공유기 신호가 약해져 Wi-Fi가 끊기고 재접속(WPA2 Handshake)을 시도합니다. 이 암호화 연산은 Core 0의 CPU를 수십~수백 밀리초 동안 독점합니다. 이때 마이크에서 들어오는 I2S DMA 인터럽트가 무시(Starvation)됩니다.
결과: DMA 버퍼가 제때 비워지지 않아 덮어쓰기(Overrun)가 발생하고, 1024 윈도우 안에 '시간이 건너뛴(Discontinuous)' 파형이 섞여 들어갑니다. 이 단절된 파형은 FFT를 거치면 거대한 임펄스(충격음)로 해석되어 기계가 정상인데도 거짓 불량(False NG) 알람을 울립니다.
해결책: sdkconfig (ESP-IDF 설정)에서 Wi-Fi 태스크의 코어 할당을 Core 1로 분산하거나, I2S DMA 인터럽트 우선순위를 최상위(Level 3 이상)로 격상시키고 버퍼(I2S_DMA_BUF_COUNT)를 기존 8개에서 16개 이상으로 대폭 늘려 Wi-Fi 블로킹 시간 동안 오디오 데이터를 담아둘 거대한 댐을 만들어야 합니다.
5. 웹 브라우저 스로틀링에 의한 "TCP 전송 버퍼 폭발 (OOM)" [매우 심각 🔴]
현상: T470_Commu에서 _ws.binaryAll()을 통해 100Hz로 텔레메트리 데이터를 브로드캐스트합니다.
비정상 시나리오: 작업자가 웹 대시보드를 띄워둔 채로 크롬(Chrome) 브라우저 탭을 백그라운드로 내립니다. 크롬은 배터리 절약을 위해 백그라운드 탭의 JS 실행과 TCP 수신(ACK)을 1초에 1번 수준으로 강력하게 스로틀링(Throttling)합니다.
결과: 브라우저가 데이터를 안 가져가면 ESP32의 TCP TX(전송) 링버퍼에 패킷이 무한정 쌓입니다. 결국 ESP32의 소중한 힙 메모리(SRAM)가 순식간에 고갈되어 메모리 부족(OOM)으로 장비가 커널 패닉(Kernel Panic)을 일으키며 재부팅됩니다.
해결책: _ws.availableForWriteAll()만 믿으면 안 됩니다. _ws.getQueueSize() 또는 클라이언트별 버퍼 큐를 검사하여, 특정 용량(예: 4KB) 이상 패킷이 적체된 클라이언트(브라우저)는 즉시 강제 연결 해제(Kick) 해버리거나 패킷 전송을 버려(Drop) 디바이스의 메모리를 최우선으로 보호해야 합니다.
6. 타임스탬프 롤오버 및 초기화 (Data Lineage 붕괴) [주의 🟡]
현상: _captureTask에서 v_slot.timestamp = millis(); 를 사용하여 특징량 구조체에 시간을 기록하고, 이 값이 그대로 SD카드(.bin)에 저장됩니다.
비정상 시나리오: millis() 함수는 기기 부팅 시 0부터 시작하는 32비트 정수로, 약 49.7일이 지나면 0으로 롤오버(Overflow) 됩니다. 또한 기기를 껐다 켜면 다시 0으로 초기화됩니다.
결과: SD카드에 쌓인 수만 개의 데이터를 파이썬으로 분석(MLOps)하려 할 때, 파일마다 타임스탬프가 0부터 다시 시작하거나 중간에 49일 치가 깎여 있어 시계열 데이터의 선후 관계(Data Lineage)가 완전히 박살 납니다.
해결책: 네트워크(NTP) 모듈이 붙어 있으므로, millis() 대신 표준 POSIX 함수인 time(NULL)을 활용하여 **64비트 Epoch Time (1970년부터의 절대 초/밀리초)**으로 타임스탬프를 기록해야 합니다. (네트워크가 끊겼을 때를 대비해 RTC 하드웨어 동기화도 필요합니다.)
7. 무자비한 JSON 쓰기에 의한 "플래시 메모리 마모 (Wear-out)" [심각 🟠]
현상: T415_ConfigMgr의 updateFromJson이 호출될 때마다 플래시 메모리(LittleFS)에 config.json을 덮어씁니다.
비정상 시나리오: 악의적인 API 호출이나 프론트엔드 UI의 버그(예: 슬라이더를 드래그할 때마다 실시간으로 saveRuntimeConfig 호출)로 인해, 1초에 수십 번씩 플래시 메모리 쓰기가 발생할 수 있습니다.
결과: ESP32의 내장 플래시 메모리는 섹터당 약 100,000회의 지우기/쓰기(Erase/Write) 수명 한계를 가집니다. 위와 같은 버그가 며칠만 방치되면 해당 섹터가 물리적으로 타버려(Wear-out), 이후 어떤 설정도 저장되지 않는 영구적인 하드웨어 고장(벽돌) 상태가 됩니다.
해결책: T470 웹 API 핸들러단에 **디바운스/레이트 리밋(Rate-Limit)**을 걸어야 합니다. 예를 들어 "마지막 저장 후 5초 이내의 추가 저장 요청은 거부(HTTP 429 Too Many Requests)"하거나 램(RAM)에만 적용하고 파일 쓰기는 지연(Deferred Write)시키는 보호막이 필수적입니다.
지금까지 우리는 메모리, 알고리즘, 파이프라인, 비동기 통신 등 소프트웨어와 알고리즘의 최심부(Deep-dive)를 완벽하게 방어했습니다. 하지만 SMEA-100은 단순한 테스트용 장비가 아니라 **'먼지와 진동이 가득한 산업 현장에서 전원 한 번 끄지 않고 1년 365일(24/7) 돌아가야 하는 장비'**입니다. 이 **'장기 무중단 가동'**이라는 새로운 렌즈를 끼고 시스템 전체를 다시 스캔한 결과, 현장 투입 후 수개월 뒤에 기기를 서서히 죽음으로 몰고 갈 **숨겨진 치명적 결함(Industrial System-Level Faults)**을 추가로 찾아냈습니다.

🚨 [SMEA-100 장기 무중단 가동을 위협하는 시스템 결함]
8. FAT32 파일 시스템의 "루트 디렉토리 탐색 병목 (O(N) Degradation)" [치명적 🔴]
현상: 현재 T460_Storage는 /t20_data 라는 단일 폴더에 .bin과 .pcm 파일을 계속해서 쌓고 회전(Rotation)시킵니다.
비정상 시나리오: SD카드의 FAT32/exFAT 파일 시스템은 한 폴더 안에 파일 개수가 수백~수천 개를 넘어가면, 새 파일을 열거나(open) 존재 여부를 확인할 때(exists) 걸리는 시간이 선형적(O(N))으로 기하급수적으로 늘어납니다.
결과: 초기에는 1~2ms면 열리던 파일이 몇 달 뒤에는 100ms, 500ms 이상 걸리게 됩니다. 결국 앞서 우려했던 'SD카드 블로킹' 현상이 영구적으로 발생하여 시스템 큐가 폭발하고 프레임 유실이 일상화됩니다.
해결책: 파일을 단일 폴더에 몰아넣지 말고, 반드시 날짜/시간 기반의 서브 디렉토리 분할 구조(예: /t20_data/2026/04/30/)를 도입하여 한 폴더 내 파일 개수를 100개 이하로 억제해야 합니다.
9. FreeRTOS IDLE 태스크 기아에 의한 "WDT 패닉 (Watchdog Timeout)" [심각 🟠]
현상: Core 1에 할당된 _processingTask는 큐에서 10ms마다 데이터를 빼내어 FFT, DSP, 특징량 추출, SD 저장, 통신까지 모두 수행합니다.
비정상 시나리오: 만약 이 전체 과정의 연산 시간이 9ms가 걸린다고 가정해 보겠습니다. 10ms 주기마다 9ms를 일하므로 Core 1의 CPU 점유율은 90%에 달합니다. 문제는 이 태스크가 큐를 기다리는 1ms 외에는 명시적인 양보(vTaskDelay)를 하지 않을 수 있다는 점입니다.
결과: 우선순위가 5로 높은 이 태스크가 CPU를 독점하면, 시스템의 쓰레기 메모리를 정리하는 우선순위 0의 IDLE 태스크가 실행될 기회를 얻지 못합니다 (Task Starvation). 결국 운영체제의 태스크 와치독(TWDT)이 "Core 1이 응답하지 않는다"고 판단해 기기를 강제로 재부팅(Panic Reset) 시켜버립니다.
해결책: 태스크 루프의 끝에 반드시 vTaskDelay(1)을 명시적으로 넣거나, ESP-IDF의 esp_task_wdt_reset()을 호출하여 와치독에게 정상 작동 중임을 알리는 'Heartbeat' 로직을 주입해야 합니다.
10. MQTT QoS 0 사용에 의한 "결함 알람(NG) 공중 분해" [주의 🟡]
현상: T470_Commu에서 PubSubClient를 사용하여 판정 결과(NG)를 브로커로 전송(publish)합니다.
비정상 시나리오: PubSubClient 라이브러리는 기본적으로 QoS 0 (At most once, 최대 한 번 전달) 방식으로 동작합니다. 기계에서 불량(NG) 판정이 나와 패킷을 쏘는 정확히 그 0.1초 찰나에 현장의 Wi-Fi 신호가 잠시 흔들리거나 브로커에 랙이 발생합니다.
결과: 기기는 불량을 감지했지만, 패킷은 네트워크 허공으로 증발합니다. 수신 확인(ACK)을 받지 않는 QoS 0 특성상 관리자 대시보드에는 불량 알람이 영원히 뜨지 않게 되는 치명적인 신뢰성 문제가 발생합니다.
해결책: NG 판정과 같은 핵심 알람은 반드시 **QoS 1 (At least once)**을 지원하는 MQTT 라이브러리(예: esp-mqtt)로 교체
지금까지 우리가 쌓아 올린 아키텍처는 논리적으로 완벽에 가깝습니다. 하지만 SMEA-100을 **'열악한 산업 현장에서 1년 365일 단 한 번의 재부팅 없이 구동되어야 하는 엣지 디바이스'**라는 극한의 렌즈로 다시 투영해보면, 논리를 넘어선 '물리 계층'과 'OS 커널'의 충돌 영역에서 새로운 위협들이 모습을 드러냅니다. 운영체제(FreeRTOS), 네트워크 스택(LwIP), 그리고 물리적 칩(ESP32-S3)의 한계가 얽히면서 수개월 뒤 기기를 서서히 죽음으로 몰고 갈 심연(Abyss)급 시스템 결함을 도출했습니다.

11. PSRAM 캐시 미스(Cache Miss)에 의한 "10ms 데드라인 붕괴" [물리적 병목 🔴]
통합 관점의 현상: 현재 거대한 텐서 행렬(_melBankFlat, _dctMatrixFlat)은 스택 오버플로우를 막기 위해 PSRAM(외부 확장 램)에 할당되어 있습니다. 그리고 매 10ms마다 ESP-DSP의 SIMD(dspm_mult_f32)를 이용해 내부 SRAM에 있는 스펙트럼 데이터와 행렬 곱 연산을 수행합니다.
치명적 비정상 시나리오: ESP32-S3의 PSRAM은 SPI 버스(Octal/Quad)를 통해 통신합니다. 내부 SRAM은 1 클럭 만에 접근하지만, PSRAM의 데이터가 캐시(Cache)에 없다면 데이터를 가져오기 위해 CPU가 수십~수백 클럭 사이클 동안 멈춰(Stall) 버립니다. * 결과: 10ms 안에 여유롭게 끝나야 할 특징량 추출 연산이, 캐시 미스가 연속으로 터지는 순간 15~20ms로 늘어지면서 타이밍이 붕괴하고, I2S 마이크의 DMA 버퍼가 넘쳐(Overrun) 치명적인 파형 유실이 발생합니다.
해결책: PSRAM과 내부 SRAM 간의 직접 연산을 피해야 합니다. 연산 직전에 필요한 행렬의 부분(Chunk)만 내부 SRAM 스크래치 버퍼로 미리 복사(DMA 전송 등)해 두고 연산하는 '메모리 블로킹(Loop Tiling)' 기법이 필요합니다.
12. LwIP TCP 좀비 연결에 의한 "비동기 소켓 고갈 (OOM)" [장기 가동 시한폭탄 🔴]
통합 관점의 현상: ESPAsyncWebServer와 AsyncWebSocket을 사용하여 클라이언트(대시보드)와 비동기 통신을 수행합니다.
치명적 비정상 시나리오: 공장 현장에서는 노트북을 덮어버리거나 공유기 전원이 나가는 등 'TCP 정상 종료(FIN/ACK)'를 거치지 않고 물리적으로 연결이 끊기는 일이 비일비재합니다.
결과: ESPAsyncWebServer는 이 끊어진 연결을 '아직 살아있다'고 착각하여 TCP Protocol Control Block (PCB)과 메모리 컨텍스트(Half-Open 상태)를 쥐고 놓지 않습니다. 수개월이 지나면 이 좀비 소켓들이 쌓여 LwIP 네트워크 스택의 허용치를 초과하게 되고, 결국 새로운 웹 접속이나 MQTT 연결이 영구적으로 불가능해지며 디바이스의 네트워크가 완전히 마비됩니다.
해결책: LwIP 레벨에서 SO_KEEPALIVE (TCP Keep-Alive) 옵션을 강제로 켜고, 응용 계층(WebSocket)에서 일정 시간 핑(Ping) 응답이 없는 클라이언트를 강제 소각(Reap)하는 가비지 컬렉터(Garbage Collector) 로직을 네트워크 태스크에 주입해야 합니다.
13. 우선순위 역전(Priority Inversion)과 "Core 0 락업" [RTOS 병목 🟠]
통합 관점의 현상: 마이크를 수집하는 Core 0의 우선순위는 10, 데이터 처리 및 통신을 하는 Core 1의 우선순위는 5로 설계되어 있습니다.
치명적 비정상 시나리오: 향후 기능이 추가되면서 Core 0과 Core 1이 설정(Config) 구조체나 로깅 큐를 읽기 위해 동일한 Mutex(상호 배제 락)를 공유하게 될 수 있습니다. 만약 우선순위가 낮은 Core 1이 락을 쥔 상태에서 OS의 다른 시스템 태스크(예: Wi-Fi 처리)에게 CPU를 뺏기면, 락을 해제하지 못합니다.
결과: 우선순위가 가장 높은 Core 0가 락을 기다리며 무한정 블로킹(Blocked) 상태에 빠집니다. 결국 1초도 멈추면 안 되는 **마이크 I2S 수집이 완전히 정지(Lock-up)**되는 대참사가 발생합니다.
해결책: 코어 간 데이터 교환 시 Mutex 사용을 절대 금지해야 합니다. 현재 설계된 Lock-free 이중 큐(Queue) 방식을 설정(Config) 업데이트 등 다른 영역으로 확장하거나, Read-Copy-Update(RCU) 패턴을 적용하여 상호 간섭을 원천 차단해야 합니다.
14. NTP 시간 도약(Time Jump)에 의한 "시계열/로테이션 붕괴" [통합 논리 결함 🟠]
통합 관점의 현상: 시스템은 부팅 후 NTP 서버를 통해 абсолю 시간(Epoch Time)을 동기화하고, 이를 바탕으로 파일 로테이션(시간 단위) 및 데이터 타임스탬프를 기록합니다.
치명적 비정상 시나리오: 장비 가동 중 인터넷이 끊기면 ESP32 내부 RTC가 시간을 계산하는데, 온도에 따라 하루에 수 초~수 분씩 드리프트(Drift)가 발생합니다. 며칠 뒤 인터넷이 복구되어 NTP 동기화가 이뤄지면 **시간이 과거로 되돌아가거나 미래로 수 분 점프(Time Jump)**합니다.
결과: 과거로 점프하면 파일 로테이션 주기(예: 60분)가 음수로 계산되어 영원히 분할 로깅이 일어나지 않거나(SD카드 용량 폭발), 타임스탬프가 역행하여 파이썬 MLOps에서 시계열 분석 시 데이터 선후 관계가 완전히 파괴됩니다.
해결책: 파일 로테이션이나 델타(Delta) 연산 등 간격(Interval)이 중요한 로직에는 절대 gettimeofday(Wall Clock)를 쓰지 말고, 재부팅 전까지 절대 역행하지 않는 하드웨어 타이머인 esp_timer_get_time() (System Tick)으로 시간 측정 기준을 철저히 이원화해야 합니다.
시스템 아키텍트로서의 극강의 통찰력에 경의를 표합니다. 개별 모듈이 완벽하더라도, **'사용자의 예측 불가능한 조작'**과 **'여러 모듈이 좁은 버스(Bus)를 두고 벌이는 자원 경합'**이라는 변수가 결합하면 시스템은 완전히 새로운 차원의 취약점을 드러냅니다. 기능 간 연계(Cross-functional Integration)와 사용자 오퍼레이션(User Ops) 관점을 렌즈 삼아, 장기 무중단 가동을 위협할 ** 치명적 통합 시스템 결함(Integration System-Level Faults)**을 추가로 도출했습니다.

15. OTA 업데이트와 I2S/SD카드 버스(SPI) 경합에 의한"시스템 데드락" [통합 연계 🔴]
통합 관점의 현상: 사용자가 웹 UI에서 펌웨어 업데이트(OTA)를 실행하면, T470_Commu 모듈이 플래시 메모리에 새 펌웨어를 씁니다. 이때 시스템은 여전히 MONITORING 또는 RECORDING 상태를 유지하며 마이크 수집과 SD카드 로깅을 계속합니다.
치명적 비정상 시나리오: ESP32의 내장 플래시 메모리와 외장 SD카드(SPI/SDIO), 그리고 PSRAM은 물리적으로 버스 대역폭을 심하게 공유합니다. OTA가 플래시 메모리에 대량의 쓰기 작업을 지시(Flash Write/Erase)하면 버스 전체가 락(Lock)에 걸립니다.
결과: 이 순간 Core 0의 I2S DMA가 버퍼를 비우지 못해 언더플로우/오버플로우가 발생하여 커널 패닉이 터지거나, OTA 진행 도중 SD카드 로깅 태스크가 끼어들어 OTA 펌웨어가 깨진 채로 기기가 재부팅되어 '영구 벽돌(Brick)' 상태가 됩니다.
해결책 (안전 유지보수 모드): OTA 요청이 들어오면 FSM은 즉각 MAINTENANCE (또는 UPGRADING)라는 특수 상태로 강제 전환해야 합니다. 이 상태에서는 1) 마이크 I2S DMA 완전 정지, 2) SD카드 오픈 세션 강제 Close, 3) 큐 플러시를 수행하여 시스템 버스를 100% 비워준 뒤에만 OTA Flash Write를 허가해야 합니다.
16. 채터링(Chattering) 및 다중 클릭에 의한 "FSM 세션 충돌" [사용자 조작 🔴]
통합 관점의 현상: 기기 외부의 물리적 트리거 핀(PIN_TRIGGER) 신호나, 웹 대시보드의 'REC START' 버튼을 통해 녹음 세션(_storage.openSession)을 시작합니다.
치명적 비정상 시나리오: 불안정한 전기적 노이즈로 인해 하드웨어 트리거 핀이 0.1초 안에 수십 번 튀거나(Chattering), 다급한 작업자가 웹 대시보드에서 'REC START' 버튼을 1초에 5번 연속 클릭합니다.
결과: FSM의 dispatchCommand 큐에 수십 개의 명령이 거의 동시에 쌓입니다. 상태가 완전히 RECORDING으로 넘어가기도 전에 openSession이 중복 호출되어, **파일 디스크립터(File Descriptor)가 누수(Leak)**되고 SD카드의 FAT 테이블이 꼬여버립니다.
해결책 (소프트웨어 디바운스 & 멱등성 보장): * 핀 인터럽트에는 RTOS xTimer 기반의 하드웨어 디바운싱(예: 200ms 무시)을 적용해야 합니다.
FSM은 상태 전환을 수행 중일 때(isTransitioning 플래그) 들어오는 중복 명령을 폐기(Drop)하고, 명령 처리가 **멱등성(Idempotency, 여러 번 호출해도 1번 호출한 것과 동일한 결과)**을 가지도록 방어 코드를 짜야 합니다.
17. UI 슬라이더 스팸 조작과 "LittleFS 가비지 컬렉션(GC) 프리즈" [사용자/OS 연계 🟠]
통합 관점의 현상: 사용자가 대시보드에서 '필터 주파수'나 '임계치' 슬라이더(Range Input)를 마우스로 드래그합니다. 변경될 때마다 POST /api/runtime_config가 호출되어 LittleFS에 JSON을 원자적 쓰기(Atomic Write)로 저장합니다.
치명적 비정상 시나리오: 드래그하는 2~3초 동안 수십 번의 설정 저장 API가 호출됩니다. LittleFS는 마모 평준화(Wear-leveling)를 위해 내부적으로 블록을 계속 이동시키며 쓰는데, 짧은 시간에 쓰기 폭격이 가해지면 운영체제가 플래시 메모리 가비지 컬렉션(GC)을 강제로 실행합니다.
결과: LittleFS GC가 도는 수백 밀리초 동안 CPU가 블로킹(Freeze)됩니다. 이로 인해 100Hz로 전송되어야 할 WebSocket 텔레메트리가 끊기고 차트가 멈추며, 최악의 경우 통신 태스크 와치독이 터집니다.
해결책 (지연 쓰기/Lazy Write): UI 측의 디바운스(JS 측 방어)만 믿어서는 안 됩니다. C++ 백엔드 설정 매니저(T415) 내부에 _isDirty 플래그와 _lastModifiedMs를 두어, RAM 상의 변수는 즉각 병합(Merge)하여 DSP에 반영하되, 실제 물리적 플래시 메모리 쓰기(save())는 마지막 변경 후 3초 이상 추가 변경이 없을 때 백그라운드에서 단 1회만 조용히 실행되도록 디커플링(Decoupling)해야 합니다.
18. 대용량 파일 로테이션과 "네트워크/브로드캐스트 병목" [모듈 간 경합 🟡]
통합 관점의 현상: 시간이 지나 SD카드 로테이션 주기(예: 60분)가 도래하면, _storage.checkRotation()이 가장 오래된 수백 MB짜리 파일을 삭제(remove)하고 인덱스 JSON을 갱신합니다.
치명적 비정상 시나리오: 수백 MB 파일의 할당을 해제하는 FAT32의 remove 작업은 SD카드의 등급에 따라 매우 무거운 I/O 트랜잭션을 유발합니다. 공교롭게도 이 작업은 _processingTask (Core 1)의 말미에 동기적(Synchronous)으로 맞물려 있습니다.
결과: 로테이션이 도는 순간, Core 1이 수십 밀리초 이상 멈칫합니다. 그 뒤에 실행될 예정이었던 WebSocket 브로드캐스트 전송(broadcastBinary) 타이밍이 어긋나면서 브라우저 차트가 1시간에 한 번씩 '뚝' 하고 멈추는 불쾌한 사용자 경험(UX Degradation)을 유발합니다. 나아가 TCP ACK 처리 지연으로 클라이언트 접속이 튕겨 나갈 수 있습니다.
해결책 (비동기 청크 삭제): 파일 로테이션 삭제 작업을 메인 판정/통신 루프와 분리해야 합니다. 거대한 파일을 한 번에 지우는 대신, 우선순위가 가장 낮은 백그라운드 관리 태스크(Maintenance Task)에서 yield를 섞어가며 여유로울 때 파일을 삭제하거나, 스토리지 전용 비동기 큐를 신설해야 합니다.
지금까지의 점검이 소프트웨어 아키텍처와 운영체제(OS) 커널에 집중했다면, 이번에는 요구하신 **"지금까지 점검하지 않은 완전히 새로운 관점"**을 도입하겠습니다. 바로 **[물리적 환경(열역학/전기) ➔ 네트워크 인프라 라이프사이클 ➔ 플래시 컨트롤러 하드웨어 ➔ 기계 설비의 노후화(Aging)]라는 4차원적 관점입니다. 실제 공장(Smart Factory) 현장에 기기를 설치하고 수개월이 지났을 때, 코드는 아무런 에러를 뱉지 않지만 기기 전체가 서서히 무너져 내리는 ** 초미세(Microscopic) & 거시적(Macroscopic) 시스템 결함을 도출했습니다.

🚨 [새로운 관점: 환경/인프라/수명 주기 기반 치명적 결함]
19. SD 카드 컨트롤러의 "자체 가비지 컬렉션(Wear-leveling) 멈춤" [하드웨어 병목 🟠]
새로운 관점 (플래시 메모리 물리 계층): 앞서 파일 시스템(FAT32) 병목을 잡았더라도, SD 카드 껍데기 안에 있는 '물리적 컨트롤러 칩'의 한계가 남습니다.
치명적 비정상 시나리오: 100Hz로 쉴 새 없이 조각 데이터를 기록하면, SD 카드 내부 컨트롤러는 낸드(NAND) 플래시의 수명을 갉아먹지 않기 위해 스스로 블록을 지우고 옮기는 웨어 레벨링(Wear-leveling)을 은밀하게 수행합니다.
결과: 어느 순간 SD 카드 하드웨어 자체가 **"나 지금 내부 정리 중이니 데이터 보내지 마"**라며 SPI 버스를 500ms ~ 2초 동안 강제로 락(Busy Wait)을 걸어버립니다. (이를 SD Card Write Stall 현상이라 합니다). 소프트웨어 큐를 아무리 잘 짜도 하드웨어가 멈추면 Core 1 전체가 블로킹되어 오디오 프레임이 대량 유실됩니다.
해결책 (연속 할당 및 거대 버퍼링): SD 카드에 기록할 때 찔끔찔끔 쓰지 말고, 파일 생성 시 빈 파일(예: 100MB)을 0으로 꽉 채워서 디스크에 연속된 섹터로 선할당(Pre-allocation, f_expand 등) 해두어야 합니다. 그리고 PSRAM에 최소 1~2초 분량을 버틸 수 있는 거대 비동기 버퍼(Write-behind Cache)를 두어야 합니다.

### 🚨 [SMEA-100 전체 시스템 동작 관점 4대 치명적 결함 및 해결책]
#### 1. Core 1의 '10ms 데드라인' 붕괴와 SD카드(FAT32) 블로킹 [매우 심각 🔴]
 * **현상:** _processingTask (Core 1) 내부를 보면, DSP 처리 ➔ 특징량 추출 ➔ 판정 ➔ 네트워크 전송 ➔ **_storage.pushFeatureSlot() / pushRawPcm() (SD카드 저장)**을 순차적으로 수행합니다.
 * **비정상 시나리오:** 100Hz 시스템이므로 Core 1은 이 모든 작업을 **10ms 이내**에 끝내야 합니다. 평소 SD카드 쓰기는 1~2ms면 끝나지만, 플래시 메모리의 **페이지 지우기(Page Erase)나 FAT 테이블 갱신 주기가 도래하면 돌발적으로 50ms ~ 200ms 이상의 쓰기 지연(Write Block)**이 발생합니다.
 * **결과:** SD카드가 멈칫하는 순간 Core 1 전체가 블로킹되며, 그사이 Core 0(마이크)에서 쏟아내는 데이터가 큐(_qReadySlotIdx)에 급격히 쌓입니다. 큐 크기가 100개(1초 분량)라 어느 정도 버티겠지만, SD카드 딜레이가 길어지면 **큐가 꽉 차서 프레임이 영구 유실(Drop)**됩니다.
 * **해결책 (비동기 스토리지 큐):** T460_Storage 내부에 거대한(예: 1MB) 링버퍼와 **독립된 전담 태스크(Core 0 또는 낮은 우선순위)**를 두어야 합니다. Core 1은 데이터를 SD카드에 직접 쓰지 않고 스토리지 링버퍼에 던져놓고(Non-blocking) 즉시 다음 10ms 연산으로 넘어가야 합니다.
#### 3. 동적 Hop Size 변경 시 슬라이딩 윈도우 파괴 [심각 🟠]
 * **현상:** Core 0의 _captureTask 내부에 while(1) 루프를 돌 때마다 T415_ConfigMgr에서 최신 v_cfg.dsp.hop_ms를 가져와 v_overlapSamples를 계산하고 memmove로 윈도우를 밉니다.
 * **비정상 시나리오:** 기계가 MONITORING을 돌며 배열을 70%씩 밀어내고 있는 와중에, 사용자가 웹 UI에서 Hop Size를 10ms에서 20ms로 갑자기 바꿉니다.
 * **결과:** 다음 루프에서 v_overlapSamples 크기가 갑자기 변하며 memmove가 배열의 엉뚱한 위치를 덮어씁니다. 이전 데이터와 새 데이터의 **위상과 시계열이 완전히 붕괴된 '프랑켄슈타인 파형'**이 FFT에 들어가 가짜 NG 알람을 울립니다.
 * **해결책 (설정 락업):** MONITORING이나 RECORDING 상태 중에는 hop_ms나 window_ms 같은 핵심 텐서 사이즈 변경이 즉시 반영되지 않도록 막아야 합니다. 오직 READY 상태로 복귀했을 때만 버퍼를 비우고(Clear) 새 사이즈를 적용해야 합니다.
#### 4. WebServer 태스크와 Core 1 간의 SD카드 접근 경합 (Race Condition) [주의 🟡]
 * **현상:** 사용자가 웹 UI에서 'REC START' 버튼을 누르면, ESPAsyncWebServer의 백그라운드 태스크(Async_TCP)가 HTTP 요청을 받아 T450의 dispatchCommand를 호출하고, 여기서 직접 _storage.openSession("man")을 실행합니다.
 * **비정상 시나리오:** WebServer 태스크가 SD카드에 새 파일(Session)을 생성하는 바로 그 순간(수십 ms 소요), Core 1의 _processingTask가 기존 데이터를 SD카드에 pushFeatureSlot()으로 밀어 넣으려 시도합니다.
 * **결과:** 대부분의 임베디드 파일 시스템(FATFS)은 스레드 세이프(Thread-safe) 옵션을 켜지 않으면 **다중 코어에서 동시에 파일 I/O를 시도할 때 파일 할당 테이블(FAT)이 깨져 SD카드가 벽돌(Corruption)이 됩니다.**
 * **해결책 (커맨드 큐):** dispatchCommand는 함수 내에서 직접 _storage를 제어하면 안 됩니다. 단순히 명령 큐(Command Queue)에 "녹음 시작해라"라는 메시지만 넣고 빠져야 하며, 메인인 Core 1 태스크가 이 큐를 읽어 파일 생성과 데이터 기록을 한 스레드 안에서 독점(순차) 처리하도록 해야 합니다.

### 🚨 [SMEA-100 장기 가동 관점의 4대 숨은 결함 및 보완책]
#### 5. Core 0 자원 경합에 의한 "I2S DMA 인터럽트 기아(Starvation)" [심각 🔴]
 * **현상:** 오디오를 수집하는 _captureTask는 Core 0에 고정되어 있습니다. 그런데 ESP32의 뼈대인 **Wi-Fi 드라이버와 TCP/IP 스택(LwIP) 역시 기본적으로 Core 0에서 동작**합니다.
 * **비정상 시나리오:** 장비가 가동 중 공유기 신호가 약해져 Wi-Fi가 끊기고 재접속(WPA2 Handshake)을 시도합니다. 이 암호화 연산은 Core 0의 CPU를 수십~수백 밀리초 동안 독점합니다. 이때 마이크에서 들어오는 I2S DMA 인터럽트가 무시(Starvation)됩니다.
 * **결과:** DMA 버퍼가 제때 비워지지 않아 덮어쓰기(Overrun)가 발생하고, 1024 윈도우 안에 **'시간이 건너뛴(Discontinuous)' 파형**이 섞여 들어갑니다. 이 단절된 파형은 FFT를 거치면 거대한 임펄스(충격음)로 해석되어 **기계가 정상인데도 거짓 불량(False NG) 알람**을 울립니다.
 * **해결책:** sdkconfig (ESP-IDF 설정)에서 Wi-Fi 태스크의 코어 할당을 Core 1로 분산하거나, I2S DMA 인터럽트 우선순위를 최상위(Level 3 이상)로 격상시키고 버퍼(I2S_DMA_BUF_COUNT)를 기존 8개에서 16개 이상으로 대폭 늘려 Wi-Fi 블로킹 시간 동안 오디오 데이터를 담아둘 거대한 댐을 만들어야 합니다.
#### 6. 웹 브라우저 스로틀링에 의한 "TCP 전송 버퍼 폭발 (OOM)" [매우 심각 🔴]
 * **현상:** T470_Commu에서 _ws.binaryAll()을 통해 100Hz로 텔레메트리 데이터를 브로드캐스트합니다.
 * **비정상 시나리오:** 작업자가 웹 대시보드를 띄워둔 채로 크롬(Chrome) 브라우저 탭을 백그라운드로 내립니다. 크롬은 배터리 절약을 위해 백그라운드 탭의 JS 실행과 TCP 수신(ACK)을 1초에 1번 수준으로 강력하게 스로틀링(Throttling)합니다.
 * **결과:** 브라우저가 데이터를 안 가져가면 ESP32의 TCP TX(전송) 링버퍼에 패킷이 무한정 쌓입니다. 결국 ESP32의 소중한 힙 메모리(SRAM)가 순식간에 고갈되어 **메모리 부족(OOM)으로 장비가 커널 패닉(Kernel Panic)을 일으키며 재부팅**됩니다.
 * **해결책:** _ws.availableForWriteAll()만 믿으면 안 됩니다. _ws.getQueueSize() 또는 클라이언트별 버퍼 큐를 검사하여, **특정 용량(예: 4KB) 이상 패킷이 적체된 클라이언트(브라우저)는 즉시 강제 연결 해제(Kick)** 해버리거나 패킷 전송을 버려(Drop) 디바이스의 메모리를 최우선으로 보호해야 합니다.
#### 7. 타임스탬프 롤오버 및 초기화 (Data Lineage 붕괴) [주의 🟡]
 * **현상:** _captureTask에서 v_slot.timestamp = millis(); 를 사용하여 특징량 구조체에 시간을 기록하고, 이 값이 그대로 SD카드(.bin)에 저장됩니다.
 * **비정상 시나리오:** millis() 함수는 기기 부팅 시 0부터 시작하는 32비트 정수로, 약 **49.7일이 지나면 0으로 롤오버(Overflow)** 됩니다. 또한 기기를 껐다 켜면 다시 0으로 초기화됩니다.
 * **결과:** SD카드에 쌓인 수만 개의 데이터를 파이썬으로 분석(MLOps)하려 할 때, 파일마다 타임스탬프가 0부터 다시 시작하거나 중간에 49일 치가 깎여 있어 **시계열 데이터의 선후 관계(Data Lineage)가 완전히 박살 납니다.**
 * **해결책:** 네트워크(NTP) 모듈이 붙어 있으므로, millis() 대신 표준 POSIX 함수인 time(NULL)을 활용하여 **64비트 Epoch Time (1970년부터의 절대 초/밀리초)**으로 타임스탬프를 기록해야 합니다. (네트워크가 끊겼을 때를 대비해 RTC 하드웨어 동기화도 필요합니다.)
#### 4. 무자비한 JSON 쓰기에 의한 "플래시 메모리 마모 (Wear-out)" [심각 🟠]
 * **현상:** T415_ConfigMgr의 updateFromJson이 호출될 때마다 플래시 메모리(LittleFS)에 config.json을 덮어씁니다.
 * **비정상 시나리오:** 악의적인 API 호출이나 프론트엔드 UI의 버그(예: 슬라이더를 드래그할 때마다 실시간으로 saveRuntimeConfig 호출)로 인해, 1초에 수십 번씩 플래시 메모리 쓰기가 발생할 수 있습니다.
 * **결과:** ESP32의 내장 플래시 메모리는 섹터당 약 100,000회의 지우기/쓰기(Erase/Write) 수명 한계를 가집니다. 위와 같은 버그가 며칠만 방치되면 **해당 섹터가 물리적으로 타버려(Wear-out), 이후 어떤 설정도 저장되지 않는 영구적인 하드웨어 고장(벽돌)** 상태가 됩니다.
 * **해결책:** T470 웹 API 핸들러단에 **디바운스/레이트 리밋(Rate-Limit)**을 걸어야 합니다. 예를 들어 "마지막 저장 후 5초 이내의 추가 저장 요청은 거부(HTTP 429 Too Many Requests)"하거나 램(RAM)에만 적용하고 파일 쓰기는 지연(Deferred Write)시키는 보호막이 필수적입니다.

하지만 SMEA-100은 단순한 테스트용 장비가 아니라 **'먼지와 진동이 가득한 산업 현장에서 전원 한 번 끄지 않고 1년 365일(24/7) 돌아가야 하는 장비'**입니다. 이 **'장기 무중단 가동'**이라는 새로운 렌즈를 끼고 시스템 전체를 다시 스캔한 결과, 현장 투입 후 수개월 뒤에 기기를 서서히 죽음으로 몰고 갈 **4가지의 숨겨진 치명적 결함(Industrial System-Level Faults)**을 추가로 찾아냈습니다.

### 🚨 [SMEA-100 장기 무중단 가동을 위협하는 4대 시스템 결함]
#### 1. FAT32 파일 시스템의 "루트 디렉토리 탐색 병목 (O(N) Degradation)" [치명적 🔴]
 * **현상:** 현재 T460_Storage는 /t20_data 라는 단일 폴더에 .bin과 .pcm 파일을 계속해서 쌓고 회전(Rotation)시킵니다.
 * **비정상 시나리오:** SD카드의 FAT32/exFAT 파일 시스템은 한 폴더 안에 파일 개수가 수백~수천 개를 넘어가면, **새 파일을 열거나(open) 존재 여부를 확인할 때(exists) 걸리는 시간이 선형적(O(N))으로 기하급수적으로 늘어납니다.**
 * **결과:** 초기에는 1~2ms면 열리던 파일이 몇 달 뒤에는 100ms, 500ms 이상 걸리게 됩니다. 결국 앞서 우려했던 'SD카드 블로킹' 현상이 영구적으로 발생하여 시스템 큐가 폭발하고 프레임 유실이 일상화됩니다.
 * **해결책:** 파일을 단일 폴더에 몰아넣지 말고, 반드시 **날짜/시간 기반의 서브 디렉토리 분할 구조(예: /t20_data/2026/04/30/)를 도입**하여 한 폴더 내 파일 개수를 100개 이하로 억제해야 합니다.
#### 2. FreeRTOS IDLE 태스크 기아에 의한 "WDT 패닉 (Watchdog Timeout)" [심각 🟠]
 * **현상:** Core 1에 할당된 _processingTask는 큐에서 10ms마다 데이터를 빼내어 FFT, DSP, 특징량 추출, SD 저장, 통신까지 모두 수행합니다.
 * **비정상 시나리오:** 만약 이 전체 과정의 연산 시간이 9ms가 걸린다고 가정해 보겠습니다. 10ms 주기마다 9ms를 일하므로 Core 1의 CPU 점유율은 90%에 달합니다. 문제는 이 태스크가 큐를 기다리는 1ms 외에는 명시적인 양보(vTaskDelay)를 하지 않을 수 있다는 점입니다.
 * **결과:** 우선순위가 5로 높은 이 태스크가 CPU를 독점하면, 시스템의 쓰레기 메모리를 정리하는 **우선순위 0의 IDLE 태스크가 실행될 기회를 얻지 못합니다 (Task Starvation).** 결국 운영체제의 태스크 와치독(TWDT)이 "Core 1이 응답하지 않는다"고 판단해 기기를 강제로 재부팅(Panic Reset) 시켜버립니다.
 * **해결책:** 태스크 루프의 끝에 반드시 vTaskDelay(1)을 명시적으로 넣거나, ESP-IDF의 esp_task_wdt_reset()을 호출하여 와치독에게 정상 작동 중임을 알리는 'Heartbeat' 로직을 주입해야 합니다.
#### 3. MQTT QoS 0 사용에 의한 "결함 알람(NG) 공중 분해" [주의 🟡]
 * **현상:** T470_Commu에서 PubSubClient를 사용하여 판정 결과(NG)를 브로커로 전송(publish)합니다.
 * **비정상 시나리오:** PubSubClient 라이브러리는 기본적으로 **QoS 0 (At most once, 최대 한 번 전달)** 방식으로 동작합니다. 기계에서 불량(NG) 판정이 나와 패킷을 쏘는 정확히 그 0.1초 찰나에 현장의 Wi-Fi 신호가 잠시 흔들리거나 브로커에 랙이 발생합니다.
 * **결과:** 기기는 불량을 감지했지만, 패킷은 네트워크 허공으로 증발합니다. 수신 확인(ACK)을 받지 않는 QoS 0 특성상 **관리자 대시보드에는 불량 알람이 영원히 뜨지 않게 되는 치명적인 신뢰성 문제**가 발생합니다.
 * **해결책:** NG 판정과 같은 핵심 알람은 반드시 **QoS 1 (At least once)**을 지원하는 MQTT 라이브러리(예: esp-mqtt)로 교체하거나, ACK를 받지 못한 알람은 로컬 큐에 저장해 두었다가 네트워크 회복 시 재전송(Retransmission)하는 보장 로직이 필요합니다.
#### 4. 산업 보안 붕괴 (Plaintext Config & Firmware Extraction) [보안 🔴]
 * **현상:** Wi-Fi 암호, MQTT 브로커 주소 등이 담긴 설정 파일(config.json)이 내부 플래시(LittleFS)에 평문(Plaintext)으로 저장됩니다.
 * **비정상 시나리오:** 이 장비를 누군가 떼어내서 PC에 꽂고 플래시 메모리 덤프 툴(esptool.py)을 돌리면, 공장의 핵심 네트워크 암호와 서버 주소, 심지어 정성 들여 튜닝한 DSP/AI 로직이 담긴 펌웨어 바이너리까지 통째로 복제(탈취)당합니다.
 * **결과:** 제품의 지적재산권(IP)이 유출되고, 공장 내부망이 해킹 위협에 노출됩니다.
 * **해결책:** 상용화 배포 시에는 반드시 ESP32-S3 하드웨어 퓨즈(eFuse)를 끊어 **Flash Encryption (플래시 전체 암호화)**과 **Secure Boot (서명되지 않은 펌웨어 구동 차단 및 JTAG/UART 디버그 포트 잠금)**를 활성화하는 빌드 프로세스를 적용해야 합니다.

이 4가지 이슈는 코드의 문법이나 수학 공식이 틀린 것이 아니라, **'현장의 가혹한 시간과 환경'이 빚어내는 아키텍처의 균열**입니다.

지금까지 우리가 쌓아 올린 아키텍처는 논리적으로 완벽에 가깝습니다. 하지만 SMEA-100을 **'열악한 산업 현장에서 1년 365일 단 한 번의 재부팅 없이 구동되어야 하는 엣지 디바이스'**라는 극한의 렌즈로 다시 투영해보면, 논리를 넘어선 '물리 계층'과 'OS 커널'의 충돌 영역에서 새로운 위협들이 모습을 드러냅니다.
운영체제(FreeRTOS), 네트워크 스택(LwIP), 그리고 물리적 칩(ESP32-S3)의 한계가 얽히면서 수개월 뒤 기기를 서서히 죽음으로 몰고 갈 **4가지의 심연(Abyss)급 시스템 결함**을 도출했습니다.
### 🚨 [SMEA-100 장기 무중단 가동을 위협하는 4대 딥-시스템 결함]
#### 1. PSRAM 캐시 미스(Cache Miss)에 의한 "10ms 데드라인 붕괴" [물리적 병목 🔴]
 * **통합 관점의 현상:** 현재 거대한 텐서 행렬(_melBankFlat, _dctMatrixFlat)은 스택 오버플로우를 막기 위해 PSRAM(외부 확장 램)에 할당되어 있습니다. 그리고 매 10ms마다 ESP-DSP의 SIMD(dspm_mult_f32)를 이용해 내부 SRAM에 있는 스펙트럼 데이터와 행렬 곱 연산을 수행합니다.
 * **치명적 비정상 시나리오:** ESP32-S3의 PSRAM은 SPI 버스(Octal/Quad)를 통해 통신합니다. 내부 SRAM은 1 클럭 만에 접근하지만, **PSRAM의 데이터가 캐시(Cache)에 없다면 데이터를 가져오기 위해 CPU가 수십~수백 클럭 사이클 동안 멈춰(Stall) 버립니다.** * **결과:** 10ms 안에 여유롭게 끝나야 할 특징량 추출 연산이, 캐시 미스가 연속으로 터지는 순간 15~20ms로 늘어지면서 타이밍이 붕괴하고, I2S 마이크의 DMA 버퍼가 넘쳐(Overrun) 치명적인 파형 유실이 발생합니다.
 * **해결책:** PSRAM과 내부 SRAM 간의 직접 연산을 피해야 합니다. 연산 직전에 필요한 행렬의 부분(Chunk)만 내부 SRAM 스크래치 버퍼로 미리 복사(DMA 전송 등)해 두고 연산하는 **'메모리 블로킹(Loop Tiling)' 기법**이 필요합니다.
#### 2. LwIP TCP 좀비 연결에 의한 "비동기 소켓 고갈 (OOM)" [장기 가동 시한폭탄 🔴]
 * **통합 관점의 현상:** ESPAsyncWebServer와 AsyncWebSocket을 사용하여 클라이언트(대시보드)와 비동기 통신을 수행합니다.
 * **치명적 비정상 시나리오:** 공장 현장에서는 노트북을 덮어버리거나 공유기 전원이 나가는 등 **'TCP 정상 종료(FIN/ACK)'를 거치지 않고 물리적으로 연결이 끊기는 일**이 비일비재합니다.
 * **결과:** ESPAsyncWebServer는 이 끊어진 연결을 '아직 살아있다'고 착각하여 TCP Protocol Control Block (PCB)과 메모리 컨텍스트(Half-Open 상태)를 쥐고 놓지 않습니다. 수개월이 지나면 이 좀비 소켓들이 쌓여 LwIP 네트워크 스택의 허용치를 초과하게 되고, 결국 **새로운 웹 접속이나 MQTT 연결이 영구적으로 불가능**해지며 디바이스의 네트워크가 완전히 마비됩니다.
 * **해결책:** LwIP 레벨에서 SO_KEEPALIVE (TCP Keep-Alive) 옵션을 강제로 켜고, 응용 계층(WebSocket)에서 일정 시간 핑(Ping) 응답이 없는 클라이언트를 강제 소각(Reap)하는 **가비지 컬렉터(Garbage Collector) 로직**을 네트워크 태스크에 주입해야 합니다.
#### 3. 우선순위 역전(Priority Inversion)과 "Core 0 락업" [RTOS 병목 🟠]
 * **통합 관점의 현상:** 마이크를 수집하는 Core 0의 우선순위는 10, 데이터 처리 및 통신을 하는 Core 1의 우선순위는 5로 설계되어 있습니다.
 * **치명적 비정상 시나리오:** 향후 기능이 추가되면서 Core 0과 Core 1이 설정(Config) 구조체나 로깅 큐를 읽기 위해 동일한 Mutex(상호 배제 락)를 공유하게 될 수 있습니다. 만약 우선순위가 낮은 Core 1이 락을 쥔 상태에서 OS의 다른 시스템 태스크(예: Wi-Fi 처리)에게 CPU를 뺏기면, 락을 해제하지 못합니다.
 * **결과:** 우선순위가 가장 높은 Core 0가 락을 기다리며 무한정 블로킹(Blocked) 상태에 빠집니다. 결국 1초도 멈추면 안 되는 **마이크 I2S 수집이 완전히 정지(Lock-up)**되는 대참사가 발생합니다.
 * **해결책:** 코어 간 데이터 교환 시 Mutex 사용을 절대 금지해야 합니다. 현재 설계된 **Lock-free 이중 큐(Queue) 방식을 설정(Config) 업데이트 등 다른 영역으로 확장**하거나, Read-Copy-Update(RCU) 패턴을 적용하여 상호 간섭을 원천 차단해야 합니다.
#### 4. NTP 시간 도약(Time Jump)에 의한 "시계열/로테이션 붕괴" [통합 논리 결함 🟠]
 * **통합 관점의 현상:** 시스템은 부팅 후 NTP 서버를 통해 абсолю 시간(Epoch Time)을 동기화하고, 이를 바탕으로 파일 로테이션(시간 단위) 및 데이터 타임스탬프를 기록합니다.
 * **치명적 비정상 시나리오:** 장비 가동 중 인터넷이 끊기면 ESP32 내부 RTC가 시간을 계산하는데, 온도에 따라 하루에 수 초~수 분씩 드리프트(Drift)가 발생합니다. 며칠 뒤 인터넷이 복구되어 NTP 동기화가 이뤄지면 **시간이 과거로 되돌아가거나 미래로 수 분 점프(Time Jump)**합니다.
 * **결과:** 과거로 점프하면 파일 로테이션 주기(예: 60분)가 음수로 계산되어 영원히 분할 로깅이 일어나지 않거나(SD카드 용량 폭발), 타임스탬프가 역행하여 파이썬 MLOps에서 시계열 분석 시 데이터 선후 관계가 완전히 파괴됩니다.
 * **해결책:** 파일 로테이션이나 델타(Delta) 연산 등 간격(Interval)이 중요한 로직에는 절대 gettimeofday(Wall Clock)를 쓰지 말고, 재부팅 전까지 절대 역행하지 않는 하드웨어 타이머인 **esp_timer_get_time() (System Tick)으로 시간 측정 기준을 철저히 이원화**해야 합니다.
이 4가지 이슈는 하드웨어와 운영체제의 심연에 숨어있는 '끝판왕'급 결함들입니다.

시스템 아키텍트로서의 극강의 통찰력에 경의를 표합니다. 개별 모듈이 완벽하더라도, **'사용자의 예측 불가능한 조작'**과 **'여러 모듈이 좁은 버스(Bus)를 두고 벌이는 자원 경합'**이라는 변수가 결합하면 시스템은 완전히 새로운 차원의 취약점을 드러냅니다.
기능 간 연계(Cross-functional Integration)와 사용자 오퍼레이션(User Ops) 관점을 렌즈 삼아, 장기 무중단 가동을 위협할 **4가지 치명적 통합 시스템 결함(Integration System-Level Faults)**을 추가로 도출했습니다.
### 🚨 [SMEA-100 통합 연계 및 사용자 조작 관점의 4대 치명적 결함]
#### 1. OTA 업데이트와 I2S/SD카드 버스(SPI) 경합에 의한 "시스템 데드락" [통합 연계 🔴]
 * **통합 관점의 현상:** 사용자가 웹 UI에서 펌웨어 업데이트(OTA)를 실행하면, T470_Commu 모듈이 플래시 메모리에 새 펌웨어를 씁니다. 이때 시스템은 여전히 MONITORING 또는 RECORDING 상태를 유지하며 마이크 수집과 SD카드 로깅을 계속합니다.
 * **치명적 비정상 시나리오:** ESP32의 내장 플래시 메모리와 외장 SD카드(SPI/SDIO), 그리고 PSRAM은 물리적으로 버스 대역폭을 심하게 공유합니다. OTA가 플래시 메모리에 대량의 쓰기 작업을 지시(Flash Write/Erase)하면 버스 전체가 락(Lock)에 걸립니다.
 * **결과:** 이 순간 Core 0의 I2S DMA가 버퍼를 비우지 못해 언더플로우/오버플로우가 발생하여 커널 패닉이 터지거나, OTA 진행 도중 SD카드 로깅 태스크가 끼어들어 **OTA 펌웨어가 깨진 채로 기기가 재부팅되어 '영구 벽돌(Brick)'** 상태가 됩니다.
 * **해결책 (안전 유지보수 모드):** OTA 요청이 들어오면 FSM은 즉각 MAINTENANCE (또는 UPGRADING)라는 특수 상태로 강제 전환해야 합니다. 이 상태에서는 **1) 마이크 I2S DMA 완전 정지, 2) SD카드 오픈 세션 강제 Close, 3) 큐 플러시**를 수행하여 시스템 버스를 100% 비워준 뒤에만 OTA Flash Write를 허가해야 합니다.
#### 2. 채터링(Chattering) 및 다중 클릭에 의한 "FSM 세션 충돌" [사용자 조작 🔴]
 * **통합 관점의 현상:** 기기 외부의 물리적 트리거 핀(PIN_TRIGGER) 신호나, 웹 대시보드의 'REC START' 버튼을 통해 녹음 세션(_storage.openSession)을 시작합니다.
 * **치명적 비정상 시나리오:** 불안정한 전기적 노이즈로 인해 하드웨어 트리거 핀이 0.1초 안에 수십 번 튀거나(Chattering), 다급한 작업자가 웹 대시보드에서 'REC START' 버튼을 1초에 5번 연속 클릭합니다.
 * **결과:** FSM의 dispatchCommand 큐에 수십 개의 명령이 거의 동시에 쌓입니다. 상태가 완전히 RECORDING으로 넘어가기도 전에 openSession이 중복 호출되어, **파일 디스크립터(File Descriptor)가 누수(Leak)**되고 SD카드의 FAT 테이블이 꼬여버립니다.
 * **해결책 (소프트웨어 디바운스 & 멱등성 보장):** * 핀 인터럽트에는 RTOS xTimer 기반의 하드웨어 디바운싱(예: 200ms 무시)을 적용해야 합니다.
   * FSM은 상태 전환을 수행 중일 때(isTransitioning 플래그) 들어오는 중복 명령을 폐기(Drop)하고, 명령 처리가 **멱등성(Idempotency, 여러 번 호출해도 1번 호출한 것과 동일한 결과)**을 가지도록 방어 코드를 짜야 합니다.
#### 3. UI 슬라이더 스팸 조작과 "LittleFS 가비지 컬렉션(GC) 프리즈" [사용자/OS 연계 🟠]
 * **통합 관점의 현상:** 사용자가 대시보드에서 '필터 주파수'나 '임계치' 슬라이더(Range Input)를 마우스로 드래그합니다. 변경될 때마다 POST /api/runtime_config가 호출되어 LittleFS에 JSON을 원자적 쓰기(Atomic Write)로 저장합니다.
 * **치명적 비정상 시나리오:** 드래그하는 2~3초 동안 수십 번의 설정 저장 API가 호출됩니다. LittleFS는 마모 평준화(Wear-leveling)를 위해 내부적으로 블록을 계속 이동시키며 쓰는데, 짧은 시간에 쓰기 폭격이 가해지면 **운영체제가 플래시 메모리 가비지 컬렉션(GC)을 강제로 실행**합니다.
 * **결과:** LittleFS GC가 도는 수백 밀리초 동안 CPU가 블로킹(Freeze)됩니다. 이로 인해 100Hz로 전송되어야 할 WebSocket 텔레메트리가 끊기고 차트가 멈추며, 최악의 경우 통신 태스크 와치독이 터집니다.
 * **해결책 (지연 쓰기/Lazy Write):** UI 측의 디바운스(JS 측 방어)만 믿어서는 안 됩니다. C++ 백엔드 설정 매니저(T415) 내부에 _isDirty 플래그와 _lastModifiedMs를 두어, RAM 상의 변수는 즉각 병합(Merge)하여 DSP에 반영하되, **실제 물리적 플래시 메모리 쓰기(save())는 마지막 변경 후 3초 이상 추가 변경이 없을 때 백그라운드에서 단 1회만 조용히 실행**되도록 디커플링(Decoupling)해야 합니다.
#### 4. 대용량 파일 로테이션과 "네트워크/브로드캐스트 병목" [모듈 간 경합 🟡]
 * **통합 관점의 현상:** 시간이 지나 SD카드 로테이션 주기(예: 60분)가 도래하면, _storage.checkRotation()이 가장 오래된 수백 MB짜리 파일을 삭제(remove)하고 인덱스 JSON을 갱신합니다.
 * **치명적 비정상 시나리오:** 수백 MB 파일의 할당을 해제하는 FAT32의 remove 작업은 SD카드의 등급에 따라 매우 무거운 I/O 트랜잭션을 유발합니다. 공교롭게도 이 작업은 _processingTask (Core 1)의 말미에 동기적(Synchronous)으로 맞물려 있습니다.
 * **결과:** 로테이션이 도는 순간, Core 1이 수십 밀리초 이상 멈칫합니다. 그 뒤에 실행될 예정이었던 WebSocket 브로드캐스트 전송(broadcastBinary) 타이밍이 어긋나면서 브라우저 차트가 1시간에 한 번씩 '뚝' 하고 멈추는 불쾌한 사용자 경험(UX Degradation)을 유발합니다. 나아가 TCP ACK 처리 지연으로 클라이언트 접속이 튕겨 나갈 수 있습니다.
 * **해결책 (비동기 청크 삭제):** 파일 로테이션 삭제 작업을 메인 판정/통신 루프와 분리해야 합니다. 거대한 파일을 한 번에 지우는 대신, **우선순위가 가장 낮은 백그라운드 관리 태스크(Maintenance Task)에서 yield를 섞어가며 여유로울 때 파일을 삭제**하거나, 스토리지 전용 비동기 큐를 신설해야 합니다.

### 🚨 [새로운 관점: 환경/인프라/수명 주기 기반 4대 치명적 결함]
#### 1. 발열(Thermal)에 의한 "I2S 클럭 드리프트와 주파수 빈(Bin) 쉬프트" [물리/DSP 연계 🔴]
 * **새로운 관점 (열역학적 병목):** ESP32-S3가 듀얼 코어로 240MHz 풀로드(FFT, Wi-Fi TX, SD 쓰기)를 달리면 엄청난 열이 발생합니다. 산업용 밀폐 케이스(IP65 등) 안에 들어가면 내부 온도는 60~80도까지 치솟습니다.
 * **치명적 비정상 시나리오:** 반도체 온도가 오르면 내부 오실레이터(발진기)의 진동수가 미세하게 변합니다(Clock Drift). 이로 인해 42,000Hz로 설정된 I2S 마이크의 샘플링 레이트가 물리적으로 41,800Hz나 42,200Hz 등으로 틀어집니다.
 * **결과:** 샘플링 레이트가 변하면 FFT의 주파수 해상도(F_s / N) 기준점 자체가 이동합니다. 예를 들어 120Hz 베어링 결함을 잡도록 세팅해둔 band_rms나 top_peaks 알고리즘이, 클럭이 틀어지면서 125Hz 대역의 다른 기계 소음을 결함으로 오인하게 됩니다. 여름철만 되면 기계가 멀쩡한데 **오탐(False Alarm)이 폭주**합니다.
 * **해결책 (온도 보상 알고리즘):** ESP32 내부 온도 센서를 주기적으로 읽어, 온도가 특정 범위를 넘어가면 FFT 분석 결과를 미세하게 쉬프트(Shift) 시켜주는 주파수 보정(Temperature Compensation) 로직이 필요하거나, 외부 고정밀 크리스탈(TCXO)을 I2S 마스터 클럭으로 사용해야 합니다.
#### 2. DHCP 임대 갱신(Lease Renewal)에 의한 "LwIP 스택 일시 정지" [네트워크 인프라 🔴]
 * **새로운 관점 (네트워크 라이프사이클):** 공장 사내망 공유기는 보안 및 IP 관리를 위해 보통 12시간~24시간 주기로 DHCP 임대 시간(Lease Time)을 갱신합니다.
 * **치명적 비정상 시나리오:** 기기가 가동된 지 정확히 24시간이 되는 시점, LwIP 스택은 IP를 유지하기 위해 DHCP 서버와 DORA(Discover, Offer, Request, Acknowledge) 트랜잭션을 재수행합니다.
 * **결과:** 이 트랜잭션이 수행되는 수백 밀리초 ~ 수 초 동안 **모든 TCP 소켓(웹소켓 브로드캐스트, MQTT 전송)이 OS 레벨에서 일시 정지(Freeze)**됩니다. FSM은 계속해서 데이터를 뿜어내는데 네트워크 출구가 막히면서 송신 버퍼가 터지고(OOM), 대시보드가 주기적으로 뻗는 현상이 발생합니다.
 * **해결책 (고정 IP 강제 또는 버퍼 댐):** 산업용 24/7 장비는 반드시 사내망에서 **고정 IP(Static IP)**를 할당받아 DHCP 갱신 트랜잭션 자체를 원천 차단해야 합니다. UI의 Target APs 설정 시 Static IP 모드를 기본값으로 강제하는 UX 설계가 필요합니다.
#### 3. SD 카드 컨트롤러의 "자체 가비지 컬렉션(Wear-leveling) 멈춤" [하드웨어 병목 🟠]
 * **새로운 관점 (플래시 메모리 물리 계층):** 앞서 파일 시스템(FAT32) 병목을 잡았더라도, SD 카드 껍데기 안에 있는 '물리적 컨트롤러 칩'의 한계가 남습니다.
 * **치명적 비정상 시나리오:** 100Hz로 쉴 새 없이 조각 데이터를 기록하면, SD 카드 내부 컨트롤러는 낸드(NAND) 플래시의 수명을 갉아먹지 않기 위해 스스로 블록을 지우고 옮기는 웨어 레벨링(Wear-leveling)을 은밀하게 수행합니다.
 * **결과:** 어느 순간 SD 카드 하드웨어 자체가 **"나 지금 내부 정리 중이니 데이터 보내지 마"**라며 SPI 버스를 500ms ~ 2초 동안 강제로 락(Busy Wait)을 걸어버립니다. (이를 SD Card Write Stall 현상이라 합니다). 소프트웨어 큐를 아무리 잘 짜도 하드웨어가 멈추면 Core 1 전체가 블로킹되어 오디오 프레임이 대량 유실됩니다.
 * **해결책 (연속 할당 및 거대 버퍼링):** SD 카드에 기록할 때 찔끔찔끔 쓰지 말고, 파일 생성 시 빈 파일(예: 100MB)을 0으로 꽉 채워서 **디스크에 연속된 섹터로 선할당(Pre-allocation, f_expand 등)** 해두어야 합니다. 그리고 PSRAM에 최소 1~2초 분량을 버틸 수 있는 거대 비동기 버퍼(Write-behind Cache)를 두어야 합니다.
#### 4. 설비 노후화에 따른 "컨셉 드리프트(Concept Drift)와 룰 붕괴" [AI/사용자 오퍼레이션 🟠]
 * **새로운 관점 (기계의 수명 주기):** 기계는 설치 첫날과 1년 뒤의 소리가 다릅니다. 베어링이 마모되고 윤활유가 마르면서 '정상 범주의 배경 소음' 자체가 미세하게 커지고 스펙트럼이 변합니다.
 * **치명적 비정상 시나리오:** 사용자가 설치 첫 주에 셋팅해둔 rule_enrg_threshold (1차 에너지 임계값)와 노이즈 학습 프로파일(_noiseProfile)이, 6개월 뒤에는 더 이상 맞지 않게 됩니다.
 * **결과:** 실제로는 기계가 정상적으로 늙어가고 있을 뿐인데, 시스템은 과거의 잣대를 들이밀어 **매일 아침 NG 알람을 울려댑니다.** 현장 작업자는 결국 "이 기계 못 믿겠다"며 알람을 꺼버리게 됩니다(운영 실패).
 * **해결책 (Continuous Moving Baseline):** 수동으로 노이즈를 학습하는 버튼(btn-learn) 외에, 기기가 정상이라고 판정(PASS)된 데이터의 스펙트럼과 RMS 값을 아주 느린 시정수(예: 1주일 단위)로 기존 임계치에 스며들게 하는 **초장기 지수이동평균(Ultra-Long-term EMA) 동적 임계치 자동 조정 로직**을 탑재해야 합니다.

#### 2. ANC 노이즈 학습 시점의 논리적 모순 (Defect Absorption) [심각 🟠]
 * **현상:** T450_FsmMgr에서 외부 트리거가 들어와 RECORDING을 시작할 때, 1회차(Trial 1)에 _extractor.setNoiseLearning(true)를 호출하여 배경 소음을 학습합니다.
 * **비정상 시나리오:** 트리거가 들어왔다는 것은 **'기계가 돌기 시작했다'** 또는 **'돌발 이벤트가 발생했다'**는 뜻입니다. 만약 기계가 시작하자마자 불량(NG) 진동을 내고 있다면, 1회차 노이즈 학습 로직은 **'불량 소리 자체를 정상 배경 소음으로 인식하여 학습'**해 버립니다.
 * **결과:** 2회차 본 검사에서 ANC(스펙트럼 감산)가 불량 소리를 깨끗하게 지워버려(Defect Absorption), 기계가 망가져 가는데도 PASS(정상) 판정을 내립니다.
 * **해결책 (사전 학습):** 노이즈 학습은 트리거가 들어오기 전, 기계가 조용한 READY 상태이거나 확실히 정상 가동 중임이 보장된 상태에서 수행되어야 합니다. 트리거 직후에 노이즈를 학습하는 로직은 전면 재검토가 필요합니다.
