# 1. System Overview

## 1.1 시스템 목적

본 시스템은 **ESP32‑S3 기반 고주파 IMU Logger**이다.

주 목적

* Gyroflow 영상 안정화용 IMU 데이터 기록
* 자전거 / 액션캠 / 드론 장착
* 고주파 센서 데이터 수집

목표

* **1000Hz 안정 로깅**
* **전원 차단 데이터 보호**
* **Gyroflow 호환 데이터 기록**

참조

Gyroflow

[https://gyroflow.xyz/](https://gyroflow.xyz/)

NiYien Senseflow A1

[https://www.niyien.com](https://www.niyien.com)

목표

The system shall tolerate SD latency up to **100ms**.

방법

* double buffer

방법

구조

## 4.6 장시간 안정성

목표

## 4.7 전력 효율

The system shall support battery operation.

---

예

SD → SdWriteTask State → SystemTask Config → ConfigManager

# 14. System Timing Budget

본 장은 **1000Hz 안정 로깅을 보장하기 위한 시간 예산(Time Budget)*\*을 정의한다.

1000Hz 샘플링에서는 **1ms (1000µs)** 안에 다음 처리가 완료되어야 한다.

* IMU 샘플 획득
* 샘플 버퍼 저장
* 로그 조립
* SD write 큐 전달

실제 SD write는 비동기 처리되므로 **sampling 경로는 절대 block 되지 않아야 한다.**

---

## 14.1 기본 시간 예산

| 단계                        | 목표 시간    |
| ------------------------- | -------- |
| IMU read                  | < 80 µs  |
| sample push (ring buffer) | < 10 µs  |
| log assemble              | < 50 µs  |
| queue send                | < 10 µs  |
| 여유 시간                     | > 850 µs |

총 합

```
~150 µs
```

즉

```
1ms budget 중 약 15% 사용
```

으로 설계한다.

---

## 14.2 IMU Sampling 경로

ImuAcquisitionTask 수행 시간

```
BMI270 SPI read
↓
struct ImuSample 생성
↓
RingBuffer push
```

예상 시간

| 항목               | 시간       |
| ---------------- | -------- |
| SPI transaction  | 40~60 µs |
| sample pack      | 10 µs    |
| ring buffer push | 5 µs     |

총

```
~70 µs
```

---

## 14.3 Log Assemble 경로

LogAssembleTask

```
RingBuffer read
↓
Binary record 생성
↓
chunk buffer append
```

예상 시간

| 항목            | 시간    |
| ------------- | ----- |
| buffer read   | 5 µs  |
| record encode | 20 µs |
| chunk append  | 10 µs |

총

```
~35 µs
```

---

## 14.4 SD Write 경로

SdWriteTask

SD write는 **비동기 chunk write**로 처리한다.

Chunk 크기 예

```
4 KB ~ 32 KB
```

SD write latency

| 유형         | 지연        |
| ---------- | --------- |
| 평균         | 1~5 ms    |
| worst case | 50~100 ms |

대응 전략

* RingBuffer
* Double Buffer
* Pre-allocated file

---

## 14.5 Buffer Size 요구사항

100ms stall 대응을 위해 필요한 버퍼

예

1000Hz 샘플

샘플 크기

```
16 bytes
```

초당 데이터

```
16 KB/s
```

100ms stall 대응

```
1.6 KB
```

안전 여유 포함

```
RingBuffer ≥ 64 KB
ChunkBuffer ≥ 32 KB
```

# 15. Memory Architecture

본 장은 **ESP32‑S3 + PSRAM 기반 메모리 구조와 버퍼 전략**을 정의한다.

목표

* 1000Hz 로깅 안정성 확보
* SD stall 대응
* 메모리 fragmentation 방지
* hot‑path dynamic allocation 제거

메모리 사용은 다음 3가지 영역으로 구분한다.

```
Internal SRAM (빠른 실시간 처리)
PSRAM (대용량 버퍼)
Flash / LittleFS (설정 저장)
```

---

## 15.1 메모리 영역 개요

ESP32‑S3 메모리 구성

| 영역            | 용도                             |
| ------------- | ------------------------------ |
| Internal SRAM | RTOS, task stack, control data |
| PSRAM         | 로깅 버퍼, chunk buffer            |
| Flash         | firmware                       |
| LittleFS      | 설정 파일                          |

설계 원칙

```
hot path 데이터 → PSRAM ring buffer
control 데이터 → internal RAM
설정 파일 → LittleFS
```

---

## 15.2 Memory Map (권장 설계)

예시 메모리 배치

```
Internal SRAM
 ├ RTOS kernel
 ├ task stacks
 ├ control structures
 └ small queues

PSRAM
 ├ IMU RingBuffer (64~256 KB)
 ├ ChunkBuffer (32~64 KB)
 ├ SD Write Queue buffers

Flash
 ├ firmware

LittleFS
 ├ /config.json
 ├ /runtime.json
 ├ /calib.json
 ├ /stats.json
```

---

## 15.3 RingBuffer

IMU 샘플 저장용 고속 버퍼

목적

* SD write stall 대응
* sampling path non‑blocking 보장

구조

```
ImuTask
  ↓
RingBuffer (PSRAM)
  ↓
LogTask
```

권장 크기

```
64 KB ~ 256 KB
```

설계 규칙

* lock‑free 또는 최소 lock
* overwrite 금지
* overflow 감지

---

## 15.4 ChunkBuffer

SD write 단위 버퍼

목적

* sequential write
* SD 효율 향상

구조

```
LogTask
  ↓
ChunkBuffer
  ↓
SdWriteTask
```

권장 크기

```
32 KB
```

가능한 옵션

```
4 KB
8 KB
16 KB
32 KB
```

권장

```
32 KB
```

---

## 15.5 SD Write Queue

SD write 요청 큐

목적

* sampling path와 SD write 분리

구조

```
LogTask
  ↓
WriteQueue
  ↓
SdWriteTask
```

큐 항목

```
chunk pointer
chunk size
file offset
```

---

## 15.6 Buffer Pool

ChunkBuffer 재사용을 위한 메모리 풀

목적

* malloc 사용 제거
* fragmentation 방지

구조

```
BufferPool
 ├ chunk0
 ├ chunk1
 ├ chunk2
```

동작

```
allocate → write → recycle
```

---

## 15.7 LittleFS 메모리 사용

LittleFS는 설정 데이터 저장용으로 사용한다.

파일

```
/config.json
/runtime.json

```

특징


---

## 15.8 Task Stack 메모리

RTOS task stack은 **Internal SRAM 사용**을 권장한다.

예시

| Task        | Stack |
| ----------- | ----- |
| ImuTask     | 4 KB  |
| LogTask     | 4 KB  |
| SdWriteTask | 6 KB  |
| WebTask     | 8 KB  |
| SystemTask  | 3 KB  |

---

## 15.9 메모리 안전 규칙

다음 규칙을 반드시 준수한다.

```
1. hot path dynamic allocation 금지
2. malloc / free 금지
3. new / delete 금지
4. String 사용 금지
```

대체

```
RingBuffer
BufferPool
Static allocation
```

---

## 15.10 메모리 요구량 요약

예상 메모리 사용

| 항목          | 크기     |
| ----------- | ------ |
| RingBuffer  | 128 KB |
| ChunkBuffer | 32 KB  |
| Write Queue | 8 KB   |
| Task stacks | ~30 KB |

총

```
~200 KB PSRAM
```

---

## 15.11 메모리 설계 핵심 요약

핵심 설계

```
1. sampling → ring buffer
2. assemble → chunk buffer
3. write → SD queue
4. config → LittleFS
```

핵심 목표

```
1000Hz 로깅에서도 메모리 병목이 발생하지 않도록 설계
```
