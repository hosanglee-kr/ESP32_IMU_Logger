# 21. Logging Pipeline Detailed Design

본 장은 **1000Hz IMU 로깅을 안정적으로 구현하기 위한 상세 로깅 파이프라인 설계**를 정의한다.

핵심 목표

* IMU 취득과 SD write 완전 분리
* worst-case SD stall 동안에도 샘플 손실 방지
* marker / event / checkpoint를 동일 timeline에 통합
* stop / flush / finalize 시퀀스 표준화
* single owner 원칙 유지

---

## 21.1 파이프라인 상위 구조

전체 데이터 흐름

```text
BMI270 INT/FIFO
→ ImuTask
→ RawSample RingBuffer
→ LogTask
→ ChunkBuilder
→ WriteReadyQueue
→ SdWriteTask
→ SDMMC 1-bit Storage
```

보조 흐름

```text
Button / EventDetect / ExternalSync
→ MarkerEventQueue
→ LogTask merge
```

제어 흐름

```text
SystemTask
→ state transition
→ start / stop / flush / error handling
```

---

## 21.2 단계별 역할 정의

### 21.2.1 IMU Interrupt / FIFO Trigger

역할

* BMI270 FIFO watermark 또는 data-ready 발생
* ISR은 최소 동작만 수행

ISR 허용 동작

* flag set
* task notify
* lightweight queue signal

ISR 금지 동작

* 긴 SPI transaction
* JSON 처리
* state transition
* SD write
* 로그 출력

---

### 21.2.2 ImuTask

역할

* FIFO burst read
* raw frame decode
* monotonic timestamp 부여
* 기본 sanity check 수행
* RawSample RingBuffer에 push

입력

* ISR notify
* 현재 IMU config snapshot

출력

* RawSample stream
* diagnostics event
* fatal / recoverable error event

핵심 원칙

* sampling jitter 최소화
* SD 상태를 직접 참조하지 않음
* state 변경 직접 수행 금지

권장 처리 흐름

```text
wait notify
→ read FIFO length
→ burst read
→ decode raw sample batch
→ assign tick_us
→ validate sample
→ ring push
```

---

### 21.2.3 RawSample RingBuffer

역할

* 고속 샘플 임시 저장
* producer / consumer 분리

producer

* ImuTask

consumer

* LogTask

설계 원칙

* single producer / single consumer
* overwrite 금지
* high watermark 감시
* overflow 발생 시 diagnostics 증가 및 warning event 생성

권장 데이터

```text
RawSample
- tick_us
- gx gy gz
- ax ay az
- temp
- status flags
```

---

### 21.2.4 MarkerEventQueue

역할

* 사용자 입력 marker와 자동 감지 marker를 LogTask로 전달

producer

* UiTask
* EventDetectService
* ExternalSync wrapper

consumer

* LogTask

지원 marker 유형

* MARKER_EVENT
* MARKER_IMPORTANT
* MARKER_SYNC
* EVENT_IMPACT
* EVENT_HIGH_ROTATION
* EVENT_FREE_FALL
* EVENT_VIBRATION_ANOMALY

설계 원칙

* marker는 low volume control path로 처리
* marker queue overflow 시 marker drop count 기록
* raw sample path를 방해하지 않아야 함

---

## 21.3 LogTask 상세 설계

역할

* RawSample RingBuffer 소비
* marker merge
* chunk build
* checkpoint 삽입 결정
* write-ready chunk 생성

입력

* RawSample RingBuffer
* MarkerEventQueue
* config snapshot

출력

* WriteReadyQueue
* diagnostics event
* overflow / flush warning event

세부 처리 단계

```text
pop raw sample
→ merge pending marker by tick order
→ append to active chunk
→ evaluate chunk threshold
→ if threshold hit, seal chunk
→ push chunk ref to WriteReadyQueue
→ acquire next buffer from pool
```

핵심 원칙

* file write 직접 수행 금지
* active chunk ownership은 LogTask
* sealed chunk ownership은 SdWriteTask로 이전
* chunk boundary에서 marker ordering 보장

---

## 21.4 ChunkBuilder 설계

목적

* raw sample를 binary record로 패킹
* marker / checkpoint / metadata를 동일 포맷으로 통합

chunk 포함 가능 레코드

* IMU_SAMPLE
* MARKER
* CHECKPOINT
* optional metadata extension

chunk seal 조건

* byte threshold 도달
* sample count threshold 도달
* periodic flush timer 도달
* segment rotate 요청
* stop/finalize 요청

권장 정책

* 고정 최대 크기 + 주기 flush 혼합
* write 효율과 recovery 단위 균형 유지

---

## 21.5 Checkpoint 삽입 설계

목적

* 전원 차단 후 salvage 가능성 향상

checkpoint 포함 정보

* last valid tick
* chunk sequence
* logical file offset
* crc

삽입 시점

* N초마다
* NKB마다
* segment close 직전
* stop/finalize 직전

권장 기준

* 5초 또는 256KB 중 먼저 도달하는 조건

---

## 21.6 WriteReadyQueue 상세 설계

역할

* sealed chunk를 SdWriteTask로 전달

producer

* LogTask

consumer

* SdWriteTask

payload 권장 항목

```text
chunk buffer reference
chunk size
chunk type
sequence number
```

원칙

* 데이터 복사 대신 buffer reference 전달
* queue depth는 backpressure 감시 대상
* critical depth 도달 시 warning / controlled degradation 수행

---

## 21.7 SdWriteTask 상세 설계

역할

* file handle 단일 소유
* chunk sequential write
* sync / latency measurement
* footer finalize
* segment rotate 처리

입력

* WriteReadyQueue
* flush / stop / rotate request
* file state from FileManager

출력

* write result event
* diagnostics metrics
* recycled buffer return

핵심 원칙

* SD write ownership = SdWriteTask
* 다른 task의 direct file write 금지
* SDMMC 1-bit storage path 단일 소유자

상세 흐름

```text
wait write-ready chunk
→ validate current file state
→ write chunk
→ measure latency
→ update counters
→ periodic sync if required
→ recycle buffer to pool
```

오류 처리

* transient error → retry
* repeated error → file abort flag
* fatal error → SystemEventQueue 보고

---

## 21.8 FileSegmentManager 연계

역할

* session file open
* pre-allocation
* segment rotate
* footer / close
* recovery metadata

파이프라인 연계

```text
SystemTask start record
→ FileSegmentManager open segment
→ SdWriteTask write chunk stream
→ rotate request
→ FileSegmentManager next segment
→ continue write
```

segment rotate 트리거

* configured time 도달
* size threshold 도달
* explicit rotate request

원칙

* rotate 중 raw sample 유실 금지
* rotate는 flush-safe sequence로 수행

---

## 21.9 Start Sequence 상세 설계

정상 시작 흐름

```text
SystemTask receives RECORD_START_REQ
→ guard check (SD mounted / IMU ready / battery ok / not USB)
→ FileSegmentManager create & pre-allocate segment
→ clear counters / arm timers
→ ImuTask sampling enable
→ LogTask chunk active
→ state = RECORDING
```

필수 guard

* SD mounted
* IMU initialized
* not USB_MSC
* not FLUSHING
* battery above critical threshold

---

## 21.10 Stop / Flush / Finalize Sequence

정상 정지 흐름

```text
SystemTask receives RECORD_STOP_REQ
→ transition RECORDING → FLUSHING
→ ImuTask stop new acquisition
→ drain FIFO batch
→ LogTask seal active chunk
→ SdWriteTask flush all queued chunks
→ write final checkpoint
→ write footer
→ sync
→ close file
→ state = READY
```

보장 목표

* stop 이후 raw sample 누락 최소화
* footer 이전에도 salvage 가능한 구조 유지
* flush 완료 전 USB_MSC 진입 금지

---

## 21.11 Marker Merge 규칙

목적

* marker와 raw sample timeline 일관성 보장

규칙

* marker는 tick 기준으로 삽입
* 동일 tick에서 marker ordering은 중요도 우선
* marker가 sample throughput을 방해하면 안 됨

권장 우선순위

```text
SYNC > IMPORTANT > EVENT > NORMAL
```

---

## 21.12 Backpressure & Overflow 대응

감시 지표

* RingBuffer fill ratio
* WriteReadyQueue depth
* free chunk count
* max write latency

단계별 대응

### warning

* diagnostics 증가
* LED / status warning

### severe

* low priority marker 제한
* verbose diagnostics 축소

### critical

* controlled degradation
* optional side-channel logging 중단

### fatal

* stop request
* recoverable flush attempt

핵심 원칙

* raw IMU sample 보존 최우선
* 부가 정보보다 핵심 샘플 우선 보호

---

## 21.13 Error Path 설계

오류 유형

* IMU FIFO overflow
* RingBuffer overflow
* ChunkBuffer 부족
* SD write retry overflow
* sync failure
* footer finalize failure

전파 규칙

```text
Driver / Task detects error
→ DiagnosticsQueue update
→ SystemEventQueue notify
→ SystemTask decides transition
```

recoverable 예

* temporary SD busy
* short write retry success

fatal 예

* repeated write failure
* file close failure with corrupted state

## 21.14 Resource Ownership 요약

## 21.15 Detailed Timing Budget

1000Hz = 1ms budget 기준

권장 세부 예산

| 단계                         | 목표 시간 |
| -------------------------- | ----- |
| FIFO status / read prep    | 10 µs |
| FIFO burst read            | 40 µs |
| raw decode                 | 20 µs |
| timestamp assign           | 5 µs  |
| ring push                  | 5 µs  |
| LogTask pack per batch 평균화 | 30 µs |
| marker merge overhead      | 5 µs  |
| chunk seal amortized       | 10 µs |

sampling critical path 목표

```text
< 125 µs
```

설계 해석

* 1ms 중 sampling 관련 path를 충분히 작게 유지
* SD write는 비동기 분리되어 평균 throughput만 확보하면 됨
* worst-case stall은 buffer headroom으로 흡수

---

## 21.16 Recommended Buffer Sizing

샘플 크기 가정

```text
raw sample ≈ 20 bytes
```

1000Hz 기준 데이터율

```text
≈ 20 KB/s + metadata overhead
```

권장 버퍼

* RingBuffer : 128 KB 이상
* ChunkBuffer : 32 KB × 3 buffers
* WriteReadyQueue : 8~16 entries

의미

* 100ms stall은 쉽게 흡수
* 긴 stall이나 sync spike도 추가 headroom으로 대응 가능

## 21.18 설계 핵심 요약

Logging Pipeline의 핵심 원칙은 다음과 같다.

1. IMU 취득과 SD write를 완전히 분리한다.
2. raw sample은 ring buffer에서 우선 보호한다.
3. binary chunk 단위로 sequential write 한다.
4. marker / checkpoint / footer를 동일 pipeline에 통합한다.
5. file write는 SdWriteTask 단일 소유로 유지한다.
6. stop / flush / finalize를 상태 머신과 연동해 표준화한다.

이 장은 **1000Hz 안정 로깅을 실제 구현으로 연결하는 핵심 설계 기준**으로 사용한다.
