# 5. 하드웨어 구성

## 5.1 SD 인터페이스 변경

본 설계의 SD 저장장치 인터페이스는 기존 SPI 기반에서 **ESP32-S3 SDMMC 1-bit 모드**로 변경한다.

변경 목적

* SPI 공유 버스 충돌 감소
* SD 전용 버스 확보
* sustained write 안정성 향상
* 1000Hz 로깅 headroom 확대
* worst-case stall 대응 여유 확보

설계 원칙

* BMI270는 SPI 유지
* SD Card는 SDMMC 1-bit 전용 버스 사용
* 초기 제품 안정화 우선으로 4-bit 대신 1-bit 채택
* 파일시스템/로깅 상위 계층은 인터페이스 추상화 유지
* 향후 보드 revision에서 4-bit 확장 가능 구조로 설계

권장 사항

* SD 배선 길이 최소화
* pull-up 및 신호 무결성 검토
* 카드 detect / power control 가능 시 반영
* hot-insert 정책은 READY 상태 중심으로 제한

# imu_driver_design_001.md

# BMI270 Driver Design Specification

## 1. 목적

이 문서는 ESP32-S3 기반 IMU Logger에서 **BMI270 드라이버 설계 기준**을 정의한다.

설계 목표:

* 1000Hz 안정 샘플 취득
* FIFO/Interrupt 기반 저지터 구조
* Gyroflow용 데이터 품질 확보
* 자동/수동 calibration 지원
* orientation 및 scale 일관성 확보

---

## 2. 드라이버 책임 범위

BMI270 드라이버는 다음 책임을 가진다.

* BMI270 초기화 및 칩 ID 검증
* SPI 통신 및 레지스터 접근
* FIFO/DRDY 인터럽트 처리 지원
* 샘플링 주파수/범위/필터 설정
* 자이로 bias calibration 적용
* raw frame decode
* 상태/오류 플래그 제공

드라이버는 다음을 직접 담당하지 않는다.

* SD 저장
* 파일 포맷 인코딩
* LED/버튼 UX
* 상위 상태 전이

---

## 3. 초기화 시퀀스

```text
power stable
→ SPI init
→ chip id read
→ soft reset
→ config file load/init sequence
→ accel/gyro range setting
→ ODR/BW setting
→ FIFO setting
→ interrupt mapping
→ self-check
→ ready
```

초기화 실패 시 구분:

* SPI 통신 실패
* chip id mismatch
* config apply 실패
* FIFO init 실패
* interrupt mapping 실패

---

## 4. 설정 항목

### 4.1 사용자 설정값

* sample_rate_hz
* gyro_range_dps
* accel_range_g
* filter profile
* fifo enable
* auto_calibration
* motion wake threshold

### 4.2 내부 설정값

* ODR mapping
* bandwidth profile
* FIFO watermark level
* interrupt polarity
* axis remap/orientation metadata

---

## 5. 샘플링 모드

### 5.1 권장 모드

1. **High Quality**

   * 1000Hz
   * gyro ±2000dps
   * accel ±16g

2. **Balanced**

   * 400Hz
   * gyro ±1000dps
   * accel ±8g

3. **Low Power Motion**

   * motion detect 위주
   * 저속 또는 event 기반 wake

### 5.2 BMI270 ODR 연계 원칙

* 최종 external log sample rate와 BMI270 internal ODR를 분리해서 생각
* internal ODR는 aliasing과 jitter 측면에서 상위 여유를 둘 수 있음
* Gyroflow 목적의 기본값은 raw integrity 우선

---

## 6. FIFO/인터럽트 설계

### 6.1 권장 구조

* BMI270 FIFO 사용
* watermark interrupt 사용
* 필요 시 data-ready fallback 제공

### 6.2 ISR 원칙

ISR에서는 최소 동작만 수행한다.

```text
ISR
→ event flag set
→ task notify / queue push
→ return
```

ISR에서 금지:

* 긴 SPI read
* dynamic allocation
* complex parsing

### 6.3 Acq Task 처리 흐름

```text
event wait
→ fifo length read
→ burst read
→ raw frame decode
→ timestamp assign
→ sample validate
→ enqueue to logger pipeline
```

### 6.4 FIFO watermark 전략

* 너무 낮으면 interrupt 빈도 증가
* 너무 높으면 latency/jitter 증가
* 1000Hz 기준 수 ms 단위 batch가 적절
* 실측 benchmark 후 보정 필요

---

## 7. timestamp 설계

### 7.1 원칙

* 샘플 timestamp는 monotonic 기준
* RTC wall-clock은 파일 메타용
* FIFO batch 내 개별 sample timestamp를 균등 간격 보간 가능

### 7.2 권장 방식

1. batch read 시점 monotonic tick 확보
2. batch sample count 확인
3. ODR 기준으로 각 sample tick 재구성
4. monotonic 역전 여부 검사

### 7.3 검증 항목

* tick 역전 여부
* batch 간 간격 이상 여부
* sample drop 추정 가능 여부

---

## 8. raw frame decode

드라이버 출력은 가급적 **정규화 전 raw 정수값** 중심으로 유지한다.

출력 구조 예시:

```text
struct Bmi270RawSample {
  uint32_t tick_us;
  int16_t gx;
  int16_t gy;
  int16_t gz;
  int16_t ax;
  int16_t ay;
  int16_t az;
  int16_t temp;
  uint16_t status;
}
```

status bit 예:

* overflow
* clipped
* interpolated_tick
* calibration_applied

---

## 9. calibration 설계

### 9.3 stillness 판정

* 일정 시간 window 유지

### 9.4 실패 정책

# logger_pipeline_design_001.md

# 1000Hz Logging Pipeline Design

## 1. 목적

이 문서는 1000Hz 고속 로깅을 안정적으로 처리하기 위한 **펌웨어 내부 파이프라인**을 정의한다.

핵심 목표:

* IMU 취득과 SD write 완전 분리
* stall 흡수 가능한 buffering 구조
* marker/event/checkpoint 통합 기록
* 전원 차단 시 복구 가능

---

## 2. 상위 파이프라인

```text
BMI270 INT
→ ImuAcquisitionTask
→ RawSampleQueue/RingBuffer
→ ChunkBuilder
→ WriteReadyQueue
→ SdWriteTask
→ FileSegmentManager
```

보조 경로:

```text
Button/Event Detector
→ MarkerQueue
→ ChunkBuilder merge
```

---

## 3. 단계별 책임

### 3.1 ImuAcquisitionTask

* FIFO burst read
* raw decode
* timestamp assign
* sample health check
* ring buffer push

### 3.2 ChunkBuilder

* raw sample를 file record로 패킹
* marker/event record merge
* active chunk 관리
* checkpoint timing 판단

### 3.3 SdWriteTask

* write-ready chunk 수신
* sequential write
* sync/latency 측정
* 오류 시 retry/recover

### 3.4 FileSegmentManager

* file open/pre-allocate
* segment rotate
* footer finalize
* recovery metadata 관리

---

## 4. 버퍼링 구조

### 4.1 버퍼 계층

1. **sensor ingest ring buffer**
2. **active chunk buffer**
3. **write-ready buffer queue**
4. **SD write in-flight buffer**

### 4.2 권장 정책

* ingest ring: 내부 RAM 우선
* active/write-ready chunk: 내부 RAM 우선
* PSRAM은 overflow 흡수용 보조 spool로만 제한적 사용

### 4.3 double vs triple buffering

* 기본은 double buffer
* SD stall worst-case가 길면 triple buffer 확장
* 설계상 N-buffer로 일반화 가능하게 구현 권장

---

## 5. chunk 설계

### 5.1 chunk 포함 가능 레코드

* IMU sample record
* marker record
* auto-event record
* checkpoint record

### 5.2 chunk 생성 기준

* fixed byte size
* 또는 fixed sample count
* 또는 시간 기반 flush

권장:

* write 효율과 recovery 균형을 위해 **고정 최대 크기 + 주기 flush** 혼합

### 5.3 chunk 전환 흐름

```text
active chunk fill
→ threshold hit
→ seal chunk
→ push write-ready queue
→ new active chunk allocate
```

---

## 6. marker/event merge 설계

marker는 샘플 스트림과 동일 tick timeline에 삽입한다.

입력원:

* button event
* external sync GPIO
* impact detector
* high rotation detector
* free-fall detector
* vibration anomaly detector

merge 규칙:

* 해당 tick 근처의 기록 순서 보장
* marker가 샘플 처리 지연을 유발하면 안 됨
* marker queue overflow 시 drop count 기록

---

## 7. checkpoint 설계

checkpoint는 비정상 종료 복구용이다.

포함 항목:

* file offset
* last valid tick
* chunk sequence
* crc

생성 시점:

* N초마다
* NKB마다
* segment rotate 직전
* stop/finalize 직전

---

## 8. backpressure 정책

### 8.1 감시 포인트

* ring buffer fill ratio
* write-ready queue depth
* SD write latency spike
* chunk recycle latency

### 8.2 단계별 대응

1. warning

   * LED warning
   * telemetry increment
2. severe

   * marker/aux record 우선순위 낮춤
3. critical

   * controlled drop policy 또는 emergency flush
4. fatal

   * recording stop + recoverable footer attempt

### 8.3 controlled degradation 원칙

고속 IMU raw sample을 최우선으로 보호하고, 다음 순서로 degrade.

* debug telemetry
* verbose diagnostics
* low-priority marker
* optional side-channel logs

---

## 9. SD write 경로 설계

```text
write-ready queue pop
→ file state validate
→ write chunk
→ update counters
→ periodic sync
→ recycle buffer
```

latency 계측:

* avg latency
* max latency
* p95/p99 추정 가능하면 유지

오류 시:

* limited retry
* file abort flag
* segment close attempt
* upper layer notify

---

## 10. stop/finalize 시퀀스

```text
record stop request
→ stop new acquisition
→ drain FIFO
→ seal active chunk
→ flush all write-ready chunks
→ write footer
→ sync
→ close file
```

보장 목표:

* stop 이후 queue 비우기
* 가능하면 마지막 checkpoint/summary 기록

---

## 11. race condition 방지 포인트

차단 대상:

* recording ↔ USB_MSC
* recording ↔ SD format
* recording ↔ WiFi heavy task
* segment rotate ↔ stop request
* low battery ↔ large write burst

해결 원칙:

* 상태 머신 단일 승인 경로
* file manager mutex
* write task ownership 명확화

---

## 12. 성능/진단 지표

* samples_written
* samples_dropped
* marker_inserted
* marker_dropped
* write_max_latency_us
* queue_high_watermark
* chunk_recycle_fail_count
* finalize_time_ms

---

## 13. 구현 권장안

* hot path에서 malloc/free 금지
* 고정 크기 buffer pool 사용
* 구조체 packing 및 alignment 명시
* task 간 데이터 복사 최소화
* state change는 event 기반으로만

# sd_performance_strategy_001.md

# SD Performance and Reliability Strategy

> 저장 인터페이스 기준: **ESP32-S3 SDMMC 1-bit mode**

## 1. 목적

이 문서는 ESP32-S3 + **SDMMC 1-bit SD Card 환경**에서 **1000Hz 로깅을 안정적으로 유지하기 위한 SD 성능 전략**을 정의한다.

핵심 목표:

* write stall 흡수
* contiguous sequential write 확보
* 전원 차단 내성
* 카드 품질 편차 대응

---

## 2. 문제 정의

SDMMC 1-bit 모드에서도 평균 속도보다 **worst-case latency spike**가 더 중요하다.

SPI 대비 SDMMC 1-bit는 전용 인터페이스 확보 측면에서 유리하지만, 여전히 카드 내부 FTL/erase/sync 지연은 존재하므로 설계 초점은 throughput보다 **latency containment**에 둔다.

문제 요소:

* FTL 내부 erase/block management
* 조각난 파일
* 카드 품질 편차
* sync 호출 시 stall
* 저전압 시 write 실패

따라서 설계 초점은 throughput보다 **latency containment**에 둔다.

---

## 3. 핵심 전략

### 3.1 Pre-allocation

* segment 시작 시 예상 크기 미리 할당
* contiguous file 확보 우선
* fragmentation 최소화

### 3.2 Large sequential write

* 작은 write 다발 대신 큰 chunk write
* metadata update 횟수 최소화

### 3.3 Buffered pipeline

* SD write를 acquisition과 분리
* write stall 동안도 sensor ingest 지속 가능
* SDMMC driver latency variation을 버퍼 계층으로 흡수

### 3.4 Periodic sync tuning

* 너무 잦은 sync는 stall 증가
* 너무 드문 sync는 손실 범위 증가
* 시간/용량 기준 혼합 정책 권장

### 3.5 Card qualification

* 허용 카드 리스트 관리
* 장치 내 benchmark로 품질 판정

---

## 4. 파일 운영 정책

### 4.1 segment rotation

* 5분 단위 권장
* 또는 용량 기준 병행
* 큰 단일 파일보다 복구/관리 유리

### 4.2 file naming

* safe character set만 사용
* RTC 기준 timestamp + segment index

### 4.3 footer/checkpoint

* segment 내 checkpoint 유지
* finalize 시 footer 작성
* 비정상 종료 시 salvage 가능

---

## 5. benchmark 전략

장치 내 SD benchmark 항목:

* sequential write throughput
* max single write latency
* repeated sync latency
* pre-allocation success
* stall frequency

판정 예시:

* PASS_1000HZ
* PASS_400HZ_ONLY
* WARNING
* FAIL

활용:

* 1000Hz 모드 허용 여부 결정
* 사용자 경고
* 서비스 진단 자료 저장

---

## 6. write latency 대응

### 6.1 latency 구간 분류

* normal
* elevated
* spike
* critical spike

### 6.2 대응 정책

* normal: 그대로 기록
* elevated: queue watermark 감시 강화
* spike: warning counter 증가
* critical spike: emergency headroom 사용

### 6.3 headroom 설계

버퍼 여유는 최소 100ms를 넘도록 잡고, 실제 상용 목표는 더 크게 잡는다.

권장:

* normal headroom: 300ms 이상
* preferred headroom: 500ms 이상
* worst-case test: 1초 근접 stall 검증

---

## 7. 카드 호환성 전략

권장 시험군:

* 일반 consumer card
* high endurance card
* 서로 다른 용량/브랜드

검증 항목:

* 장시간 1000Hz
* 저전압 근접 상황
* fragment 많은 상태
* 반복 파일 생성/삭제 후
* 장시간 온도 변화 환경

정책:

* 검증된 카드 whitelist
* 미검증 카드 사용 시 warning

---

## 8. 전원 차단 대응

전원 차단 시 핵심 목표는 **파일 시스템 완전성**보다 **복구 가능한 마지막 유효 chunk 보존**이다.

전략:

* checkpoint 삽입
* footer 이전에도 salvage 가능 구조
* brownout detect 시 신규 취득 중단
* flush 가능한 범위까지만 시도

하드웨어 협조:

* bulk capacitor
* 저전압 조기 감지
* hold-up 시간 확보

## 9. 파일 시스템 운영 원칙

* SD format은 장치 내 포맷 권장
* PC 임의 포맷 후 first-use benchmark 수행
* log directory 구조 단순 유지
* 동시 접근 금지 (MSC expose 중 내부 write 금지)

---

## 10. 진단 항목

저장 권장 지표:

* last_card_benchmark_grade
* max_write_latency_us
* avg_write_latency_us
* sync_max_latency_us
* segment_finalize_fail_count
* emergency_buffer_use_count
* last_recovery_scan_result

---

## 11. 구현 권장안

* ESP32-S3 SDMMC 1-bit driver 기반 구현
* block aligned chunk size 실험
* write size tuning benchmark 자동화
* card insert 시 간단 quick test 가능
* service mode에서 full benchmark 제공
* HAL 계층에서 SDMMC와 추후 4-bit/대체 인터페이스 전환 가능하게 추상화

---

## 12. 최종 권고

SD 전략의 핵심은 다음 5가지다.

1. pre-allocation
2. sequential large write
3. deep buffering
4. checkpoint + recovery
5. card qualification

이 5가지를 초기에 설계에 포함해야 1000Hz 제품화 성공 확률이 높아진다.

특히 이번 단계부터는 다음 3가지를 고정하는 것이 좋다.

1. `StorageManager` / `SdmmcStorageDriver` 분리
2. `ChunkBuilder` / `LoggerService` 분리
3. `ConfigManager` 단일 저장 경로 강제
