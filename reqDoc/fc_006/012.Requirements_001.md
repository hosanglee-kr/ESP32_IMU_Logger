# IMU Logger Requirements Bundle

본 문서는 IMU Logger 시스템의 요구사항을 다음 세 가지 관점으로 정리한다.

1. User Feature Requirements (사용자 기능 요구사항)
2. System Functional Requirements (시스템 기능 요구사항)
3. System Non-Functional Requirements (시스템 비기능 요구사항)

설명은 사용자 및 개발자의 이해를 돕기 위해 **한글 기준**으로 작성한다.

---

# 1. User Feature Requirements

사용자가 실제로 경험하는 기능을 정의한다.

즉

"사용자가 무엇을 할 수 있는가"를 설명한다.

---

## 1.1 IMU 데이터 기록

사용자는 장치를 통해 IMU 센서 데이터를 기록할 수 있다.

기록 데이터

* Gyroscope (X,Y,Z)
* Accelerometer (X,Y,Z)
* Temperature
* Timestamp

지원 샘플링 속도

* 200 Hz
* 400 Hz
* 800 Hz
* 1000 Hz

사용 예

* 액션캠 영상 안정화 (Gyroflow)
* 자전거 주행 데이터 분석
* 드론 비행 분석
* 충격 분석

---

## 1.2 기록 시작 / 종료

사용자는 버튼을 이용하여 데이터 기록을 시작하거나 종료할 수 있다.

버튼 동작

* short press → recording start
* long press → recording stop

LED 상태 표시

* READY : 녹색
* RECORDING : 빨간색

---

## 1.3 이벤트 마커 삽입

사용자는 기록 중 특정 시점에 이벤트 마커를 삽입할 수 있다.

버튼 입력

* short press → marker
* double press → important marker

마커 활용

* 테스트 구간 표시
* 충격 구간 표시
* 특정 이벤트 표시

---

## 1.4 자동 이벤트 감지

시스템은 IMU 데이터를 기반으로 자동 이벤트를 감지할 수 있다.

감지 이벤트

* 충격 감지
* 각속도 급증
* free fall
* 진동 이상

발생 시 자동 marker 기록

---

## 1.5 자동 파일 분할

긴 기록 데이터를 관리하기 위해 로그 파일을 자동으로 분할한다.

기본 설정

* 5분 단위 파일 생성

파일 예

IMU_20260310_0001.bin
IMU_20260310_0002.bin

---

## 1.6 USB 파일 접근

사용자는 USB로 장치를 PC에 연결하여 기록된 데이터를 복사할 수 있다.

동작

* USB 연결 시 Mass Storage 장치로 인식
* SD 카드 파일 접근 가능

---

## 1.7 WiFi 설정 기능

사용자는 스마트폰 또는 PC를 통해 장치 설정을 변경할 수 있다.

지원 기능

* 샘플링 속도 변경
* 센서 범위 설정
* WiFi 설정

접속 방식

* 장치 AP 모드 접속
* 웹 페이지 설정

---

## 1.8 LED 상태 표시

WS2812 LED를 이용하여 장치 상태를 직관적으로 표시한다.

상태 표시 예

BOOT : white breathing
READY : green
RECORDING : red
ERROR : red blink
WIFI : cyan
USB : purple

---

## 1.9 센서 캘리브레이션

사용자는 센서 정확도를 향상시키기 위해 캘리브레이션을 수행할 수 있다.

방식

* 자동 캘리브레이션
* 수동 캘리브레이션

저장 위치

calib_001.json

---

## 1.10 전원 관리

배터리 사용 환경에서 전력 소비를 줄이기 위한 기능을 제공한다.

지원 기능

* 자동 절전
* motion wakeup
* low battery 보호

---

# 2. System Functional Requirements

시스템이 반드시 수행해야 하는 동작을 정의한다.

문장은 다음 형태로 작성한다.

"The system shall ..."

---

## 2.1 IMU 데이터 샘플링

The system shall acquire IMU samples at configured sample rate.

지원 샘플링

* 200 Hz
* 400 Hz
* 800 Hz
* 1000 Hz

데이터 항목

* gyro xyz
* accel xyz
* temperature
* timestamp

---

## 2.2 IMU 데이터 버퍼링

The system shall buffer IMU samples using ring buffer before logging.

목적

* SD write 지연 대응
* 데이터 손실 방지

---

## 2.3 로그 기록

The system shall write IMU samples to storage in binary format.

특징

* binary logging
* high speed write
* chunk 기반 기록

---

## 2.4 로그 파일 관리

The system shall create segmented log files.

기본 정책

* 시간 기반 분할
* 자동 파일 rotation

---

## 2.5 이벤트 마커 기록

The system shall insert marker records when events occur.

이벤트

* button marker
* auto detection marker
* external sync marker

---

## 2.6 전원 차단 복구

The system shall recover log files after unexpected power loss.

방법

* checkpoint
* chunk CRC

---

## 2.7 설정 관리

The system shall store configuration files in LittleFS.

파일

* config_001.json
* runtime_001.json
* calib_001.json
* stats_001.json

---

## 2.8 USB MSC 모드

The system shall expose SD storage as USB Mass Storage when connected to a PC.

조건

* recording 상태에서는 USB MSC 금지

---

## 2.9 WiFi 설정 서버

The system shall run a configuration web server in WiFi mode.

지원 기능

* config read
* config write
* status 조회

---

## 2.10 상태 관리

The system shall manage system states using a state machine.

주요 상태

* BOOT
* READY
* RECORDING
* WIFI_CONFIG
* USB_MSC
* ERROR

상태 전이는 SystemTask에서 관리한다.

---

## 2.11 로그 파이프라인

The system shall process logging pipeline using task based architecture.

구조

ImuTask → LogTask → SdWriteTask

---

## 2.12 Config 접근 정책

The system shall access configuration only through ConfigManager.

직접 JSON 접근 금지.

---

## 2.13 상태 전이 관리

The system shall perform state transitions only in SystemTask.

다른 task에서는 직접 상태 변경 금지.

---

# 3. System Non-Functional Requirements

시스템의 성능, 안정성, 제약 조건 등 품질 요구사항을 정의한다.

---

## 3.1 로깅 성능

The system shall support stable IMU logging up to **1000 Hz** without data loss.

요구사항

* 최대 1000 Hz 샘플링
* 로그 손실 0
* 장시간 기록 안정성

---

## 3.2 SD 카드 지연 허용

The system shall tolerate SD card write latency up to **100 ms** without dropping samples.

대응 방법

* RingBuffer
* Double Buffer
* Chunk logging

---

## 3.3 데이터 무결성

The system shall ensure log integrity even in unexpected power loss.

보장 방법

* chunk CRC
* checkpoint
* file recovery

목표

* 최소 데이터 손실

---

## 3.4 메모리 안정성

The system shall avoid dynamic memory allocation in the logging hot path.

규칙

* malloc/free 금지
* new/delete 금지
* String 사용 금지

대체 방법

* BufferPool
* RingBuffer

---

## 3.5 실시간 처리 안정성

The system shall separate sensor acquisition and storage tasks.

아키텍처

ImuTask → LogTask → SdWriteTask

목표

* sampling jitter 최소화

---

## 3.6 전력 소비

The system shall support low power operation in battery environments.

지원 기능

* idle sleep
* motion wakeup
* low battery shutdown

---

## 3.7 시스템 복구

The system shall recover to READY state after abnormal reboot.

방법

* runtime state 저장
* incomplete file recovery

---

## 3.8 시스템 안정성

The system shall operate continuously for long-duration recording sessions.

목표

* 수시간 이상 기록
* memory leak 없음

---

## 3.9 안전 모드

The system shall prevent unsafe operations during recording.

예

* recording 중 USB MSC 금지
* recording 중 SD format 금지

---

## 3.10 구성 데이터 안정성

The system shall validate configuration files before applying them.

정책

* JSON schema validation
* config backup
* migration 지원

---

## 3.11 동시성 및 리소스 소유 규칙

The system shall enforce single ownership of shared resources.

원칙

* 하나의 리소스는 하나의 owner task가 관리
* 다른 모듈은 queue 또는 API를 통해 접근

예

* SD card owner → SdWriteTask
* state transition owner → SystemTask
* config owner → ConfigManager

---

## 3.12 시스템 확장성

The system architecture shall support future feature extensions.

확장 예

* 추가 센서
* BLE 통신
* GPS 데이터 로깅

---
